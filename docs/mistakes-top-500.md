Each item: the mistake, why it hurts, the severity, and what vbuff should do instead.

## 1. Capture & monitoring mistakes

1. **Fixed-interval polling instead of OS change events** `[high]` - Polling the clipboard every N seconds (commonly 0.2-5s) means copies that happen and are overwritten between polls are missed entirely, while the periodic wakeups burn CPU and battery even when nothing is copied. Users report missed clips on fast successive copies and the app showing as a top energy consumer. → **vbuff:** Subscribe to native change notifications wherever they exist: AddClipboardFormatListener / WM_CLIPBOARDUPDATE on Windows, NSPasteboard change-count observation via a tight but event-coalesced loop on macOS (there is no push API, so check changeCount, never re-read content, on a short interval and back off when idle), and wlr-data-control / ext-data-control 'selection' events or X11 XFIXES SelectionNotify on Linux. Only fall back to polling where the platform offers nothing. _(seen in: CopyClip, Maccy, CopyQ, Windows 11 Win+V)_
2. **Polling changeCount but re-reading content every tick** `[high]` - On macOS the only signal is NSPasteboard.changeCount; apps that read the full pasteboard data on every poll cause the source app to re-render/transform formats repeatedly, slowing the source app and spiking the manager's CPU even when nothing changed. → **vbuff:** On each tick read ONLY the integer changeCount. Read actual pasteboard contents exactly once, and only when changeCount has incremented since the last observed value. Never touch data otherwise. _(seen in: Maccy)_
3. **Missing copies during rapid successive copying** `[high]` - When the user copies several things in quick succession, only the last one lands in history; intermediate clips are silently lost, forcing re-copying. This is a top recurring complaint across platforms. → **vbuff:** Drive capture off the change event/changeCount edge, not a slow timer, so every distinct change is observed. Where the platform coalesces (Windows clipboard service, Wayland), capture synchronously inside the event handler and enqueue, processing on a separate worker so the capture path never blocks and miss the next event. Add an integration test that fires N copies in a tight loop and asserts N entries. _(seen in: CopyClip, Maccy, Windows 11 Win+V, CopyQ)_
4. **Wayland data offer dies with the source window (no persistence)** `[critical]` - On Wayland the clipboard data lives in the source app's memory; when the source window closes the data offer is destroyed and the copy is lost. On X11 the server only holds a reference and asks the source for bytes at paste time, so closing the app loses the copy. This is the single most reported decades-old Linux clipboard frustration. → **vbuff:** vbuff must materialize (read out all offered MIME types into its own store) the moment a selection event fires, not lazily at paste time. On X11, take ownership / cache bytes immediately rather than relying on the source. Effectively act as the persistence helper (wl-clip-persist behavior) built in, so closing the source app never loses a clip. _(seen in: X11 core, Wayland core, GPaste, Klipper, Diodon, Clipman, CopyQ)_
5. **Relying on a compositor protocol that GNOME refuses to implement** `[critical]` - wlr-data-control / ext-data-control let managers read the clipboard, but GNOME's Mutter refuses to implement them on security grounds, so wl-clipboard-based managers simply do not function on GNOME Wayland: empty history, nothing captured. → **vbuff:** Detect the compositor and capability at startup. On GNOME Wayland, fall back to a supported path (a GNOME Shell extension bridge, the portal, or XWayland-scoped capture) and clearly tell the user what is and isn't captured rather than silently recording nothing. Never assume data-control exists. _(seen in: cliphist, clipman, wl-clipboard, GNOME Mutter, wl-clip-persist, Diodon)_
6. **Capture only works while the manager window is open** `[high]` - Some managers only see clipboard changes when their UI/window is active, so background copies are never recorded; the user copies, switches away, and the clip is gone. → **vbuff:** Run a persistent background capture daemon independent of any window/UI lifecycle. The capture listener must be registered at process start and survive window close, minimize, and (on XWayland) main-window close. _(seen in: CopyQ, cliphist, GPaste)_
7. **Ignoring the concealed/transient pasteboard flag** `[critical]` - Password managers mark copied secrets with OS hints (macOS org.nspasteboard.ConcealedType / transient, KDE x-kde-passwordManagerHint, Windows ExcludeClipboardContentFromMonitorProcessing / CanIncludeInClipboardHistory) so they are not stored. Managers that ignore these capture passwords, 2FA codes, and keys into persistent history anyway. → **vbuff:** Before storing, check every concealment/transient hint for the platform and skip capture entirely (do not even write a redacted placeholder to disk) when present. On macOS honor org.nspasteboard.ConcealedType, TransientType and AutoGeneratedType. On Windows honor the ExcludeClipboardContentFromMonitorProcessing / CanIncludeInClipboardHistory formats. On Linux/KDE honor x-kde-passwordManagerHint=secret. _(seen in: Maccy, CopyQ, Ditto, GPaste, Flycut, Clipy, Jumpcut)_
8. **Capturing transient/auto-generated clips that were never user copies** `[medium]` - macOS marks some pasteboard writes as transient or auto-generated (e.g. internal app handoffs); recording them pollutes history with junk the user never intentionally copied and can leak app-internal data. → **vbuff:** Skip pasteboard changes flagged org.nspasteboard.TransientType and org.nspasteboard.AutoGeneratedType. Only persist changes that look like deliberate user copies. _(seen in: Maccy, Flycut, Clipy)_
9. **No source-app attribution at all** `[high]` - History entries show content with no record of which app produced them, so users cannot tell a password copied from a password manager apart from random text, cannot build per-app trust rules, and cannot audit leaks. Source detection is also outright broken on Wayland for some managers. → **vbuff:** At capture time record the foreground/owner app identity (bundle id on macOS via NSWorkspace.frontmostApplication, window owner PID -> executable on Windows via GetClipboardOwner/GetWindowThreadProcessId, and the data-source client on Wayland where exposed). Store it with each entry and surface it in the UI. Where the platform cannot reliably attribute (Wayland GNOME), mark it 'unknown' explicitly rather than guessing. _(seen in: Diodon, GPaste, Klipper)_
10. **Capturing from every source with no exclude list by default** `[high]` - With no app/source filtering, the manager records from password managers, banking apps, terminals, and 2FA tools the moment they touch the clipboard, capturing secrets users never wanted stored. Users often don't know ignore rules even exist. → **vbuff:** Ship a sane default deny-list (known password managers, KeePassXC, 1Password, Bitwarden, browser password fields where detectable, terminal apps optionally) and make per-app include/exclude rules first-class and discoverable. Combine with concealment-hint honoring so exclusion does not depend solely on a hardcoded list. _(seen in: Ditto, Flycut)_
11. **PRIMARY vs CLIPBOARD selection confusion on Linux** `[high]` - Linux has two independent selections: PRIMARY (select-to-copy, middle-click paste) and CLIPBOARD (explicit Ctrl+C). Managers that mix them up capture select-highlights as if they were deliberate copies (flooding history with every text selection) or miss real Ctrl+C copies, or paste back the wrong selection. → **vbuff:** Treat PRIMARY and CLIPBOARD as distinct sources with separate handling. By default capture CLIPBOARD (deliberate copies); make PRIMARY capture an explicit opt-in because it fires on mere selection. Never conflate the two when storing or restoring. _(seen in: X11 core, GPaste, Klipper, CopyQ)_
12. **Capturing every PRIMARY selection floods history** `[medium]` - If PRIMARY (select-to-copy) capture is on, simply highlighting text anywhere creates a history entry, so history fills with accidental fragments and the real copies are buried. → **vbuff:** If PRIMARY capture is enabled, debounce/coalesce rapid selection changes (a selection in progress fires many events), and only commit a PRIMARY entry after the selection settles for a short interval. Strongly prefer CLIPBOARD-only by default. _(seen in: X11 core, GPaste)_
13. **Reading only text/plain and dropping richer offered types** `[medium]` - When the source offers text/html, RTF, image/png AND text/plain, a manager that reads only text/plain permanently loses the formatting/image at capture time; no later 'paste with formatting' can recover what was never stored. → **vbuff:** At capture, enumerate ALL offered MIME types/representations and store every one (or a chosen high-fidelity subset), keyed by type. Render/paste decisions (plain vs rich) happen later from the stored set. Never discard available representations during capture. _(seen in: Clipman, GPaste, Diodon, cliphist, Maccy, CopyQ)_
14. **Storing the text/html representation of an image instead of the pixels** `[medium]` - Copying an image from a browser offers both text/html (an <img> tag/URL) and image/png; managers that grab the first or the html type store markup, so pasting into an image editor fails and the actual image is lost. → **vbuff:** When both image and html/text representations are offered, prefer and store the binary image type (image/png, public.tiff) as the primary, keeping html as secondary. On Wayland this means actively reading the image type (the equivalent of running a dedicated image watcher), not relying on the default text grab. _(seen in: cliphist, clipman, wl-clipboard sync tools)_
15. **Lazy/delayed-rendering clipboard data not requested at capture** `[high]` - Apps like Excel and many Office/graphics apps advertise formats but render the bytes only on demand (delayed rendering / promised data). A manager that records the format list but never requests the bytes ends up with blank or un-pasteable entries; later the source app is gone and the bytes can never be produced. → **vbuff:** At capture, actively request (realize) the promised/delayed-rendered data for the formats you intend to keep, while the source still owns the clipboard. On Windows handle WM_RENDERFORMAT/WM_RENDERALLFORMATS semantics by pulling formats now; on macOS resolve NSPasteboard promised file/data providers. If realization fails, mark the entry as incomplete rather than storing an empty row. _(seen in: Windows 11 Win+V, CopyQ)_
16. **Blank/empty entries stored when capture partially fails** `[medium]` - When a read times out, returns null data, or only a format header is present, managers still create a visible-but-empty history row that cannot be pasted and confuses users (the clip 'exists' but is useless). → **vbuff:** Validate that at least one usable representation has actual bytes before committing an entry. If all reads fail or return null, retry briefly, and if still empty drop the entry (or store it as a clearly-flagged error state) instead of polluting history with un-pasteable rows. _(seen in: CopyQ, Maccy, Windows 11 Win+V)_
17. **Re-asserting clipboard ownership creates feedback loops** `[medium]` - Managers that take ownership of the clipboard to persist content (so it survives source close) can trigger their own change events, fight with other apps for ownership, and cause copies to be silently dropped or the clipboard to 'stop working after a while'. → **vbuff:** When vbuff writes/owns the clipboard for persistence or restore, tag those writes and suppress self-triggered capture (compare against a known-last-written fingerprint and ignore changes whose owner is vbuff itself). Avoid aggressive continuous re-assertion; assert once and back off, yielding ownership cleanly to other writers. _(seen in: Firefox (Wayland), wl-clip-persist, Klipper)_
18. **Echo loop: re-capturing your own restored/written clips** `[high]` - When the manager puts an old item back on the clipboard (restore, or self-write), naive capture sees a new change and re-records it, creating duplicates, reordering history, or an infinite re-add loop that pins a CPU core. → **vbuff:** Before storing, compare the incoming clip against a fingerprint (hash + owner) of what vbuff just wrote. If it matches a self-write, bump the existing entry's timestamp at most (or do nothing) instead of inserting a new row. Maintain a short suppression window after every self-write. _(seen in: GPaste, Clipy, CopyQ)_
19. **Infinite loop re-adding the same image to history** `[high]` - A bug where copying an image makes the daemon repeatedly re-add the same image, consuming all RAM and pinning a CPU core in an infinite loop. → **vbuff:** Combine self-write suppression with content-hash dedup so an identical image cannot be inserted twice in a row regardless of how many change events fire. Cap insert rate per unit time and assert in tests that copying one image yields exactly one entry. _(seen in: GPaste, Clipy)_
20. **Capture path blocks on huge clipboard payloads** `[high]` - Copying hundreds of MB of text or a large image makes the monitor read it synchronously, pinning a CPU core at 100%, ballooning RAM (~1.7GB), and crashing the monitor subprocess with 'bad allocation' so capture stops entirely. → **vbuff:** Stream/copy clipboard bytes off the event thread with a configurable size cap. Above the cap, store a truncated preview plus metadata and either skip or store-by-reference the full payload, never loading it all into the capture thread. Wrap reads in size guards and handle allocation failure without killing the capture process. _(seen in: CopyQ)_
21. **Storing raw Retina TIFF screenshots uncompressed** `[high]` - macOS screenshots arrive as 10-30 MB TIFF; storing the raw clipboard bytes means hundreds of MB/day and gigabytes/week, bloating the store and slowing the whole app from the capture path inward. → **vbuff:** At capture, transcode large images to a compressed format (PNG/WebP) and/or downscale a thumbnail for the list, storing the full image as an external blob by reference rather than inline. Make per-item image size limits configurable. _(seen in: Ditto, Maccy)_
22. **Capture monitor crash/hang takes down all recording silently** `[high]` - The clipboard listener service crashes or hangs and the tray/app keeps running, so the app looks alive but records nothing; users only notice much later that history stopped. On Windows the Clipboard User Service can crash/hang and break Win+V entirely. → **vbuff:** Run capture in a supervised component with a heartbeat/watchdog; auto-restart the listener on crash and re-register the OS hook. Surface a visible 'capture stopped' indicator when the listener is unhealthy instead of failing silently. Add a self-test that confirms the listener is registered. _(seen in: Windows 11 Win+V, ClipboardFusion, CopyQ)_
23. **Capture silently stops on the broken clipboard-chain model** `[high]` - On Windows, joining the legacy SetClipboardViewer chain breaks if any app in the chain misbehaves or doesn't forward messages, silently severing the manager from all future updates; other apps (RDP, VMware/VirtualBox, AutoHotkey, a second manager) fighting for the chain also stop capture. → **vbuff:** Use the modern AddClipboardFormatListener API (not the fragile SetClipboardViewer chain). Detect when notifications stop arriving (heartbeat after a known self-write) and re-register the listener. Tolerate coexisting clipboard tools without assuming exclusive chain ownership. _(seen in: Ditto, CopyQ)_
24. **Two managers double-capture and corrupt each other** `[medium]` - Running vbuff alongside another clipboard tool causes double-capture, wrong content pasted, and lost formatting because both react to and rewrite the clipboard. → **vbuff:** Detect other known clipboard managers at startup and warn. Make capture idempotent via content+owner fingerprinting so another tool's rewrites don't create spurious entries, and never assume exclusive ownership of the clipboard. _(seen in: Maccy, Pure Paste, CopyQ, Ditto)_
25. **No deduplication of repeated identical copies** `[low]` - Copying the same text repeatedly creates many identical rows, cluttering history and pushing useful older items out; some apps continuously re-copy the same value. → **vbuff:** At capture, hash the normalized content; if it matches the most recent (or any existing) entry, bump that entry's timestamp/position instead of inserting a duplicate. Make dedup the default with a toggle. _(seen in: Maccy)_
26. **Capture races with the clipboard owner (read-before-ready)** `[high]` - Reading the clipboard the instant a change fires can race the source app that is still writing additional formats, yielding partial data, the wrong format, or a transient lock failure ('Activating Wayland clipboard took 5000 ms', 'Null data in clipboard'). → **vbuff:** Open the clipboard for the shortest possible time, retry transient lock failures with a short backoff, and let the source finish writing all formats (small settle delay or wait for the offer to stabilize) before reading. Never hold the clipboard open longer than the single read; release immediately so the owner isn't blocked. _(seen in: CopyQ)_
27. **Holding the clipboard open / delayed-render lock stalls global paste** `[high]` - A manager that keeps the clipboard open or sits as a slow delayed-render owner makes every copy/paste system-wide stall up to a second, freezing typing and the whole keyboard. → **vbuff:** Never hold the clipboard handle across operations; acquire, read/write, release within a single short critical section. If vbuff offers data via delayed rendering, render it immediately and cheaply (from already-materialized bytes), never doing heavy work inside the render callback. _(seen in: Windows Clipboard History, CopyQ)_
28. **X11/Wayland/XWayland clipboard split not bridged** `[high]` - Copies don't propagate between Wayland-native apps and XWayland apps, so a manager watching one world misses copies made in the other; content copied in one toolkit isn't visible/pastable in the other. → **vbuff:** Watch both the Wayland selection (via data-control) and the X11/XWayland selection (via XFIXES) simultaneously, deduplicate across them by content fingerprint, and offer captured items into whichever world the paste target lives in. Don't assume a single clipboard. _(seen in: Diodon, CopyQ, XWayland bridge)_
29. **Capture stops when the main window is closed under XWayland** `[medium]` - Under XWayland, monitoring stops the moment the manager's main window is closed, so background copies after that point are never recorded even though the daemon appears to be running. → **vbuff:** Decouple the X11 event-receiving window from the user-facing window: keep a dedicated hidden, persistent message/selection window alive for the lifetime of the daemon so capture continues regardless of UI window state. _(seen in: CopyQ)_
30. **Per-item size cap silently drops large files/images at capture** `[medium]` - A hard per-item limit (e.g. Windows 4MB, or images/large files not saved at all) means the user copies something, it never enters history, and there is no indication it was rejected. → **vbuff:** Make size limits generous and configurable, store large items by external reference rather than rejecting, and when an item truly can't be kept, record a visible placeholder entry ('large image not stored, NN MB') so the user knows capture happened but was capped, instead of silent loss. _(seen in: Windows 11 Win+V, Ditto, CopyQ)_
31. **OCR/text-recognition run on every captured image** `[medium]` - Running text recognition over every copied image adds real CPU cost on the capture path and turns copied photos of IDs, cards, and receipts into searchable plaintext, broadening what's at risk from a feature that fires automatically. → **vbuff:** Make OCR opt-in and run it off the capture thread (lazy/background, after the item is stored), never blocking capture. Respect the same exclude/concealment rules so sensitive-source images are never OCR'd, and let users disable OCR for images entirely. _(seen in: KeepClips, Clipory, ClipZ, Alfred, Raycast, Maccy)_
32. **Polling wakeups burn CPU/battery while idle** `[high]` - A short poll interval pegs a core (~45% CPU / ~1.5 cores reported) and drains battery even when the user isn't copying anything, making an always-on utility a top energy consumer. → **vbuff:** Prefer event-driven capture so the process sleeps when idle. Where polling changeCount is unavoidable (macOS), use an adaptive interval that backs off when idle and tightens briefly after recent activity, and never read content on the idle ticks. _(seen in: Maccy)_
33. **Capturing into history offers no app-context for trust decisions on Wayland focus changes** `[medium]` - On Wayland, source-app detection is broken for some managers, so even when content is captured there is no reliable way to know it came from a sensitive app, defeating exclude rules and attribution. → **vbuff:** Where the compositor exposes the data source client, capture it; where it does not, fall back to correlating with the last-focused app via the portal or shell extension, and explicitly degrade exclude rules to 'best effort, source unknown' rather than silently capturing sensitive sources as if they were safe. _(seen in: Diodon, GPaste)_
34. **Refusing to capture content with certain characters** `[low]` - Some managers reject text containing special/control characters, so legitimate copies silently never enter history and the user thinks the copy worked. → **vbuff:** Treat clipboard payloads as opaque bytes (with declared encoding) at capture; do not validate or sanitize content as a precondition for storage. Store anything the OS offered, escaping only at display time. _(seen in: GPaste)_
35. **No suppression window means restore/clear operations re-enter history** `[high]` - When vbuff clears the live clipboard for security (e.g. honoring a password manager's auto-clear) or restores an item, the resulting clipboard change can be captured as a new entry, re-introducing the secret or duplicating items, defeating the clear. → **vbuff:** Wrap all self-initiated clipboard mutations (restore, clear, format-normalize) in a suppression window keyed by content fingerprint so the resulting change events are recognized as vbuff's own and never recorded. Coordinate with password-manager auto-clear so the cleared value is also purged from history, not just the live clipboard. _(seen in: Klipper, Plasma clipboard, KeePassXC, CopyQ)_
36. **Treating capture as best-effort with no observability** `[medium]` - When capture silently degrades (permission dropped, listener died, compositor unsupported, chain broken), users have no way to know recording stopped until they need a lost clip, undermining trust in a tool meant to never lose copies. → **vbuff:** Expose capture health: a status indicator showing 'capturing / paused / unsupported on this compositor / listener restarting', a recent-capture timestamp, and optional logging of captured-vs-skipped (with reason: concealed, excluded app, too large). Make silent failure impossible to miss. _(seen in: CopyQ, Diodon, Windows 11 Win+V)_

## 2. History & storage mistakes

37. **History wiped wholesale by an app update** `[critical]` - After upgrading the app, the entire saved history (sometimes including pinned items) vanishes or capture silently stops, with no recovery path. Users lose months of clips they trusted the tool to keep. → **vbuff:** Treat the on-disk store as a stable contract independent of app version. Never delete/recreate the DB on upgrade. Run forward-only, idempotent, transactional schema migrations gated by a stored schema_version; if a migration can't apply, keep the old DB untouched and refuse to start rather than wipe. Take an automatic pre-migration backup copy of the DB file before any schema change. _(seen in: Ditto, Maccy)_
38. **History not persisted across reboot/shutdown** `[high]` - Clipboard history lives only in memory or a volatile store, so a reboot, logout, or crash loses everything except items the user manually pinned. The 'history' isn't actually durable. → **vbuff:** Persist every captured clip to durable on-disk storage synchronously (or with a bounded write-behind flush + fsync) at capture time, not only on graceful quit. Verify the store reopens with full contents after a hard power loss. Never rely on an at-exit flush as the only persistence path. _(seen in: Windows 11 Win+V, Flycut)_
39. **History/settings persist only when manually configured** `[medium]` - Persistence is an opt-in toggle that defaults off (or is buried), so out of the box users lose their clipboard history and preferences after a restart and don't discover why. → **vbuff:** Persist history and preferences by default. If a privacy-conscious 'memory-only / no disk' mode exists, make it an explicit opt-in choice, clearly labeled, never the silent default. Show current persistence state in the UI. _(seen in: Flycut)_
40. **Pinned/favorite items lost or silently reordered** `[critical]` - Pins disappear after an OS update or reboot, silently switch order, can't be reordered, or get evicted by trimming logic. Pinned items are the user's most valuable curated content, and losing them is the worst kind of data loss. → **vbuff:** Store pinned items in a separate logical class that is exempt from size/age-based eviction entirely. Persist an explicit user-defined sort order (a stable order column, not insertion time) and preserve it across restarts and migrations. Cover pins in migration tests and in the automatic backup. _(seen in: Maccy, CopyQ)_
41. **Pinned items counted against the history cap and evicted** `[high]` - When the history hits its item/size limit, the trimming pass treats pinned clips as ordinary rows and deletes the oldest, silently destroying intentionally-kept items. → **vbuff:** Exclude pinned items from all eviction queries (item-count cap, age cap, size cap). Cap only unpinned history. If a separate cap on pins is desired, make it a distinct, explicit setting and warn before exceeding it instead of silently dropping. _(seen in: Maccy)_
42. **Unbounded database growth (deleted rows never reclaimed)** `[critical]` - Deleted clips linger as dead pages in the file, so the DB only ever grows; it bloats into gigabytes (sometimes tens or hundreds of GB), slowing every query and eventually the whole machine. → **vbuff:** Enforce a hard ceiling on row count and total bytes with continuous background eviction. Reclaim freed space automatically: enable SQLite auto_vacuum=INCREMENTAL (or run incremental_vacuum on a schedule), and run a full VACUUM periodically or after large bulk deletes. Surface current DB size in settings. _(seen in: Ditto)_
43. **Database never shrinks without manual compaction** `[high]` - The file only reclaims space if the user runs a manual 'Compact & Repair' / external SQLite VACUUM, and the built-in compaction is inconsistent or missing. Non-technical users never recover the space. → **vbuff:** Automate space reclamation: schedule incremental vacuuming during idle time and a periodic full VACUUM, with a manual 'Compact now' button as a fallback. Make compaction crash-safe (VACUUM in a transaction, on a copy if needed) and report bytes reclaimed. _(seen in: Ditto)_
44. **No per-item size cap; giant clips bloat the store** `[high]` - A single huge clip (hundreds of MB of text, a multi-MB Retina TIFF screenshot, a large file payload) is stored verbatim, ballooning the DB and memory and slowing capture, search, and load. → **vbuff:** Set a configurable per-item byte cap. For oversized payloads: store a bounded preview/thumbnail inline and the full payload as an external file referenced by hash (don't inline large blobs in the row). Compress text and re-encode images (e.g. TIFF -> PNG/lossless) before storing. Tell the user when an item exceeds the cap instead of silently truncating. _(seen in: Ditto, Maccy, Windows 11 Win+V)_
45. **Silent truncation/rejection of large items** `[medium]` - Items over an undocumented per-item limit (e.g. 4MB) are silently truncated or never saved at all, so the user later pastes a corrupted/partial clip or finds the item simply isn't there. → **vbuff:** Make any size limit explicit and configurable, and never store a partial clip masquerading as complete. If an item exceeds the cap, either store it fully via external-blob storage or skip it with a visible, non-modal indication ('item too large to keep'). Mark truncated previews clearly as previews. _(seen in: Windows 11 Win+V, Ditto, ClipboardFusion)_
46. **Hardcoded / low history limit silently drops items** `[high]` - A small fixed cap (e.g. 25 items, or a low hardcoded N) silently discards older clips the user expected to still be there, with no warning and no way to raise it. → **vbuff:** Make the history limit a user-configurable setting with a sane, generous default and a clear maximum tied to disk budget rather than an arbitrary small constant. When trimming, prefer eviction by total-size budget over a tiny fixed count. Optionally let users see how many items were trimmed. _(seen in: Windows 11 Win+V, Maccy)_
47. **Duplicate entries instead of dedup/bump** `[low]` - Copying the same content repeatedly creates many identical rows, cluttering history, wasting storage, and pushing useful older clips out of the cap faster. → **vbuff:** Deduplicate on a content hash (over the full normalized payload, all relevant MIME types) at insert time. When a duplicate is copied, bump the existing row's last-used timestamp and move it to the top instead of inserting a new row. Make dedup the default with an opt-out for users who want a literal event log. _(seen in: Maccy)_
48. **Dedup hashes the wrong representation, causing false merges** `[high]` - Hashing only one MIME flavor (e.g. text/plain) treats two genuinely different clips as identical, so distinct items (same visible text but different HTML/RTF/image, different source flags) get wrongly merged and one is lost. Or hashing too narrowly fails to merge true duplicates. → **vbuff:** Compute the dedup key over a canonical, multi-format fingerprint: hash each stored MIME representation and combine. Normalize consistently (line endings, trailing whitespace) but never collapse items that differ in any retained representation. Make normalization rules explicit and testable.
49. **Missed copies during rapid successive copying** `[medium]` - Items copied quickly in a row don't all land in history; the capture pipeline coalesces or outruns its own writes, so intermediate clips are lost and users must re-copy. → **vbuff:** Decouple capture from persistence: push each clipboard change onto a durable, ordered queue immediately, then persist asynchronously. Never debounce away distinct change events. Handle delayed-rendering sources by resolving the data before enqueueing, and back-pressure rather than drop on burst. _(seen in: CopyClip, Maccy, Windows 11 Win+V)_
50. **No schema versioning / no migrations** `[high]` - The schema has no version marker, so a new app version reading an old DB either crashes, misreads columns, or 'fixes' it by recreating (wiping) the store. There's no safe path to evolve the data model. → **vbuff:** Stamp the DB with a schema_version (e.g. PRAGMA user_version). Ship an ordered set of forward-only migration steps applied in a single transaction with the pre-migration backup in place. Refuse to open a DB whose version is newer than the app understands (don't downgrade-corrupt it).
51. **Migrations not transactional / not crash-safe** `[high]` - A migration that fails or is interrupted (crash, power loss, kill) leaves the DB half-migrated and corrupt or inconsistent, losing history with no clean rollback. → **vbuff:** Wrap each migration in a single transaction so it commits atomically or rolls back fully. Always copy the DB to a timestamped backup before migrating. After migration, run an integrity check; if it fails, restore the backup automatically and report the problem.
52. **Database corruption with no recovery, requires deleting the DB** `[high]` - When the file grows too large or a write is interrupted, the DB corrupts or the app freezes; the only 'fix' offered is renaming/deleting the DB so a fresh empty one is created, losing all history. → **vbuff:** Use a journaling mode that survives interruption (WAL with synchronous=NORMAL/FULL appropriate to durability needs). Run PRAGMA integrity_check on startup; on corruption, attempt automatic recovery (`.recover`/dump-and-reload) and fall back to the most recent good backup rather than discarding data. Keep rolling backups so a wipe is never the only option. _(seen in: Ditto)_
53. **DB on a cloud-synced folder causes lock/corruption and conflict files** `[high]` - Routing the live SQLite DB through Dropbox/Google Drive/iCloud produces .conflict copies, file locking errors, and corruption because cloud sync isn't safe for an actively-written DB with WAL/journal sidecar files. → **vbuff:** Store the DB in a per-machine app-data location that is never auto-synced. If cross-device sync is wanted, implement it at the record level (sync individual clips through a proper sync protocol), not by syncing the raw DB file. Detect when the DB path is inside a known cloud folder and warn the user. _(seen in: Ditto)_
54. **Secret residue survives deletion in freelists/WAL/journal** `[high]` - After a user deletes a clip (e.g. a copied password), the bytes remain recoverable in SQLite freelist pages, the WAL, the rollback journal, and unallocated pages, so a 'deleted' secret is still on disk. → **vbuff:** On deletion of sensitive items, overwrite then delete and force reclamation: run secure_delete (PRAGMA secure_delete=ON or per-op), checkpoint and truncate the WAL, and VACUUM so freed pages are reclaimed. For sensitive content, prefer storing it encrypted at rest so residue is ciphertext, and zeroize plaintext buffers in memory. _(seen in: Ditto)_
55. **Residual data left on disk after 'clear all' / uninstall** `[medium]` - After 'Clear history' or uninstall, the DB file, WAL/journal sidecars, external blob/thumbnail files, and config dirs remain, so the supposedly-wiped, secret-laden history survives. → **vbuff:** 'Clear all' must delete every artifact: main DB, -wal/-shm/-journal, external blob store, thumbnail cache, search index, and then VACUUM/recreate. Provide an uninstall path (or documented manual steps) that removes all app-data directories. Verify with a post-clear scan that nothing recoverable remains. _(seen in: Ditto)_
56. **No retention policy / no auto-expiry** `[high]` - Everything is kept forever with no age-based expiry, so secrets, 2FA codes, and stale data accumulate indefinitely, growing the store and the exposure window. → **vbuff:** Offer configurable retention: a max age for ordinary clips and a shorter, separate auto-expiry for items flagged sensitive ('clear after N minutes'). Run eviction continuously, not only at cap. Keep pinned items exempt. Default to a reasonable retention window rather than infinite. _(seen in: Maccy, Ditto, CopyQ)_
57. **No export / backup capability** `[high]` - Users can't export their history or pinned snippets, so they can't back it up, migrate to a new machine, or recover after corruption. The data is trapped and fragile. → **vbuff:** Provide explicit export to an open, documented format (e.g. JSON/JSONL plus a folder of referenced blobs, or a self-contained archive) and a matching import. Offer automatic periodic local backups of the DB with a retained rolling set. Document where data lives so users can back it up themselves.
58. **No import / no migration between machines or apps** `[medium]` - Even if export exists, there's no import, so moving to a new computer or coming from another clipboard manager means starting from an empty history and losing curated pins/snippets. → **vbuff:** Ship an import that round-trips the export format losslessly (content, pins, order, timestamps, types). Provide importers for common formats where feasible. Validate imported data and merge by content hash to avoid creating duplicates on re-import.
59. **Backup includes plaintext secrets with no protection** `[medium]` - An export/backup writes the entire history, including copied passwords and tokens, as plaintext to a file the user may email or store in cloud, widening exposure of secrets that were never meant to leave the machine. → **vbuff:** Exclude items flagged sensitive from exports by default (or require an explicit 'include sensitive' opt-in). Offer encrypted, passphrase-protected exports. Warn clearly when an export will contain potentially sensitive content.
60. **Image/large blobs stored inline in the row** `[medium]` - Image and binary clips stored inline in the DB row bloat the file, slow every query that touches the table, blow up memory when the UI loads rows, and make saves progressively slower (especially with Core Data/SwiftData-style inline blobs). → **vbuff:** Store large binaries externally (content-addressed files on disk, deduplicated by hash) and keep only a small thumbnail/preview plus metadata in the row. Stream blobs lazily on demand. Keep the hot table narrow so list/search queries stay fast regardless of payload size. _(seen in: Maccy, Windows 11 Win+V, ClipboardFusion)_
61. **Raw screenshots stored uncompressed, exploding disk usage** `[high]` - Retina TIFF screenshots are stored verbatim at 10-30 MB each, so normal daily copying produces hundreds of MB/day and gigabytes/week, filling disk and slowing the app. → **vbuff:** Transcode captured images to a compact lossless format (PNG, or WebP/AVIF where acceptable) before storing, downscale oversized images for the inline preview, and deduplicate identical images by hash. Track and cap total bytes consumed by image clips separately from text. _(seen in: Ditto, Maccy)_
62. **Infinite loop re-adding the same image to history** `[high]` - A capture bug re-detects its own clipboard write (or a flickering image source) and re-inserts the same image repeatedly, pinning CPU and growing the DB without bound until killed. → **vbuff:** Tag clips the app itself places on the clipboard and ignore self-originated changes. Dedup by content hash before insert so a re-seen identical image bumps rather than re-inserts. Add a rate/identity guard so the same hash can't be inserted in a tight loop. _(seen in: GPaste, Clipy)_
63. **Loading the entire history into memory/UI at once** `[high]` - Opening the picker loads all rows (and sometimes their blobs) into the view, so large histories (30k-50k items) freeze the app, scroll sluggishly, and feel crashed. Memory tracks total history size, not what's visible. → **vbuff:** Page the store: query only the visible window with LIMIT/OFFSET or keyset pagination, load thumbnails lazily, and back the list with a virtualized view. Never SELECT * the whole table to render. Keep the data layer able to serve the last N quickly regardless of total count. _(seen in: Maccy, CopyQ, Pastebot)_
64. **No index for search; full scans as history grows** `[high]` - Search/filter scans every row (and decompresses/decodes blobs) so it becomes laggy or freezes once history reaches a few thousand items or contains large single-line clips. → **vbuff:** Maintain a dedicated text search index (SQLite FTS5 over a normalized text projection of each clip) kept in sync transactionally with inserts/deletes. Index OCR-derived text separately if offered. Never search by scanning blob columns. Keep the index covered by the same backup/migration paths. _(seen in: CopyQ, Maccy)_
65. **Clip ordering corrupts (newest not on top)** `[medium]` - Sort relies on a fragile or mutated field, so newest clips stop appearing on top, history goes out of chronological order (often triggered by pin/sticky logic writing the order field), and users can't find recent copies. → **vbuff:** Order by an immutable monotonic sequence (a rowid/auto-increment or a dedicated insert-sequence column), separate from any user-editable pin order. Don't reuse the same column for both 'recency' and 'manual pin order'. Make ordering deterministic and covered by tests, including the pin-interaction case. _(seen in: Ditto)_
66. **Blank/empty items recorded in history** `[medium]` - Items show up with no visible content because the app stored a reference or wrong MIME flavor but not usable bytes (e.g. captured a delayed-render promise that never resolved), leaving un-pasteable, confusing rows. → **vbuff:** Resolve and validate the actual bytes for at least one renderable representation before committing a row; if nothing renderable resolves (delayed-render not fulfilled), don't store a phantom entry. Validate non-empty content per stored MIME type and drop or retry empties instead of persisting them. _(seen in: CopyQ, Maccy)_
67. **Rich representations discarded at storage time (lossy capture)** `[medium]` - Only text/plain is stored, so the original HTML/RTF/image/file-URL representations the source offered are gone forever; the user can never get the formatted or original version back from history. This is irreversible data loss at the storage layer. → **vbuff:** Capture and persist all offered representations of a clip together (plain text, html, rtf, image, file URLs, source metadata) keyed to one history entry. Let presentation/paste decide which flavor to use; storage should keep everything so no representation is lost. Store representations efficiently (compressed/external) to keep this affordable. _(seen in: Clipman, GPaste, Diodon, cliphist)_
68. **Source/metadata not captured, so history isn't reconstructable** `[low]` - Clips are stored without source app, timestamp, or type metadata, so users can't filter by app, can't tell when something was copied, can't build exclusion rules, and forensic/cleanup decisions are impossible. → **vbuff:** Store per-clip metadata: capture timestamp, last-used timestamp, source application identifier, content type(s), byte size, and a sensitive flag. Keep this in indexed columns so filtering and retention rules can use it. This metadata also powers app-based exclusion and per-app retention.
69. **Concealed/transient pasteboard flag ignored, so secrets get persisted** `[high]` - Password managers mark a copy as concealed/transient ('org.nspasteboard.ConcealedType', x-kde-passwordManagerHint, Windows ExcludeClipboardContentFromMonitorProcessing), but the manager stores it anyway, so credentials land in persistent on-disk history. → **vbuff:** Honor OS sensitivity hints at capture time and never persist flagged clips to durable history (keep them transient-only or skip entirely). Make this the default. Treat honoring these flags as a storage-layer policy, not just a UI nicety, and test that flagged items never reach the DB. _(seen in: Maccy, CopyQ, Ditto, Flycut, GPaste)_
70. **No app-based exclusion list to keep content out of storage** `[medium]` - By default the manager persists copies from every source, including password managers, banking apps, and terminals, so sensitive content enters the durable store unless users find and configure ignore rules they may not know exist. → **vbuff:** Support a per-source-app exclusion list applied before persistence, with sensible defaults that pre-exclude known password managers. Combine with the concealed-flag check so excluded/sensitive content never touches disk. Make exclusions discoverable in onboarding. _(seen in: Ditto, Flycut)_
71. **Memory grows with total history and is never released** `[high]` - RAM grows linearly the longer the app runs and with history size, and isn't freed until quit/relaunch; even clearing history doesn't release it, so the always-on tool becomes a memory hog. → **vbuff:** Keep only a bounded working set resident: don't cache the whole store in memory, evict caches under pressure, and drop blob/thumbnail caches when history is cleared. Ensure 'clear history' frees its memory and reclaims disk. Profile for leaks; cap in-memory queues and caches with explicit limits. _(seen in: Maccy, CopyClip, CopyQ)_
72. **Synchronous DB writes block capture/paste, causing global lag** `[medium]` - Persisting a clip (especially a large one) on the main/clipboard thread holds a lock or stalls, so copy/paste and even typing freeze for up to a second, and rapid copies queue up behind disk I/O. → **vbuff:** Do persistence off the UI/clipboard thread via a bounded async write queue. Use WAL so readers don't block writers. Keep the capture path's only synchronous work as enqueueing the change; never block clipboard ownership handoff on disk I/O. _(seen in: CopyQ, Windows Clipboard History)_
73. **No integrity/health check; corruption discovered only on failure** `[medium]` - The app never validates the store, so corruption (from a bad shutdown, disk error, or interrupted write) is discovered only when the app freezes or queries fail, by which point recovery options are gone. → **vbuff:** Run a lightweight PRAGMA quick_check on startup and a deeper integrity_check periodically. On detected corruption, automatically attempt recovery and fall back to the latest good backup. Keep rolling backups (e.g. daily, last N) specifically so recovery is always possible. _(seen in: Ditto)_
74. **Settings/config silently reset or corrupted** `[low]` - Preferences (limits, exclusions, retention, pin order, snippet/tab properties) silently reset or corrupt on rename/upgrade, erasing user customization and sometimes changing storage behavior (e.g. dropping the configured cap) without notice. → **vbuff:** Store config in a versioned, validated file with its own migrations and an atomic write (write-temp-then-rename). Keep a backup of the last-good config and restore on parse failure instead of resetting to defaults. Treat config the same as data for backup/migration purposes. _(seen in: CopyQ)_
75. **Non-atomic writes corrupt the store on crash mid-write** `[high]` - Writing the DB/blob/config in place (truncate-then-write, or partial blob write) leaves a corrupt or half-written file if the app crashes or power is lost mid-write, losing the item or the whole store. → **vbuff:** Make all writes atomic: rely on SQLite's transactional WAL for the DB, and for external blobs/exports write to a temp file then fsync and rename into place. Never overwrite a good file in place. Ensure interrupted writes leave the previous good state intact.
76. **Bulk delete/clear is non-atomic and partially applies** `[medium]` - Clearing or bulk-deleting many items isn't transactional, so an interruption leaves orphaned external blobs, a stale search index, or a half-emptied table, and 'clear' appears to succeed while data lingers. → **vbuff:** Perform bulk deletes in a single transaction that also removes external blobs and updates the search index, then checkpoint/vacuum. Garbage-collect orphaned blobs (referenced by no row) on startup. Verify item count and reclaimed space after a clear.
77. **Capture stops silently when another process owns the clipboard** `[high]` - When RDP, a VM, AutoHotkey, or a second clipboard manager fights for the clipboard chain, capture silently stops; the app stays alive and looks fine but records nothing, so the user discovers a gap in history only later. → **vbuff:** Use robust change detection (OS clipboard-change notifications with re-subscription on failure rather than a fragile viewer chain), detect and recover from lost ownership, and surface a visible 'capture paused/failed' state instead of silently recording nothing. Log capture health so gaps are detectable. _(seen in: Ditto, CopyQ)_
78. **Conflicting managers double-capture or cross-corrupt history** `[medium]` - Running two clipboard tools at once causes double-capture (duplicate rows), one tool re-asserting ownership in a feedback loop, or wrong/lost content, polluting and corrupting each other's stored history. → **vbuff:** Tag self-originated clipboard writes and ignore them, dedup defensively by content hash, and avoid re-asserting clipboard ownership to 'persist' content (don't enter ownership fights). Detect a likely second manager and warn, but never corrupt the store as a result. _(seen in: Maccy, Pure Paste, Firefox (Wayland))_
79. **No bulk selection / management of stored items** `[low]` - Users can't select, delete, or export multiple items at once, making it impossible to clean up the store, purge a batch of sensitive clips, or curate history; the store grows messy and users can't maintain it. → **vbuff:** Support multi-select with bulk delete, bulk pin/unpin, and bulk export operating efficiently as set-based DB operations (single transaction, not row-by-row). Make bulk delete reclaim space and update the index. This is core data management, not a luxury. _(seen in: Maccy)_
80. **External blobs orphaned or rows dangling (referential drift)** `[medium]` - When images/large payloads live in external files referenced by the DB, deletes or crashes leave orphaned blob files (wasting disk) or DB rows pointing at missing files (un-pasteable, broken previews). → **vbuff:** Use content-addressed storage with reference counting: a blob is deleted only when no row references it. On startup, reconcile: GC unreferenced blobs and flag/repair rows whose blob is missing. Keep blob and row lifecycle changes in the same transaction where possible.
81. **Legitimate copies silently rejected from history** `[low]` - The manager refuses to store certain content (e.g. text with special characters, certain MIME types), so legitimate copies silently never enter history and the user assumes they were saved. → **vbuff:** Store content as opaque bytes per MIME type; do not parse/validate text content in a way that can reject valid input (handle full Unicode, control characters, and binary safely). If a clip truly can't be stored, surface a visible reason rather than dropping it silently. _(seen in: GPaste)_
82. **Database file world-readable in a predictable location** `[high]` - The history DB is written with permissive permissions in a guessable path, so any local process or user can read the full copy history (including any secrets that slipped in) straight off disk. → **vbuff:** Create the data directory and DB with owner-only permissions (0700 dir / 0600 files) in the platform's per-user app-data location. On platforms that support it, consider encryption at rest for the store. Never store the DB in a shared/world-readable directory. _(seen in: Ditto, CopyQ, Windows 11 Win+V)_
83. **OCR-derived text persisted unbounded and unmanaged** `[medium]` - Running OCR over every copied image and storing the recognized text turns photos of receipts/IDs/cards into searchable plaintext in the store, and the extra text bloats the DB and index with no separate control or retention. → **vbuff:** Make OCR opt-in, store OCR text in a clearly separated field subject to its own retention and the sensitive-exclusion rules, and let users disable/purge it independently. Account for OCR text in the search index and size budget. Never silently persist recognized text from sensitive images. _(seen in: Alfred, Raycast, Maccy)_

## 3. Search & filtering mistakes

84. **No search at all** `[critical]` - User accumulates hundreds or thousands of clips but the only way to find an old one is to scroll the entire list by hand, which is slow and often gives up before the item is found. The history becomes write-only. → **vbuff:** vbuff must ship a search box that is focused by default when the picker opens, so the user can start typing immediately to filter. Treat search as the primary navigation method for any history beyond ~20 items, not an afterthought. _(seen in: Windows 11 Win+V, Flycut, CopyClip)_
85. **Exact-substring-only matching, no fuzzy** `[high]` - The user must remember the exact characters they copied. A typo, a remembered-but-wrong word order, or matching on a token in the middle of a clip returns nothing, so the item appears lost even though it is in history. → **vbuff:** Implement fuzzy subsequence matching (e.g. nucleo/fzf-style) on top of, or instead of, plain substring. Rank exact and prefix matches highest, then contiguous substrings, then fuzzy subsequences. Fall back gracefully so a near-miss still surfaces the item. _(seen in: Maccy, Flycut, Diodon, Clipy)_
86. **Synchronous search that blocks the UI thread** `[critical]` - Typing each keystroke scans the whole history on the UI thread, so the picker freezes and stutters between characters. With large histories the app feels crashed while it filters. → **vbuff:** Run search off the UI thread (background task / worker) and stream results back incrementally. The input field must always remain responsive; never call a blocking full-scan inside the keypress handler. _(seen in: Maccy, CopyQ, Pastebot)_
87. **Linear full-table scan instead of an index** `[high]` - Every query re-scans every row and re-reads every blob, so latency grows linearly with history size. At 30k-50k items search becomes visibly laggy with a delay between typing and results. → **vbuff:** Maintain a persistent full-text index (SQLite FTS5, or Tantivy) over text content updated on insert/delete. Keep large binary blobs out of the searchable rows. Target sub-50ms query latency regardless of history depth. _(seen in: Maccy, CopyQ, Ditto)_
88. **No debounce / no query cancellation** `[high]` - Each keystroke launches a fresh search while the previous one is still running, so results flicker, arrive out of order, and stale results overwrite fresh ones. Fast typists see the wrong list, and CPU spikes. → **vbuff:** Debounce input (~30-80ms) and cancel any in-flight query when a newer keystroke arrives (cancellation token / generation counter). Only render results whose query matches the current input text.
89. **Case-sensitive search** `[high]` - Searching 'github' fails to find a clip stored as 'GitHub', so the user wrongly believes the item is gone. This is invisible and maddening because the content is clearly there. → **vbuff:** Default to case-insensitive matching using proper Unicode case folding (not ASCII tolower). Offer an optional case-sensitive / smart-case mode (case-sensitive only when the query contains an uppercase letter).
90. **Naive ASCII lowercasing instead of Unicode case folding** `[medium]` - Using ASCII-only lowercasing means 'STRASSE' vs 'straße', Turkish dotted/dotless I, and Greek sigma forms mismatch, so non-English users get inconsistent or missing results in their own language. → **vbuff:** Use Unicode case folding (Rust's char::to_lowercase / unicode-case-fold semantics) for both the indexed text and the query. Be aware of locale-specific folding (Turkish i) and document the chosen behavior.
91. **No diacritic / accent-insensitive matching** `[high]` - Searching 'cafe' does not find 'café', 'resume' misses 'résumé', and 'Munchen' misses 'München'. Users who do not type accents (or type them inconsistently) cannot find their own clips. → **vbuff:** Normalize both index and query with Unicode NFD and strip combining marks for a fold-accents matching layer (ASCII-folding), while preserving the original bytes for display and paste. Make accent sensitivity a toggle for users who need it. _(seen in: Paste, Alfred)_
92. **No Unicode normalization (NFC/NFD) before comparison** `[medium]` - The same visible string copied from different apps can be stored in different normalization forms (precomposed vs decomposed). Searching one form fails to match the other, producing phantom 'missing' clips, common with macOS-sourced text. → **vbuff:** Normalize all indexed text and all queries to a single canonical form (NFC) before matching. Store original bytes separately for faithful paste. This must happen at index time and query time consistently.
93. **Poor relevance ranking (chronological-only results)** `[high]` - Search returns matches in raw recency order, so a clip where the query is a tiny incidental substring outranks a clip that is exactly the query. The best match is buried below noise. → **vbuff:** Rank by a composite score: exact match > prefix match > word-boundary match > contiguous substring > fuzzy; then tie-break by recency and pin/frequency. Surface the top match first so Enter pastes the obvious answer. _(seen in: Maccy, Alfred)_
94. **No frequency/recency signal in ranking** `[medium]` - The clip the user pastes ten times a day ranks the same as one pasted once a month ago, so frequent items never float to the top and the user retypes more of the query than necessary. → **vbuff:** Track per-item paste count and last-used timestamp; blend a frecency score (frequency + recency decay) into ranking so habitually used clips surface early for short queries.
95. **Search only matches the start of the clip (prefix-only)** `[high]` - If matching anchors to the beginning of the stored text, a query that appears in the middle or end of a clip (e.g. a token inside a long URL or code block) never matches, so most long clips are unsearchable by their meaningful content. → **vbuff:** Match anywhere in the content, not just the prefix. Use prefix only as a ranking boost, not a filter. Tokenize so individual words inside the clip are independently matchable.
96. **No content-type filter (text vs image vs file vs link)** `[high]` - User wants 'that image I copied' or 'a file I copied' but must wade through hundreds of text clips. There is no way to narrow by kind, making image/file recall practically impossible. → **vbuff:** Tag each clip with a type (plain text, rich text, URL, image, file path, color, code) at capture time and expose first-class type filters (chips or a `type:image` token). Default views can be filtered by type instantly. _(seen in: Alfred, Raycast, Maccy)_
97. **No source-application filter** `[medium]` - User remembers 'I copied that from the browser' or 'from Slack' but cannot filter by origin app, so they cannot use the strongest memory cue they have to narrow the list. → **vbuff:** Capture the source application (bundle id / process name / window title) with each clip and offer an app filter (`app:Safari` token or a dropdown). Falls back gracefully when source cannot be determined (Wayland), but never blocks search. _(seen in: Diodon)_
98. **No date / time-range filter** `[medium]` - User knows they copied something 'last Tuesday' or 'this morning' but cannot constrain the search by time, so they scroll endlessly or get drowned in matches spanning months. → **vbuff:** Store a precise capture timestamp and expose date filters: relative presets (today, yesterday, last 7 days) and a `before:`/`after:`/`date:` query syntax. Combine with text search rather than replacing it.
99. **Images / non-text content excluded from search entirely** `[medium]` - Copied images, screenshots, and files have no searchable representation, so they are invisible to search and can only be found by scrolling. Image-derived text (OCR) is never matchable. → **vbuff:** Index searchable metadata for non-text clips: filename, file extension, source app, dimensions, and optional on-device OCR text (opt-in, off by default for privacy). Let images match on those fields even without OCR. _(seen in: Alfred, Paste, Maccy)_
100. **OCR-everything making image text silently searchable (privacy footgun)** `[medium]` - Running OCR over every copied screenshot turns a copied photo of an ID, a card, or a receipt into searchable plaintext stored on disk, broadening what an attacker or shoulder-surfer can recover via search. → **vbuff:** Make OCR strictly opt-in and per-source configurable; run it on-device only, never store OCR text for clips matching sensitive heuristics, and let the user purge OCR text without deleting the image. Search must not be the back door that exposes OCR'd secrets. _(seen in: KeepClips, Clipory, ClipZ)_
101. **Search broken outright for extended periods (untested regressions)** `[high]` - Search silently stops returning results after an update and stays broken for long stretches, so the core retrieval feature is dead and users cannot find anything until a fix months later. → **vbuff:** Cover search with automated tests (unit tests on the matcher/ranker, integration tests on the index) and a smoke test in CI that indexes sample clips and asserts query results. Treat a search regression as a release blocker. _(seen in: Paste, Alfred)_
102. **Cannot search older / archived items** `[high]` - Search only covers a recent window (or only the in-memory subset) while older clips on disk are excluded, so exactly the items the user has forgotten and most needs to find by search are unreachable. → **vbuff:** Index the entire persisted history, not just the loaded page. Search must query the on-disk index/store directly and return matches from any age, paginating results rather than capping the searched set. _(seen in: Paste, Alfred)_
103. **Loading entire history into the view to filter it** `[high]` - The UI pulls all 30k-50k rows (with blobs) into memory to filter client-side, causing freezes, sluggish scrolling, and slow summon even when the user only wants the top few matches. → **vbuff:** Push filtering down to the index/database with LIMIT and lazy/virtualized rendering. Load only the visible window of results plus a small buffer; fetch full content/blobs only for the highlighted item. _(seen in: Maccy, Pastebot, CopyQ)_
104. **No match highlighting in results** `[medium]` - Results show clips but do not indicate why each matched or where the query occurs, so for long clips the user cannot tell which is the right one and must open each to check, slowing selection. → **vbuff:** Highlight the matched span(s) in each result row and, for long clips, show a snippet/window centered on the first match rather than just the leading characters. Make the matched region visible at a glance.
105. **No multi-term / AND search** `[medium]` - Typing two words searches for the literal phrase 'word1 word2' and fails unless those words are adjacent in that order, so the user cannot narrow by combining independent terms they remember. → **vbuff:** Split the query on whitespace and require all terms to match (AND), order-independent, anywhere in the clip. Reserve quotes for exact-phrase search. This matches user expectation from web search.
106. **Whitespace / leading-space breaks matching** `[medium]` - A trailing or leading space in the query (easy to introduce), or differing internal whitespace (tabs vs spaces, CRLF) between query and stored clip, causes no match, so code and indented text become hard to find. → **vbuff:** Trim the query and normalize internal whitespace runs for matching (while preserving original content for paste). Treat tabs, newlines, and multiple spaces as equivalent during search.
107. **Search resets / loses state on each open** `[low]` - Filters, the last query, and scroll position are discarded every time the picker closes, so a user iterating on a search or repeatedly grabbing from a filtered subset must retype and re-filter every single time. → **vbuff:** Optionally remember the last query/filter for a short session window (configurable), and always start with the input focused and text selected so a fresh query overwrites instantly without manual clearing.
108. **Filters and text search are mutually exclusive** `[medium]` - User can filter by type OR type a query but not both, so 'find the image from Figma containing the word logo' is impossible and they fall back to scrolling. → **vbuff:** Make all filters (type, app, date, pinned) compose with free-text search as conjunctive constraints. A single query bar should accept structured tokens (`type:image app:Figma logo`) plus free text simultaneously.
109. **No way to scope search to pinned/favorites** `[low]` - Pinned items the user curated as important cannot be searched separately, so as pins grow they become as hard to navigate as the full history, defeating the point of pinning. → **vbuff:** Provide a `pinned:` filter and a dedicated pinned view that is itself searchable. Pinned items should also rank higher in normal search results. _(seen in: Paste)_
110. **Pasting search results corrupts ranking state / wrong item** `[high]` - After filtering, the displayed index and the underlying history index diverge, so pressing Enter or a number key pastes a different item than the one highlighted in the filtered list. → **vbuff:** Bind selection and paste to the result object identity (stable clip id), never to a positional index that assumes the unfiltered list. Verify the highlighted item is what gets pasted under any active filter.
111. **Search re-indexes synchronously on every keystroke / no incremental index** `[high]` - If the index is rebuilt or the store re-tokenized on demand rather than maintained incrementally, search latency and CPU spike with history size and the app burns battery while idle after copies. → **vbuff:** Update the search index incrementally on insert/delete/dedup (transactional with the data write). Never rebuild the whole index to answer a query. Rebuild only on schema migration, in the background, with progress shown. _(seen in: Maccy)_
112. **No CJK / no-space-language tokenization** `[medium]` - Word-boundary tokenization assumes spaces, so Chinese, Japanese, and Thai text is treated as one giant token and substring queries inside it fail or behave unpredictably, leaving CJK users unable to search their clips. → **vbuff:** Use an n-gram or ICU-based tokenizer (or character-level matching) for scripts without word delimiters. SQLite FTS5 with the trigram tokenizer, or a CJK-aware analyzer in Tantivy, handles this. Test with real CJK content.
113. **Special characters in query treated as operators / cause errors** `[medium]` - Querying text containing FTS operator characters (quotes, parentheses, `*`, `-`, `:`) either errors out, returns nothing, or is silently rejected, so searching for code, URLs, or symbol-heavy clips fails. → **vbuff:** Escape or quote user input before passing it to the FTS engine; treat the query as literal text by default and only interpret operators behind an explicit advanced-syntax mode. Never let a user query crash or no-op the search. _(seen in: GPaste)_
114. **No feedback for empty / zero-result searches** `[low]` - When a query matches nothing the list just goes blank with no message, so the user cannot tell whether search is broken, the item was never captured, or it was dropped by a history cap. → **vbuff:** Show an explicit empty state ('No matches for X') and, when relevant, hint at why (e.g. 'older items beyond the history limit are not searched' or 'enable OCR to search image text'). Distinguish 'no results' from 'still searching'.
115. **Search ignores rich-text / underlying plain text mismatch** `[medium]` - For rich/HTML clips the searchable text indexes the HTML markup (tags, style attributes) instead of the rendered plain text, so queries match tag names and miss the actual visible words, or vice versa. → **vbuff:** Extract and index the plain-text representation of rich clips (strip markup) for matching, while keeping the rich payload for paste. Never index raw HTML/RTF as the search corpus. _(seen in: cliphist, CopyQ)_
116. **cliphist-style id<TAB>preview format breaks fuzzy-picker column parsing** `[medium]` - When search results are piped to fzf/rofi/fuzzel as `id<TAB>preview`, the picker splits on the delimiter and matches/selects the wrong column, so recall silently returns the wrong clip or fails to match obvious queries. → **vbuff:** If vbuff exposes a CLI/pipe interface, search the preview text but key selection on the id with an unambiguous, configurable delimiter (NUL byte). Ship a built-in picker so users are not forced into fragile column-splitting configs. _(seen in: cliphist, clipcat)_
117. **No keyboard-only result navigation** `[medium]` - Search filters the list but the user must reach for the mouse to pick a result, breaking the fast type-and-paste flow and forcing a context switch on every search. → **vbuff:** Keep results fully keyboard-navigable while the input stays focused: arrows/Ctrl-n/p to move, Enter to paste, number keys for the top N, Esc to cancel. Selection must never require leaving the search field.
118. **Stale results after history changes mid-search** `[low]` - If a new clip is captured, or an item is deleted/deduped, while a search is open, the result list does not reflect it (or worse, indices shift and the wrong item is acted on), confusing the user. → **vbuff:** Re-run the active query against the updated store when the history changes (or mark the result set dirty and refresh on next interaction), keying actions to stable ids so concurrent updates never cause a wrong-item paste.

## 4. Paste mechanics mistakes

119. **Paste silently does nothing after OS/app update** `[critical]` - User picks an item, the picker dismisses, but nothing is inserted into the focused field. The clip looks selected but the paste never happens, so the workflow silently breaks and the user doesn't know why. → **vbuff:** Treat auto-paste as a fallible operation with explicit verification: after injecting the paste, detect failure (e.g. accessibility call returned error, target didn't accept) and surface a visible, actionable error ('Paste failed - accessibility permission missing, click to fix'). Add a self-test on launch and after detecting an OS version change that exercises the paste path against a hidden test field and warns if it regressed. _(seen in: Maccy, Paste, Pastebot, ClipMenu, Flycut, Clipy)_
120. **Paste creates a notification instead of inserting text** `[critical]` - When the injection path fails, the app falls back to showing the clip text in a system notification rather than pasting it. The user has to manually re-copy from the notification, which defeats the entire point of one-keystroke paste. → **vbuff:** Never use a notification as a paste fallback. If keystroke injection cannot reach the target, leave the content on the system clipboard and tell the user to press the native paste shortcut, or retry the injection. A notification dump is a bug, not a graceful degradation. _(seen in: Maccy)_
121. **Picker steals focus from the target app and breaks the paste destination** `[high]` - Opening the clipboard window changes the OS notion of the frontmost/focused app, so when the app pastes, the destination caret no longer exists or is the wrong window. User gets nothing pasted or must click back into the field first. → **vbuff:** Capture and store the target window/app reference BEFORE showing the picker (record the frontmost app and focused element at hotkey time). Show the picker as a non-activating/accessory panel that does not become the key window (like Spotlight), and explicitly restore focus to the stored target immediately before injecting the paste. _(seen in: Maccy, ClipMenu, CopyQ)_
122. **Paste lands in the wrong (previously-active or background) window** `[high]` - On multi-window / multi-monitor / Wayland setups the app refocuses the last X window or a background window instead of the one the user was actually typing in, so the paste is injected somewhere unexpected, sometimes overwriting real content. → **vbuff:** Persist a precise target descriptor at trigger time (window id, PID, and on platforms that allow it the focused UI element), and refocus exactly that target. If the original target window is gone or ambiguous, abort the auto-paste and leave content on the clipboard rather than guessing a window. _(seen in: CopyQ, cliphist, Ditto)_
123. **Injecting the literal keystroke character instead of pasting** `[high]` - When simulating Ctrl+V/Cmd+V via synthetic key events, a timing or modifier-state bug sends the 'v' as a plain character so the field receives a literal 'v' instead of the clipboard content, silently corrupting input. → **vbuff:** Hold and verify the modifier key is registered as down before sending the V keycode, and release in the correct order. Prefer platform paste APIs (e.g. AX 'paste' action, NSPasteboard + paste menu invocation) over raw keystroke synthesis where available. Add an integration test asserting the field receives clipboard content, not the letter v. _(seen in: Ditto)_
124. **Picker window opens behind other windows / un-typeable** `[high]` - The hotkey summons a panel that appears behind the active window or cannot receive keyboard input, so the user can't navigate or filter the list and the whole picker is unusable. → **vbuff:** Show the picker at an elevated window level (floating panel) that is guaranteed above the active app but does not steal activation, and ensure it accepts key input for filtering/arrow navigation. On Wayland where focus-stealing is restricted, document the required compositor support and degrade to a clearly-explained manual-paste mode rather than a dead window. _(seen in: Ditto, Diodon, CopyQ)_
125. **No real plain-text paste / hard to reach** `[high]` - The user wants to paste unformatted text into a destination, but the app only offers rich paste, or 'paste as plain text' is buried behind multiple clicks/menus, so users get unwanted fonts, colors, and styles dragged into their document. → **vbuff:** Make plain-text paste a first-class, single-keystroke action (e.g. a distinct modifier on the paste hotkey: Enter pastes as-is, Shift+Enter / Cmd+Enter pastes plain text). Offer a global default 'always paste as plain text' toggle. Compute the plain-text variant from the stored text/plain representation, never by trying to strip markup from HTML. _(seen in: Maccy, Paste, Alfred, Pastebot)_
126. **Plain-text path mangles content (strips file URLs, breaks data)** `[high]` - The 'paste without formatting' code path destroys legitimate non-style content, e.g. flattening file:// URLs, dropping line breaks, or collapsing tabs, so plain paste produces broken or lossy output. → **vbuff:** Define plain-text paste as: select the stored text/plain (or public.utf8-plain-text) representation verbatim and paste only that. Do not run regex/HTML-stripping transforms. For non-text flavors (file references, URLs) keep their native flavor; only fall back to text when the user explicitly asked for plain text. _(seen in: Maccy, Paste, Pastebot)_
127. **Rich-text/HTML formatting dropped or mangled on round-trip** `[medium]` - A clip captured with formatting pastes as bare text or with broken styling because only text/plain was stored, or the HTML/RTF representation was discarded. Users lose the formatting they deliberately copied. → **vbuff:** Capture and store ALL offered representations (text/plain, text/html, RTF, image/png, file URLs) for each clip, and re-offer the full multi-flavor pasteboard on paste so the destination picks the best one. Never collapse a multi-format clip down to a single representation at capture time. _(seen in: ClipAngel, Comfort Clipboard, Windows 11 Win+V, Clipman, GPaste, Diodon, cliphist, CopyQ)_
128. **Image clip stored/pasted as HTML instead of pixels** `[medium]` - When copying an image from a browser, the app grabs the text/html offer rather than image/png, so pasting into an image editor fails or inserts an HTML tag instead of the picture. → **vbuff:** When capturing, prefer image/png (or platform image flavor) when present and store the raw bitmap, while also keeping any accompanying html/text. On paste into a target that requests image data, serve the image flavor. Don't let the presence of a text/html alternative override an available image representation. _(seen in: cliphist, clipman, Maccy, CopyQ)_
129. **Auto-paste off by default / requires extra Ctrl+V after picking** `[high]` - Selecting a clip only copies it to the clipboard; the user still has to manually press the paste shortcut in the target, turning a one-action picker into a two-action chore and defeating the core value proposition. → **vbuff:** Make pressing Enter on a selected item paste it directly into the target by default (with a setting to fall back to copy-only for users who want it). Treat 'pick = paste' as the default contract, not an opt-in feature. _(seen in: Ditto, ArsClip)_
130. **Clobbering the user's current clipboard without restoring it** `[high]` - To auto-paste a history item the app overwrites whatever the user currently had on the clipboard and never puts it back, so the user silently loses their live copy and is confused when their next normal paste yields the wrong thing. → **vbuff:** Snapshot the live clipboard before overwriting it for the paste, perform the paste, then restore the original clipboard contents (all flavors) after a short delay. Make the temporary swap invisible to the user's own copy/paste flow.
131. **Clipboard restore races the destination's async paste read** `[high]` - The app restores the original clipboard too soon, before the (sometimes asynchronous) destination app has actually read the pasted data, so the destination pastes the restored/old content or empty data. This is a subtle timing bug that appears intermittently and only in some apps. → **vbuff:** Do not restore on a fixed sleep. Where possible confirm the paste consumed the data (or use a clipboard-change/owner-change signal) before restoring, and use a conservatively long, configurable delay as the fallback. Prefer pasteboard change-count / generation tracking over arbitrary timers.
132. **Hotkey collides with built-in OS shortcuts** `[medium]` - Default paste/select shortcuts overlap with reserved OS combos (e.g. macOS Cmd+Shift+3/4/5 screenshots, Cmd-X), so triggering a paste fires a screenshot or cut instead, doing the wrong destructive thing. → **vbuff:** Ship defaults that don't shadow documented OS shortcuts, detect at registration time whether the chosen combo is already claimed by the OS or another app, and refuse silently-failing bindings with a clear 'this shortcut is taken by X' message plus a suggested alternative. _(seen in: Maccy, CopyClip 2, CopyQ)_
133. **'Failed to set hotkey' on startup leaving paste with no trigger** `[medium]` - Another app already grabbed the global shortcut, so the hotkey registration fails and the user has no way to summon the picker or paste, but the app keeps running as if fine. → **vbuff:** On registration failure, surface a persistent, clearly-worded warning (which app holds it, how to rebind) instead of a one-time dialog, and provide a fallback trigger (menu bar / tray click). Re-attempt registration when the conflicting app releases the combo. _(seen in: ArsClip, Ditto, ClipboardFusion)_
134. **Hotkey+Enter opens picker but never pastes** `[high]` - The summon shortcut works and an item can be selected, but the confirm action doesn't actually inject the paste, so the user goes through the motions and gets nothing. → **vbuff:** Treat the full chain (summon -> select -> confirm -> focus restore -> inject -> verify) as one tested flow. Add an end-to-end test that drives the hotkey and asserts the target field received content, so a broken confirm step can't ship silently. _(seen in: Maccy, Flycut, CopyQ)_
135. **Paste inserts at wrong position / overwrites a selection** `[medium]` - Instead of inserting at the caret, the paste replaces an existing selection or the whole field (e.g. clobbering a filename during rename), because the app sends select-all or its keystroke timing disturbs the selection. → **vbuff:** Never send select-all or navigation keys as part of paste. Restore focus, then issue only the platform paste action so the destination's own paste semantics (insert at caret, replace selection if user had one) apply. Do not assume the field is empty or fully selected. _(seen in: ArsClip)_
136. **Paste fails in Spotlight / Electron / secure fields** `[medium]` - Keystroke-injection auto-paste silently doesn't reach certain targets (Spotlight, some Electron apps, secure/password fields), so the paste appears to do nothing in exactly the places users try it. → **vbuff:** Detect target classes known to reject synthetic events and adapt: try the AX paste action, fall back to leaving content on the clipboard with a 'press Cmd+V here' hint when injection is impossible. Maintain a per-app behavior table and test against Spotlight, an Electron app, and a secure field explicitly. _(seen in: Maccy)_
137. **Fragile per-app paste with broken focused-field detection** `[medium]` - Auto-paste works in most apps but breaks in specific browsers/editors because the tool can't reliably figure out which control is focused, so paste goes nowhere or to the wrong control in those apps. → **vbuff:** Don't depend on fragile focused-element guessing for the common case; rely on restoring app/window focus and letting the OS paste into the real caret. Keep a maintained per-app quirks list with overrides, and when detection is uncertain, fall back to clipboard-only paste rather than injecting into a guessed control. _(seen in: ArsClip)_
138. **Accessibility permission silently dropped, auto-paste dies with no indication** `[high]` - After an update, re-signing, or OS upgrade the accessibility/automation permission gets revoked (or the TCC database corrupts), so auto-paste stops working with the checkbox sometimes still appearing enabled, and the user has no idea why. → **vbuff:** Check the permission state at launch and before every auto-paste, not just at first run. If missing or revoked, show an unmistakable in-app banner with a one-click deep link to the exact OS settings pane and a 'remove and re-add the app' instruction. Detect the 'checkbox on but call fails' corrupted-TCC case and call it out explicitly. _(seen in: Maccy, Paste, Pastebot, Flycut, Clipy)_
139. **Accessibility prompt never appears, blocking first use** `[high]` - The OS permission prompt required for auto-paste never shows, so the user can't grant it and auto-paste can never work, with nothing explaining the dead end. → **vbuff:** Proactively trigger the permission request from a clear in-app onboarding step, and if the OS prompt is suppressed, link directly to the settings pane with screenshots/instructions. Verify the granted state afterward rather than assuming the prompt succeeded. _(seen in: Maccy, Flycut, Clipy)_
140. **Auto-paste simply not implemented on Wayland** `[high]` - On Wayland, apps can't synthesize input or refocus arbitrary windows, so the manager can only copy-to-clipboard and the user must always manually paste, while the app markets auto-paste generally and doesn't say it's unavailable here. → **vbuff:** Detect the session type at runtime and, on Wayland without the needed protocol support, clearly state auto-paste is unavailable and default to a reliable copy-then-manual-paste flow. Where a virtual-keyboard / input-method protocol or a compositor-specific path exists, use it; otherwise don't pretend auto-paste works. _(seen in: CopyQ)_
141. **Paste invocation crashes the receiving app or the session** `[critical]` - Invoking the paste window/IPC under Wayland (or via fragile focus protocols) crashes the target application or the whole desktop session, which is far worse than a paste that just fails. → **vbuff:** Isolate the injection/focus logic from the main process so a failure can't take down the receiver, wrap all compositor/IPC calls in error handling with timeouts, and never block on a focus protocol that can hang. Fuzz/soak-test the paste path on the supported Wayland compositors. _(seen in: CopyQ)_
142. **Global paste lag from holding/delayed-rendering the clipboard** `[high]` - The manager holds a lock or registers as a delayed-render clipboard owner, so every copy/paste system-wide stalls up to a second and typing freezes, degrading the whole machine, not just the app. → **vbuff:** Capture clipboard contents promptly and release ownership; avoid delayed-rendering unless strictly necessary, and serve data quickly when asked. Do all heavy work (hashing, dedup, persistence) off the clipboard-callback thread so the system clipboard chain is never blocked. _(seen in: Windows Clipboard History)_
143. **No PRIMARY vs CLIPBOARD distinction on Linux** `[high]` - Linux has two independent selections (middle-click PRIMARY and explicit Ctrl+C CLIPBOARD); a manager that conflates them captures or pastes from the wrong one, so Ctrl+C content lands only in the middle-click buffer or paste retrieves the wrong selection. → **vbuff:** Track PRIMARY and CLIPBOARD as separate sources, let the user choose which one(s) to record and which one paste targets, and never silently cross-write between them. Default to CLIPBOARD for the history/paste flow while offering PRIMARY capture as an explicit option. _(seen in: X11 core, GPaste, Klipper, CopyQ)_
144. **Content lost because clipboard data dies with the source app** `[critical]` - On X11/Wayland the clipboard only references the source app's memory, so closing that app (or the terminal) loses the copy; a manager that doesn't persist the bytes ends up with un-pasteable empty entries. → **vbuff:** At capture time, eagerly request and store the actual bytes of all offered MIME types into vbuff's own storage, so paste always serves from vbuff's copy and never depends on the source app still being alive. This is core to making paste reliable on Linux. _(seen in: X11 core, Wayland core, GPaste, Klipper, Diodon, Clipman, cliphist)_
145. **X11/Wayland/XWayland clipboard split breaks cross-toolkit paste** `[high]` - Copies don't propagate between Wayland-native apps and XWayland apps, so content copied in one toolkit can't be pasted into the other and a manager watching only one side misses or mis-serves clips. → **vbuff:** Bridge both worlds: monitor and serve the clipboard on both the Wayland and X (XWayland) sides so a clip captured in one is pasteable in the other. Document which bridge mode is active and verify round-trips across the boundary. _(seen in: Diodon, CopyQ, XWayland bridge)_
146. **~1 second UI freeze when pasting right after selecting** `[medium]` - Selecting then immediately pasting an item freezes the UI for around a second (observed on Wayland when the app accesses its own clipboard data), making paste feel broken and unresponsive. → **vbuff:** Perform clipboard set + focus-restore + injection asynchronously without blocking the UI thread, and avoid synchronous round-trips to the compositor for the app's own data. Pre-stage the selected clip's bytes so confirming the paste is instant. _(seen in: CopyQ)_
147. **Misses items on rapid successive copies** `[medium]` - When the user copies several things quickly in a row, intermediate copies don't all make it into history, so the item they want to paste isn't there and they must re-copy. → **vbuff:** Use event-driven clipboard change notifications (not polling) and queue each change for processing so bursts are not dropped. Process the queue on a worker so a slow persistence step can't cause the next change to be missed. Test with rapid programmatic copy bursts. _(seen in: CopyClip, Maccy, Windows Clipboard History)_
148. **Capture silently stops when another tool fights for the clipboard chain** `[high]` - RDP, VMs, AutoHotkey, or a second clipboard manager grab the clipboard chain, and the tray app keeps running while silently recording and serving nothing, so paste history quietly goes stale. → **vbuff:** Monitor clipboard ownership/health and detect when capture has stopped (e.g. no change events for an expected interval despite known copies, or losing the listener chain). Re-register the listener and surface a 'clipboard monitoring interrupted' indicator instead of failing silently. _(seen in: Ditto, CopyQ)_
149. **Two clipboard tools running together corrupt each other's paste** `[medium]` - When vbuff coexists with another manager, they double-capture, fight over ownership, and produce wrong-content or lost-formatting pastes, harming the user even though vbuff alone would work. → **vbuff:** Detect known competing managers / clipboard-owner churn and warn the user, debounce self-induced clipboard changes (ignore changes vbuff itself just made so it doesn't react to its own paste-restore), and avoid aggressive ownership re-assertion that triggers feedback loops. _(seen in: Maccy, Pure Paste, Firefox (Wayland), wl-clip-persist)_
150. **Reacting to vbuff's own clipboard writes (paste feedback loop)** `[medium]` - When the app sets the clipboard to perform a paste (or restore the original), its own clipboard-change handler fires and re-captures that content, creating duplicate/wrong history entries or an ownership fight that eventually breaks copying. → **vbuff:** Tag or track clipboard generations that vbuff itself produced and ignore them in the capture handler. Suppress capture during the paste/restore window so the app never records its own writes. _(seen in: wl-clip-persist, sync tools)_
151. **No paste-stack / multi-paste workflow** `[medium]` - Users assembling several snippets in order must reopen the picker for every single item, with no way to queue clips and paste them sequentially, making multi-snippet assembly tedious. → **vbuff:** Provide a paste-stack / paste-queue: let the user mark several items (or a pinned set) and paste them one after another with repeated paste presses, optionally with a separator. Support paste-in-order and FIFO/LIFO modes so collecting then dumping snippets is one fluid flow. _(seen in: Maccy)_
152. **Picker can't take keyboard focus, so arrow/Enter/Esc don't work** `[medium]` - On some setups the popup can't receive keyboard focus, so the user can't arrow through items, press Enter to paste, or Esc to dismiss, forcing mouse-only operation or making paste impossible by keyboard. → **vbuff:** Ensure the picker reliably accepts keyboard focus on every supported platform (request input focus explicitly; on restricted compositors use the supported focus protocol). Make arrow navigation, Enter-to-paste, and Esc-to-cancel work without the mouse, and test them per platform. _(seen in: Diodon, CopyQ)_
153. **Pasting many/pinned items shuffles order or crashes** `[high]` - Pasting a tab of all-pinned items reorders them unpredictably or crashes the app, so a deliberately ordered set of clips comes out scrambled or the paste destroys the session. → **vbuff:** Preserve explicit user ordering for pinned items as a stable, persisted sequence independent of recency, and make batch paste iterate that order deterministically. Guard batch paste with bounds/error handling so one bad item can't crash the run. _(seen in: Maccy, CopyQ)_
154. **Newest clip not on top / chronological order corrupts** `[medium]` - After pinning/sticky-clip settings go wrong, new clips stop appearing at the top and the list is out of order, so the user reaches for the most-recent item and pastes the wrong one. → **vbuff:** Keep a robust, monotonic capture timestamp/sequence as the source of truth for recency ordering, render unpinned items strictly by it, and keep pinned items in their own separately-ordered region so pin logic can never scramble the recents list. _(seen in: Ditto)_
155. **Blank/empty entries that can't be pasted** `[medium]` - Items appear in history with no visible content and paste nothing, because capture stored a placeholder/reference instead of the data (e.g. lazy Wayland offer not resolved, or a flavor that yielded nothing). → **vbuff:** Validate at capture that at least one usable representation actually contains bytes before adding the entry; if a flavor resolves empty, retry or skip rather than persisting an un-pasteable ghost. Show a clear indicator if a clip's data could not be retrieved. _(seen in: CopyQ, Maccy)_
156. **Duplicate entries on repeated identical copies** `[low]` - Copying the same text several times creates many identical rows, cluttering the list so the user scrolls past redundant entries to find what to paste. → **vbuff:** De-duplicate by content hash across stored representations: if an incoming clip matches an existing one, bump the existing entry to the top (refresh its timestamp) instead of inserting a new row. Make dedup the default with a toggle for users who want raw history. _(seen in: Maccy)_
157. **No bulk select/paste/delete** `[low]` - Users can't act on multiple items at once, so assembling multiple snippets or cleaning up history is a slow one-at-a-time chore. → **vbuff:** Support multi-select with keyboard and mouse, and offer batch actions: paste-all (into the stack), copy-all, and delete-all-selected. Tie batch paste into the paste-stack so multi-snippet assembly is a single gesture. _(seen in: Maccy)_
158. **Picker dismiss timing makes paste require extra clicks** `[medium]` - The picker either dismisses too late (still on screen, intercepting the focus that should go to the target) or requires a click to confirm before paste, adding clicks and breaking the keyboard-only flow. → **vbuff:** On confirm, hide the picker first, restore target focus, then inject paste, in that order, with no intermediate user action required. Make selection-confirm a single keypress; never require a mouse click to complete a keyboard-initiated paste. _(seen in: Ditto, CopyQ)_
159. **Refusing legitimate content so it can't be captured or pasted** `[low]` - The manager rejects clips containing certain special characters or shapes, so legitimate copies silently never enter history and can't be pasted back later. → **vbuff:** Treat clipboard payloads as opaque bytes per MIME type; do not validate, sanitize, or reject content based on characters. Store and re-serve exactly what was captured, including arbitrary Unicode, control characters, and binary blobs. _(seen in: GPaste)_
160. **Line-ending / whitespace mangling on paste** `[medium]` - The app normalizes CRLF<->LF, trims trailing whitespace, or converts tabs/spaces when storing or pasting, so code and structured text paste back subtly altered and break (e.g. indentation-sensitive languages, Windows files). → **vbuff:** Preserve clipboard text byte-for-byte, including line endings and whitespace. Offer any normalization (trim, convert EOL) only as an explicit opt-in transform the user invokes, never as a silent default on the paste path.
161. **Pasting via slow per-character type-out instead of a real paste** `[medium]` - Some tools 'paste' by synthesizing each character as a keystroke; this is slow for large clips, fires per-character autocomplete/auto-indent in editors, drops characters under load, and mangles the result. → **vbuff:** Always paste through the real clipboard + paste action, not character-by-character typing. Reserve simulated typing only for explicit targets that genuinely reject clipboard paste, and even then warn it may be lossy.
162. **Modifier keys held during the hotkey leak into the paste** `[medium]` - If the user is still holding the summon modifiers when the app injects Cmd/Ctrl+V, the held modifiers combine with the synthetic paste (e.g. Cmd+Shift+V triggers paste-and-match-style or a different command), so the paste does the wrong thing. → **vbuff:** Before injecting the paste, wait for / force-clear the physical modifier state (or read current modifiers and compensate) so only the intended paste combination reaches the target. Don't fire the synthetic paste while user-held modifiers are still down.
163. **No feedback when paste succeeds or fails** `[medium]` - The user can't tell whether the paste actually went through, so on silent failure they don't notice the content is missing until later, and on success they may redundantly paste again. → **vbuff:** Give lightweight, non-intrusive confirmation appropriate to the result: succeed silently into the target (the inserted text is its own confirmation) but on a detected failure show a clear, actionable message. Never leave a failed paste indistinguishable from a successful one.
164. **Paste timing too fast for the target to be ready after refocus** `[medium]` - The app refocuses the target and immediately injects paste before the window has actually become active / the caret is restored, so the paste is dropped or goes to the previous focus, intermittently and app-dependently. → **vbuff:** After issuing focus restore, wait for confirmation that the target is frontmost/active (event or short poll) before injecting, rather than firing on a fixed tiny delay. Make any fallback delay configurable for slow apps and document it. _(seen in: CopyQ, cliphist)_

## 5. Data type & format mistakes

165. **Text-only capture (drops all non-text flavors)** `[critical]` - The clipboard offered image/png, RTF, HTML, file URLs, but the manager only read text/plain. Copying a formatted table, an image, or a file silently lands as bare text or an empty entry, so the user cannot paste back what they actually copied. → **vbuff:** Enumerate ALL available representations the source advertises (NSPasteboard types / Win clipboard formats / Wayland MIME offers) and store every flavor of an item together as a multi-representation record. Never reduce to text/plain at capture time. _(seen in: Clipman, GPaste, Diodon, Flycut, Jumpcut, various GTK managers)_
166. **Storing the wrong representation (HTML tag instead of pixels)** `[high]` - Images copied from a browser are offered as both text/html and image/png; the manager grabs the text/html offer, so pasting into an image editor yields markup or fails. The user sees an image in history but pastes garbage. → **vbuff:** Rank representations by fidelity per content class: for visual content prefer image/png or image/tiff over text/html; for tabular prefer the richest table format. Detect the dominant type from the offer set, not from arbitrary enumeration order. _(seen in: cliphist, clipman, wl-clipboard sync tools, CopyQ)_
167. **Requiring a separate watcher per data type** `[high]` - Keeping images requires running 'wl-paste --type image --watch cliphist store' alongside the text watcher. Users who set up only the text watcher silently lose every image they copy, with no indication anything is missing. → **vbuff:** A single capture path must subscribe to every offered MIME type in one pass. Do not require per-type daemons or per-type configuration; type coverage is the manager's job, not the user's. _(seen in: cliphist, clipman)_
168. **No content-type detection / mislabeled entries** `[high]` - The manager guesses type from the first available flavor or file extension and gets it wrong: a PNG stored as 'text', a CSV treated as plain prose, JSON shown without structure. Wrong preview, wrong paste behavior, broken search. → **vbuff:** Detect type by content sniffing (magic bytes for binary, charset/BOM probing for text, structural probes for HTML/RTF/JSON/CSV) combined with the declared MIME/UTI, not by extension or flavor order. Record a canonical normalized type plus the raw flavors. _(seen in: Windows 11 Win+V, ClipAngel, Comfort Clipboard)_
169. **Assuming clipboard text is UTF-8 (encoding mojibake)** `[critical]` - Windows clipboard CF_TEXT is in the system code page and CF_UNICODETEXT is UTF-16LE; X11 selections may be Latin-1 or other charsets. Decoding everything as UTF-8 turns accented characters, Cyrillic, CJK, and emoji into mojibake (Ð¿ÑÐ¸Ð²ÐµÑ) that is permanently corrupted on store. → **vbuff:** Always prefer the Unicode flavor (CF_UNICODETEXT / UTF8_STRING / public.utf8-plain-text). When only a legacy flavor exists, decode using its declared charset/code page, then re-encode to UTF-8 internally. Never assume a fixed input encoding.
170. **Dropping or mangling the BOM** `[medium]` - Text copied from sources that include a UTF-8/UTF-16 byte-order mark gets either stripped (breaking tools that need it) or double-encoded into a visible  prefix. Pasting back no longer round-trips byte-for-byte. → **vbuff:** Treat BOM as part of the encoding-detection step only: strip it after using it to identify UTF-16/UTF-8, store text as canonical UTF-8 without BOM internally, and reproduce the original byte sequence faithfully when the user requests an exact paste.
171. **Lossy line-ending normalization without preserving original** `[high]` - The manager rewrites CRLF to LF (or LF to CRLF) on capture. Code pasted into Windows tools, fixed-width data, or here-docs breaks; diffs show phantom changes; pasting into a terminal sends extra ^M. The user cannot get the bytes they copied. → **vbuff:** Store the original byte stream verbatim. Offer line-ending conversion only as an explicit, opt-in paste transform (paste as LF / paste as CRLF), never as a silent capture-time mutation. Default paste reproduces source bytes exactly.
172. **Trailing-whitespace / NUL stripping that corrupts binary or significant text** `[high]` - Auto-trimming whitespace or chopping at the first NUL byte truncates binary blobs, base64 with trailing newline, password strings with intentional spaces, or null-delimited data. Paste produces silently truncated or altered content. → **vbuff:** Never auto-trim or NUL-truncate stored content. Distinguish binary from text up front; store binary as opaque bytes. Offer trimming as an explicit per-paste action, never as a default applied at capture.
173. **Losing rich text on round-trip (paste comes back plain)** `[medium]` - A clip captured with RTF/HTML formatting pastes as plain text or with broken styling because only text/plain was stored or the rich flavor was discarded. Users lose colors, bold, links, list structure. → **vbuff:** Persist the rich flavors (RTF, HTML, attributed-string) alongside plain text and re-offer all of them at paste time so the destination app picks the best one. Keep formatting lossless through the store/restore cycle. _(seen in: ClipAngel, Comfort Clipboard, Windows 11 Win+V, cliphist, Maccy, CopyQ)_
174. **No reliable 'paste as plain text' path** `[high]` - When the user explicitly wants unformatted text, the manager either offers no plain-text path or its plain-text path mangles content (e.g. strips file URLs, drops newlines). Pasting formatted text into a plain field injects unwanted styling. → **vbuff:** Always derive a clean plain-text representation from the richest source and expose a first-class, keyboard-reachable 'paste without formatting' action that emits only text/plain, without touching the stored rich copy. _(seen in: Maccy, Paste, Alfred, Pastebot)_
175. **Storing huge images uncompressed (raw TIFF/BMP)** `[high]` - Retina screenshots arrive as 10-30 MB uncompressed TIFF; storing raw clipboard bytes balloons the DB to hundreds of MB/day and gigabytes/week, slowing search and risking disk-full. The data type itself drives the bloat. → **vbuff:** Transcode large raster images to a compressed format (PNG lossless, or quality-bounded for thumbnails) before persisting, while keeping the original flavor available on demand or re-derivable. Cap per-item raw retention and store image blobs externally, not inline in the row. _(seen in: Ditto, Maccy, CopyQ, generic)_
176. **No preview for non-text content** `[medium]` - Images, files, and rich data appear as blank rows, a generic icon, or a raw type name. The user cannot tell items apart and must paste blindly to find the right one. → **vbuff:** Generate and cache a small thumbnail for images, a filename+icon+size summary for files, a rendered snippet for HTML/RTF, and a truncated text preview for long text. Render previews from the cached thumbnail, never by loading the full blob into the list view. _(seen in: Alfred, Raycast, Maccy, CopyQ)_
177. **Blank/empty entries for content it can't represent** `[medium]` - Something was copied but the row shows nothing and is un-pasteable, because the manager stored an unrecognized flavor as empty or failed silently on an unknown type. Confusing and useless. → **vbuff:** If no known representation can be parsed, still store the raw bytes with their MIME/UTI label and show a typed placeholder ('image', 'file', 'application/x-…') so the item remains pasteable as its original flavor even when not previewable. _(seen in: CopyQ, Maccy)_
178. **Dropping file references / file URLs on paste** `[high]` - Copying files in the file manager yields file URLs / CF_HDROP; a manager that reduces these to a text path string breaks paste-into-folder operations and 'paste without formatting' that strips file URLs makes file paste fail entirely. → **vbuff:** Capture file references as the native file-list flavor (NSFilenamesPboardType / CF_HDROP / text/uri-list) and re-offer that exact flavor on paste so destination apps treat them as files, not as text. Keep a text path as a secondary representation only. _(seen in: Maccy, Paste, Alfred, Pastebot)_
179. **Per-item size cap silently truncates or rejects large clips** `[medium]` - Hard per-item limits (e.g. Win+V's ~4MB, Ditto/CopyQ caps) silently drop or truncate large text, big images, or files. The user believes a large copy succeeded but the stored item is partial or absent. → **vbuff:** Make the size limit generous and configurable, and when an item exceeds it, surface an explicit indicator (stored truncated / skipped) rather than silently discarding. Prefer streaming large items to external storage over dropping them. _(seen in: Windows 11 Win+V, Ditto, ClipboardFusion)_
180. **Ignoring delayed-rendering / promised data** `[high]` - Apps often place promised (delayed-render) data on the clipboard and only produce bytes when a consumer requests them. A manager that reads eagerly without honoring the promise protocol gets empty or partial content for spreadsheets, images, and large objects. → **vbuff:** Honor delayed-rendering: request the actual bytes through the platform's render-format callback (WM_RENDERFORMAT / NSPasteboard provider / Wayland data-source) at capture time, and handle the case where rendering is asynchronous before treating the item as final. _(seen in: Windows 11 Win+V)_
181. **Wayland: losing content when the source app closes** `[critical]` - On Wayland (and X11) the data lives in the source app's memory; when the app or terminal closes, the offer is destroyed and the copy is gone. The user copies, quits the app, and the clip vanishes. → **vbuff:** On capture, eagerly read the full bytes of every offered MIME type into vbuff's own storage immediately (acting as a persistence layer), rather than holding a reference. This is the core reason a manager exists on Wayland; do not defer the read. _(seen in: X11 core, Wayland core, GPaste, Klipper, Diodon, Clipman, wl-clip-persist)_
182. **PRIMARY vs CLIPBOARD selection confusion (Linux)** `[high]` - Linux has two independent selections: PRIMARY (select-to-copy, middle-click paste) and CLIPBOARD (Ctrl+C). Mixing them up means Ctrl+C content lands only in the middle-click buffer, or paste retrieves the wrong selection. Data appears lost or wrong. → **vbuff:** Model PRIMARY and CLIPBOARD as distinct sources, capture from both with clear labeling, and let the user choose which one a stored item targets on paste. Never silently conflate the two buffers. _(seen in: X11 core, GPaste, Klipper, CopyQ)_
183. **X11/Wayland/XWayland clipboard split not bridged** `[high]` - Content copied in a Wayland-native app isn't visible to XWayland apps and vice versa; a manager listening in one mode can't see the other's clipboard, so clips appear to not propagate between toolkits. → **vbuff:** Bridge both transports: monitor and write through both the Wayland (wlr/ext-data-control) and X11/XWayland selection paths so any captured item can be pasted into either toolkit. Normalize MIME types across the bridge. _(seen in: Diodon, CopyQ, XWayland bridge)_
184. **Wrong UTI/MIME mapping across platforms** `[high]` - Cross-platform type identifiers differ (UTI 'public.png' vs MIME 'image/png' vs Windows CF_DIB). A naive 1:1 or missing mapping means a clip captured on one OS becomes an unknown/unpasteable type on another in a sync scenario, or types collide. → **vbuff:** Maintain an explicit canonical internal type taxonomy and bidirectional mapping tables between UTIs, Windows clipboard format names/IDs, and MIME types. Round-trip through the canonical type so identity is preserved across platforms.
185. **CF_DIB alpha/stride mishandling on Windows** `[medium]` - Windows device-independent bitmaps (CF_DIB / CF_DIBV5) have row padding (stride alignment), bottom-up row order, and ambiguous alpha. Naive parsing yields upside-down images, color shifts, or lost transparency when storing or re-offering. → **vbuff:** Parse DIB headers correctly (handle BITMAPV5HEADER, row stride to 4-byte boundaries, top-down vs bottom-up, premultiplied alpha), convert to a well-defined internal RGBA, and regenerate a correct DIB on paste. Prefer CF_DIBV5 to preserve alpha.
186. **Treating multi-flavor items as separate clips (or keeping only one)** `[high]` - A single copy carries text/plain + text/html + image atomically; storing each flavor as its own history row (or keeping just one) breaks paste, because the destination can no longer negotiate the best representation and dedup/recall get confused. → **vbuff:** Model a clip as one record containing the full set of co-offered representations captured atomically. Re-offer the whole set on paste so the receiving app negotiates the format it wants.
187. **Spreadsheet/tabular data flattened to lossy text** `[medium]` - Excel/Sheets copies offer multiple rich flavors (HTML table, RTF, Biff/SYLK, image). Storing only tab-separated text loses cell types, formulas, number formatting, and merged cells; pasting back into a spreadsheet yields plain strings. → **vbuff:** Preserve the rich tabular flavors (HTML table, native spreadsheet formats) alongside the TSV fallback, and re-offer them so paste into a spreadsheet restores structure. Detect tabular content for a structured preview. _(seen in: Windows 11 Win+V)_
188. **Mangling emoji / surrogate pairs / combining sequences** `[medium]` - Naive char-by-char or UTF-16-index truncation splits surrogate pairs and grapheme clusters, corrupting emoji, flags, skin-tone modifiers, and combining accents into replacement characters or broken glyphs in previews and on paste. → **vbuff:** Operate on Unicode scalar values / grapheme clusters (e.g. Rust char and grapheme segmentation) for any truncation, search, or preview. Never slice text at arbitrary byte or UTF-16 code-unit boundaries.
189. **Bidirectional / RTL and control-character handling in previews** `[medium]` - Pasted text containing RTL overrides, zero-width characters, or bidi control codes can render previews misleadingly (Trojan-source style) or let invisible characters hide in stored clips, surprising the user on paste. → **vbuff:** Render previews with explicit display of bidi/zero-width/invisible control characters (or a 'contains hidden characters' badge), while storing the exact bytes. Surface invisible content so the user knows what they will actually paste.
190. **Re-encoding / lossy transcoding on store** `[high]` - Converting captured images to JPEG, downscaling, or re-encoding text through an intermediate charset to 'save space' permanently degrades the data. The user pastes a lower-quality or altered version than they copied. → **vbuff:** Default to lossless storage (PNG for raster, exact bytes for text/binary). Any lossy optimization must be opt-in, applied only to derived previews/thumbnails, and never replace the canonical stored representation used for paste.
191. **Charset mislabel: HTML/CSV/JSON stored without its declared encoding** `[medium]` - HTML fragments and data files carry an encoding declaration (meta charset, response charset) that differs from the clipboard text flavor. Ignoring it and assuming one charset corrupts non-ASCII content when the fragment is later re-offered or previewed. → **vbuff:** When a structured text flavor declares its own charset, honor that declaration for decode/preview, store the canonical Unicode form, and reproduce the original byte sequence on paste. Track per-representation encoding metadata.
192. **Windows HTML clipboard fragment header corruption** `[medium]` - The Windows 'HTML Format' clipboard type uses a byte-offset header (StartFragment/EndFragment/StartHTML). Storing or rebuilding it with wrong offsets after any text edit produces invalid HTML that pastes as broken markup or plain text. → **vbuff:** Parse the CF_HTML header, keep fragment boundaries consistent, and recompute the byte offsets correctly whenever the payload is regenerated. Validate the header round-trips before re-offering it on paste.
193. **No type-aware dedup (binary vs text compared wrong)** `[low]` - Dedup that hashes only the text flavor treats two different images (or an image and its text alt) as duplicates, or fails to dedup identical binary clips, cluttering history or silently dropping a distinct item. → **vbuff:** Dedup on a content hash computed over the full set of representations (or the dominant binary blob), not just text/plain. Bump/reposition an existing identical multi-flavor item instead of creating a near-duplicate. _(seen in: Maccy)_
194. **Loading full blobs into memory for every list render** `[high]` - Rendering the history view by decoding each item's full image/text blob (especially with inline-blob storage) spikes memory and freezes the UI as history grows, because the data type's size is paid on every open. → **vbuff:** Store large blobs externally (file-backed), keep only metadata + a small cached thumbnail/preview in the list model, and lazy-load the full payload only when an item is actually pasted or expanded. _(seen in: Maccy, CopyQ)_
195. **Image-derived/OCR text not stored as a searchable representation** `[medium]` - When OCR or text recognition runs on copied images, the extracted text either isn't kept as a searchable field or older/image-derived items can't be searched, so search across mixed content types is broken. → **vbuff:** If OCR is enabled (opt-in for privacy), store the recognized text as a separate searchable representation attached to the image record, indexed alongside text clips, while keeping it clearly marked as derived and excludable. _(seen in: Paste, Alfred, Maccy, Raycast)_
196. **Rejecting content with special characters** `[low]` - Some managers refuse to store text containing certain control or special characters, so legitimate copies (code with control chars, unusual Unicode) silently never enter history and the user thinks the copy failed. → **vbuff:** Accept arbitrary byte content. Treat text as opaque Unicode/bytes and never filter, reject, or sanitize at capture based on character content; store everything and handle display safely at render time. _(seen in: GPaste)_
197. **Metadata stripped from images (EXIF/ICC/orientation)** `[medium]` - Re-encoding or naive image handling drops EXIF orientation (so the pasted image is rotated), ICC color profiles (so colors shift), and other metadata. The pasted image differs visibly from what was copied. → **vbuff:** Preserve image metadata through storage: keep the original encoded bytes as the canonical paste source, and when generating previews honor EXIF orientation and embedded ICC profiles so previews match the source.
198. **Truncating long single-line text breaking structure** `[medium]` - Very long single-line content (minified JSON, base64, a long URL, a SQL dump) gets truncated for storage or display in a way that loses the tail, so pasting yields incomplete data and the structure is destroyed. → **vbuff:** Store the full content regardless of line length; truncate only the visual preview (with a clear 'truncated' indicator) while keeping the complete payload for paste. Never let display limits affect stored bytes. _(seen in: CopyQ)_
199. **No distinction between binary and text leading to garbage previews** `[medium]` - Binary clipboard data (custom app pasteboard types, serialized objects, archives) shown as decoded 'text' renders as garbage/mojibake in the list and may be corrupted if the manager applies text transforms to it. → **vbuff:** Classify each representation as text or binary via charset validity probing; render binary as a typed/hex/size summary, never as decoded text, and apply zero text transforms (trim, line-ending, normalization) to binary blobs.
200. **Unicode normalization form changed silently (NFC/NFD)** `[medium]` - Capturing from macOS (often NFD/decomposed filenames) and re-emitting as NFC (or vice versa) silently changes the byte content of paths and text, breaking exact-match comparisons, filename paste, and round-trip fidelity. → **vbuff:** Do not normalize Unicode by default; store and reproduce the exact code-point sequence. If normalization is needed for search/dedup, compute it on a derived index only, never on the stored canonical bytes used for paste.

## 6. Organization mistakes

201. **No pinning or favorites at all** `[high]` - Items the user wants to keep handy (a frequently-pasted address, a meeting link, a code snippet) sink down the timeline and get evicted by the rolling history limit, forcing the user to re-copy them constantly. The clipboard becomes useless for anything beyond the last few seconds. → **vbuff:** vbuff ships first-class pinning: a pinned item is exempt from history eviction, dedup-bumping, and auto-expiry, stored in a separate persistent table from rolling history so cap policies never touch it. Pin/unpin is a single keystroke from the picker. _(seen in: Flycut, Windows 11 Win+V (history wiped unless pinned))_
202. **Pins silently lost on update/reboot/OS upgrade** `[critical]` - Users curate a set of pinned items over weeks, then an app update, a reboot, or an OS upgrade wipes them with no warning and no recovery path. The one part of history the user explicitly marked as important is the part that vanishes. → **vbuff:** Store pins in a dedicated, schema-versioned table separate from volatile history; write them through a durable transaction (WAL + fsync on pin), and run a migration that preserves pins across schema changes. Add an automatic pre-migration backup of the pins table and a restore command. _(seen in: Maccy, Windows 11 Win+V, CopyQ)_
203. **Pinned items cannot be reordered** `[medium]` - Pins accumulate in copy-order, so the most-used pin can sit at the bottom of a long pinned list. The user has to scan or scroll every time, defeating the speed pinning was supposed to provide. → **vbuff:** Give pins an explicit user-controllable sort order (drag, or move-up/move-down keys) persisted as an integer rank column independent of timestamp. Default new pins to the top, but let the user lock an order that survives restarts. _(seen in: Maccy)_
204. **Pin order shuffles on its own** `[high]` - Pinned items silently switch order after a reboot or after pasting a tab of all-pinned items, so the muscle-memory position the user relied on (item 1 = work email, item 2 = home address) becomes unreliable and they paste the wrong thing. → **vbuff:** Persist an explicit rank, never derive pinned order from insertion time, last-access time, or row id at load. Load pins ordered by the stored rank column with a stable tiebreaker, and never mutate rank as a side effect of paste or app restart. _(seen in: Ditto, Maccy)_
205. **No tags or labels** `[medium]` - Everything lives in one undifferentiated stream. A user with snippets for work, personal, code, and addresses has no way to filter to just one category, so finding the right clip means scrolling a flat list or hoping search guesses right. → **vbuff:** Support lightweight tags (free-form labels) on any item, with a tag filter in the picker and tag-scoped search (e.g. type a tag prefix to narrow). Tags are many-to-one, persisted per item, and shown as chips in the list.
206. **No folders or collections** `[medium]` - Without any container abstraction, users who want to assemble a reusable set (onboarding boilerplate, a set of canned replies, signing templates) must keep re-copying each piece. There is no concept of a named group of saved clips. → **vbuff:** Provide folders/collections (named containers) that hold pinned or saved items, browsable as a tree or list in the picker. Allow nesting one level deep and moving items between folders. Keep them separate from the rolling timeline so they are not subject to eviction.
207. **Flat unmanageable single list** `[high]` - The only view is one long chronological list. Once it has thousands of entries it is impossible to navigate by anything but recency or search; there is no grouping, no sections, no overview, so the history is effectively write-only. → **vbuff:** Offer multiple structured views over the same data: chronological timeline, pinned, by tag, by type (text/image/link/file), and grouped by source app. Default to timeline but make the alternate views one keystroke away. Group the timeline into Today / Yesterday / Earlier sections. _(seen in: Maccy, CopyClip, Ditto)_
208. **No separate pasteboards / stacks** `[medium]` - Power users who want to collect several clips and paste them in sequence (filling a form, assembling a document) have only the single global history. They cannot set aside a working set without it being polluted by every unrelated copy they make in between. → **vbuff:** Support multiple named pasteboards/stacks the user can switch between, plus an append-mode 'collect' stack where successive copies accumulate and a sequential-paste mode that walks the stack item by item. The active history stays separate from these working sets. _(seen in: Pastebot (paste stacks done well), Ditto)_
209. **No notes or annotations on items** `[medium]` - A saved clip like 'sk_live_...' or a long URL is opaque in the list; the user cannot remember what it was for. Without a label or note, identifying the right saved item among many similar ones means pasting each to check. → **vbuff:** Allow an optional user-supplied title/note per item, shown as the primary label in the list with the raw content as a secondary line. Saved/pinned items default to an editable title derived from content but overridable. Make the note searchable.
210. **No custom display name, only raw content shown** `[medium]` - List rows show raw first-line content, so multiline snippets, base64 blobs, or near-identical URLs are indistinguishable at a glance. The user scrolls and squints instead of recognizing items instantly. → **vbuff:** Render a smart label per item: user title if set, else a type-aware summary (URL host+path, first line for text, dimensions+thumbnail for images, filename for files). Truncate cleanly with a tooltip/preview for the full content. _(seen in: CopyQ (blank/opaque rows), Maccy)_
211. **Clunky reordering of items** `[medium]` - Moving an item to a better position is impossible or requires deleting and re-copying, which also loses its metadata (pin state, tag, note). Curation is so painful users give up and tolerate a messy list. → **vbuff:** Support direct reordering of pinned/saved items via drag and keyboard move commands, operating on a persisted rank without touching content or metadata. Reordering never re-timestamps or duplicates the item. _(seen in: Paste, Maccy)_
212. **Duplicate entries clutter the list instead of de-duplicating** `[low]` - Copying the same text repeatedly creates many identical rows, so the history fills with noise, the real diversity of recent clips is pushed off the visible window, and search returns the same hit many times. → **vbuff:** De-duplicate by content hash: when an identical item is copied, bump the existing entry to the top and update its timestamp/use-count instead of inserting a new row. Make dedup the default with an opt-out, and preserve the existing item's pin/tag/note on bump. _(seen in: Maccy)_
213. **Dedup destroys pinned/annotated metadata on re-copy** `[high]` - A naive 'remove old, insert new' dedup deletes the existing row and creates a fresh one, silently wiping the pin, tag, or note the user attached. The user discovers their curated item reverted to plain status. → **vbuff:** Implement dedup as an in-place update (bump timestamp, increment use-count) on the existing row, explicitly preserving pin state, tags, notes, and rank. Never delete-and-reinsert for dedup.
214. **No bulk selection or bulk operations** `[medium]` - Cleaning up, deleting a range of junk, tagging a batch, or assembling several snippets must be done one item at a time. Maintenance becomes so tedious the list is never cleaned, accelerating the descent into an unmanageable flat list. → **vbuff:** Support multi-select (shift/ctrl-click and shift+arrows) with bulk delete, bulk pin, bulk tag, bulk move-to-folder, and bulk copy/paste-as-one. Provide a 'select all matching current filter' action. _(seen in: Maccy)_
215. **No way to merge or paste multiple items together** `[low]` - Assembling content from several clips (e.g. paste three snippets into one field) requires switching to the picker once per item, breaking flow. There is no concept of combining selected items. → **vbuff:** Add 'paste selected as one' that concatenates multi-selected items with a chosen separator (newline, space, custom), and a sequential-paste mode where each paste invocation advances to the next item in the selection/stack.
216. **No type-based filtering or sections** `[medium]` - Text, links, images, files, and color codes are all mixed into one timeline. A user hunting for an image they copied must scroll past dozens of text clips, and there is no 'images only' or 'links only' view. → **vbuff:** Classify each item by content type at capture and expose per-type filters and quick-jump keys (e.g. one key for links, one for images). Show a small type icon per row and allow grouping the list by type. _(seen in: Alfred, Maccy)_
217. **No grouping by source application** `[low]` - The user often remembers 'I copied it from the browser' or 'from the terminal' but not when. With no source grouping or source filter, that strong recall cue is wasted and they fall back to scrolling. → **vbuff:** Record the source app per item and offer a source filter and group-by-source view, with the source app icon shown on each row. Combine with type and time filters for fast narrowing.
218. **Only sort is reverse-chronological** `[low]` - The list can only be viewed newest-first. A user who wants their most-used clips, alphabetical snippets, or oldest-first cannot get there, so frequently needed items are no easier to reach than one-off junk. → **vbuff:** Offer multiple sort orders: recency, frequency (use-count), alphabetical, and manual rank for pinned/saved sets. Persist the chosen sort per view. Track a use-count so a 'most used' sort is meaningful.
219. **Saved/pinned items intermixed with volatile history** `[high]` - Curated keepers and disposable recent copies share one list, so important pins are buried among transient junk and a 'clear history' can wipe them, while disposable clips clutter the view of the things that matter. → **vbuff:** Maintain two conceptually and physically separate stores: volatile rolling history (subject to cap and expiry) and a durable saved/pinned/foldered library (never auto-evicted). Show them in distinct sections or views; 'clear history' only touches the volatile store. _(seen in: Paste, Maccy)_
220. **'Clear all' nukes pinned and saved items too** `[high]` - A user clears history to remove a sensitive clip or just declutter, and their entire pinned/saved library is destroyed along with it because the operation does not distinguish curated from volatile items. → **vbuff:** Scope 'clear history' to the volatile store only and require explicit, separately-labeled confirmation to delete saved/pinned items. Offer 'clear unpinned' as the default destructive action and never include pins in a blanket clear. _(seen in: Maccy)_
221. **Hardcoded or low history cap silently evicts wanted items** `[medium]` - A fixed, low item cap drops older clips with no warning, so a user who expected something from an hour ago to still be there finds it gone. The cap is invisible and unconfigurable, so users cannot reason about what survives. → **vbuff:** Make the history cap configurable (item count and/or age and/or total size) with a clear default, surface the current cap and how full it is, and never count pinned/saved items against it. Warn before first eviction if the user has never seen the cap. _(seen in: Maccy, Windows 11 Win+V)_
222. **No expiry policy, history grows into an unmanageable wall** `[medium]` - With no time-based pruning the volatile list grows unbounded, so old irrelevant clips accumulate forever, making the list slower to scan and search noisier, and indefinitely retaining things the user never wanted kept. → **vbuff:** Support optional age-based auto-expiry for volatile (unpinned) items (e.g. keep last N days), configurable and off by default for privacy-neutral users but recommended in onboarding. Pinned/saved items are never expired.
223. **Search cannot scope to pins, tags, folders, or type** `[medium]` - Search runs only over the flat blob of all content, so a query that would be unique within 'work tags' or 'links' returns dozens of unrelated hits across the whole history, and the organizational structure provides no help in narrowing results. → **vbuff:** Make search composable with the organizational axes: filter by tag/folder/type/source first, then search within that scope, and support inline scope tokens (e.g. tag:work, type:link). Search the title/note in addition to content. _(seen in: Paste, Alfred)_
224. **No quick-access numbered or hotkey slots** `[medium]` - Even when items are pinned, reaching a specific one means opening the picker and navigating to it. There is no direct binding from 'the address I paste 20 times a day' to a single key, so a saved item is barely faster than a regular one. → **vbuff:** Allow assigning items to numbered slots (paste slot 1..9 directly) and/or per-item global hotkeys, so a chosen saved clip is pasteable with one chord without opening the picker. Show the assigned slot/hotkey on the row.
225. **No snippet library / static saved text separate from history** `[medium]` - Reusable boilerplate (signatures, addresses, canned replies) has to live in the same volatile history as ad-hoc copies, where it competes for cap space, gets reordered, and is hard to find, so users keep boilerplate in a notes app and copy-paste manually instead. → **vbuff:** Provide a dedicated snippet library: named, foldered, never-evicted static entries that the user authors directly (not just captured), browsable and pasteable from the picker. Allow promoting a history item into a snippet in one action. _(seen in: Clipy (snippets, but crash-prone editor))_
226. **Reordering or organizing re-timestamps items and corrupts chronology** `[medium]` - Acting on an item for organization (pinning, tagging, moving) bumps its timestamp, so it jumps to the top of the chronological view and the real copy-order history becomes a lie. Newest-on-top stops meaning newest. → **vbuff:** Separate 'created/copied time' from 'last modified' and 'last used'; organizational actions update only the relevant field and never the copy-time used for chronological sorting. The timeline always reflects true capture order. _(seen in: Ditto (ordering corrupts with sticky/pinned settings))_
227. **No drag-out / drag-in organization** `[low]` - Users cannot drag an item out to a folder or app, nor drag external content in to save it, so the manager is an island; moving content between the manager and the rest of the OS requires copy-paste round-trips that themselves pollute history. → **vbuff:** Support dragging items out of the picker (to apps, to the Finder/Explorer as a file for images/files) and dragging content in to add it as a saved item without it passing through the volatile clipboard. Dragging out does not re-copy or re-order.
228. **No preview, so organizing requires pasting to identify** `[medium]` - Without an inline preview of full text or image, users cannot tell similar items apart while organizing (which of three long URLs is the right one to pin?), so curation devolves into trial-and-error pasting and the list stays disorganized. → **vbuff:** Show a rich preview pane (full text, image thumbnail, file metadata) for the focused item in the picker, updating as the user navigates, so identification for pin/tag/delete decisions needs no paste. Keep it keyboard-driven. _(seen in: Paste, Maccy)_
229. **Organization is per-device and not portable** `[medium]` - Pins, tags, folders, notes, and snippets are tied to one install, so reinstalling or moving to a new machine loses all curation even when the user kept a backup of raw history. The organizational work is the hardest to recreate and the least protected. → **vbuff:** Make the organizational layer (pins, tags, folders, notes, snippets, ranks) exportable/importable as a portable document independent of volatile history, with explicit backup. If sync exists, sync the organizational metadata, not just raw clips.
230. **No empty/onboarding guidance toward organizing** `[low]` - The manager presents a bare list with no hint that pinning, tagging, or snippets exist, so users treat it as a dumb buffer, never organize, and the history degrades into the very flat unmanageable wall the features were meant to prevent. → **vbuff:** Surface lightweight affordances: a one-time hint to pin frequently re-copied items, a visible 'pin' action on every row, and a suggestion to promote an item to a snippet when the same content is copied repeatedly. Make the organizing path discoverable in the default UI.
231. **Deleting an item leaves dangling references in folders/collections** `[low]` - When an item is deleted from history but it was also placed in a folder or collection, the folder shows a broken/empty slot or the deletion fails to propagate, so collections silently rot and the user cannot trust them. → **vbuff:** Model folder membership with foreign keys and cascade or reconcile on delete; deleting a volatile item that is also saved/foldered should keep the saved copy (promote), and removing from a folder should be distinct from deleting the item. Never leave dangling references.

## 7. Snippet & expansion mistakes

232. **No placeholders or variables in snippets** `[critical]` - Snippets are static dead text. The user expands a template and then has to manually hunt-and-edit the name, date, ticket number, or recipient every single time, which defeats the point of snippets for anything but trivial constants. → **vbuff:** Support typed placeholders from day one: literal fields (%name%), dynamic ones (date/time with format strings, clipboard contents, last-snippet output), and a defined-once-reuse-everywhere variable so the same value (e.g. {first_name}) repeated in a template is entered once. Parse the placeholder grammar at save time and surface errors then, not at expansion time. _(seen in: Maccy, Flycut, CopyClip, Ditto (basic snippets))_
233. **No interactive fill-in form for placeholders** `[high]` - Even when variables exist, the user has no prompt to fill them. Expansion either dumps raw %placeholder% tokens into the document or silently inserts empty strings, so the user must go back and find each gap by hand. → **vbuff:** On expansion, if a snippet has unfilled placeholders, show a small inline form listing every field with labels, defaults, and dropdown/multiple-choice options where defined. Tab between fields, Enter to commit, Esc to cancel cleanly. Remember last-used values per snippet as defaults.
234. **No cursor-position marker after expansion** `[high]` - After a snippet expands, the caret lands at the very end of the inserted text. For an email-signature or HTML-tag-pair snippet the user wants the cursor in the middle (between the tags, after 'Dear '), and has to arrow-key back manually every time. → **vbuff:** Support an explicit caret token (e.g. %|% or $cursor$) that places the insertion point after expansion. Support a selection token pair so a region can be pre-selected for immediate overtype. If multiple caret tokens are present, define deterministic behavior (first one wins) rather than silently dropping them.
235. **No nested or embedded snippets** `[medium]` - Common building blocks (a signature, a legal disclaimer, today's date block) get copy-pasted into dozens of snippets. When the block changes, the user must edit every snippet that embedded it, and they inevitably miss some, shipping stale text. → **vbuff:** Allow snippets to reference other snippets by key/id, expanding recursively at use time. Detect and reject reference cycles at save time with a clear error. Cap recursion depth and show which snippet broke if expansion fails.
236. **No shell/script-backed dynamic snippets, or running them unsafely** `[high]` - Users want computed output (a UUID, a formatted git branch name, an API lookup). Tools that lack this force manual computation; tools that bolt it on naively run arbitrary shell on every keystroke match, which is both a performance and a security hole (a synced malicious snippet executes code). → **vbuff:** Offer sandboxed dynamic snippets with an explicit opt-in per snippet, a visible 'this snippet runs code' badge, a timeout, and a confirmation the first time a script snippet runs after sync import. Never auto-execute script snippets that arrived via sync without user approval.
237. **Trigger/abbreviation collisions silently shadow each other** `[high]` - Two snippets get the same trigger (e.g. 'addr'), or one trigger is a prefix of another ('@@' vs '@@home'). The app silently fires whichever it finds first, so the user gets the wrong expansion with no warning and can't tell why their snippet 'stopped working'. → **vbuff:** Enforce trigger uniqueness within a scope at save time and refuse or warn on exact duplicates. For prefix overlaps, define explicit longest-match-wins semantics, show a conflict indicator in the editor list, and provide a 'find conflicts' view. Let prefix overlaps be resolved by a delimiter or by an on-ambiguity picker.
238. **Triggers fire mid-word inside unintended text** `[high]` - A bare abbreviation like 'eg' or 'ml' expands inside ordinary words ('eggs', 'html'), corrupting normal typing. Users disable the feature entirely because they can't trust it. → **vbuff:** Default to word-boundary matching (expand only when the trigger is preceded and followed by a separator) and let the user pick per-snippet matching: word-boundary, immediate, or requires-explicit-key. Encourage non-alphanumeric trigger prefixes (e.g. ';sig', ':addr') and ship sensible defaults.
239. **No way to undo an expansion in one step** `[medium]` - When a snippet fires by mistake (wrong trigger, mid-word match), undoing it takes multiple Ctrl+Z presses that may delete the user's own surrounding text too, or the deletion of the trigger and the insertion are not grouped so undo behaves unpredictably. → **vbuff:** Make each expansion a single atomic undo unit. Provide an immediate 'undo expansion / restore the literal trigger' shortcut that backs out the inserted text and re-types the original abbreviation, so a misfire is one keystroke to revert.
240. **Expansion fires in the wrong apps (terminals, password fields, IDEs)** `[critical]` - Auto-expansion triggering in a terminal, a code editor's vim mode, a password field, or a game injects garbage into command lines or sends credentials/abbreviations where they cause real damage. Users get burned and turn the whole tool off. → **vbuff:** Provide per-app and per-field-type allow/deny lists with safe defaults: disable expansion in secure/password fields and in apps the user marks as excluded. Detect terminal apps and offer a terminal-safe mode. Ship a global pause toggle and a visible status indicator of where expansion is active.
241. **Snippet expansion broken or unavailable on Wayland** `[high]` - Keystroke-injection-based expansion relies on synthetic input and global hooks that Wayland restricts, so the cross-platform tool silently does nothing on GNOME/Wayland while appearing to run, exactly the platform fragmentation users already hate. → **vbuff:** On Wayland use supported portals (e.g. virtual-keyboard / input-method protocols, RemoteDesktop portal) where the compositor allows, detect when injection is unavailable, and fall back to a paste-based insertion or an explicit picker. Never claim a snippet expanded when the injection silently failed; surface a clear 'expansion not supported in this session' state. _(seen in: Wayland core, GNOME Mutter)_
242. **Expansion mangles content in apps that swallow fast keystrokes** `[high]` - Synthetic typing too fast for Electron apps, remote-desktop sessions, or web forms drops characters, reorders them, or leaves the deleted trigger partially intact, so the expanded text comes out corrupted and unreliable per-app. → **vbuff:** Make injection timing tunable globally and per-app, default to a paste-based insertion for large or rich snippets (set clipboard, send paste, restore prior clipboard), and verify the target field accepted input where the platform permits. Offer a per-app 'use clipboard paste instead of keystrokes' override. _(seen in: ArsClip, CopyQ)_
243. **No rich-text / HTML / Markdown snippets** `[high]` - Users need formatted signatures, styled email templates, or snippets with links and bold, but plain-text-only tools strip all formatting. Users keep a second hacky tool just for the signature. → **vbuff:** Store snippets in a format-preserving representation (e.g. keep both a plain-text and a rich/HTML payload) and expand the right one based on the target field's accepted types. Author rich snippets in a proper editor; let Markdown source render to rich on expansion where supported. _(seen in: Maccy, Flycut, Diodon, GPaste)_
244. **Rich snippet always pastes formatting with no plain-text path** `[medium]` - The reverse problem: a formatted snippet drags its fonts/colors into a plain destination (a terminal, a Markdown editor), so the user gets mismatched typefaces or literal style markup and has to clean it up. → **vbuff:** Detect the destination's accepted MIME types and downgrade to plain text automatically when rich isn't wanted; always offer an explicit 'expand as plain text' modifier (e.g. hold a modifier key while triggering). Make per-snippet default representation configurable. _(seen in: Maccy, Paste, Alfred, Pastebot)_
245. **No image or non-text snippets** `[low]` - Users want a reusable logo, a stamped signature image, or a small diagram as a snippet, but text-only snippet stores can't hold them, so those assets live elsewhere and break the one-shortcut workflow. → **vbuff:** Allow binary/image payloads as snippet content with a thumbnail in the editor, store them as external blobs (not inline in the main DB) to avoid bloat, and paste the correct image MIME type into targets that accept it with a sensible fallback (e.g. a text alt or a file path) where they don't.
246. **No folders, groups, or tags for organizing snippets** `[high]` - All snippets sit in one flat list. Once a user has more than a few dozen, finding and maintaining them is miserable, and they can't enable a whole group of work snippets while disabling personal ones. → **vbuff:** Support nested folders/collections plus tags, with per-folder enable/disable and per-folder default settings (trigger prefix, target-app scope). Make folders the unit of sharing and of scoping so a 'Work' collection only fires in work apps. _(seen in: Maccy, Flycut, CopyClip)_
247. **No search across snippets / no quick picker by name** `[high]` - When the user forgets a snippet's exact trigger, there's no way to find it except scrolling the editor. The snippet exists but is effectively lost, so they re-create duplicates. → **vbuff:** Provide a fast fuzzy picker that searches snippet name, trigger, tags, and body, openable by a global hotkey, that inserts the chosen snippet directly into the focused field (with placeholder fill-in) without needing to remember the trigger. _(seen in: Paste, Alfred)_
248. **Clunky, cramped, or modal snippet editor** `[medium]` - Editing happens in a tiny single-line field or a popover that can't show the full body, has no monospace/wrap toggle, loses scroll position, and forces save-and-reopen to test. Authoring multi-line templates is painful enough that users avoid creating snippets. → **vbuff:** Ship a real editor pane: resizable, multi-line, word-wrap, monospace toggle, placeholder-token insertion buttons/autocomplete, live preview of the expanded result, and inline validation. Allow testing a snippet from inside the editor without leaving it. _(seen in: Paste, Clipy (snippet editor crashes))_
249. **Snippet editor crashes or loses unsaved edits** `[high]` - The editor crashes mid-edit or doesn't autosave, so a long template the user just wrote disappears. They learn not to trust the editor with anything substantial. → **vbuff:** Autosave drafts continuously to a separate draft store, keep an in-editor edit history/undo stack, and recover the last draft after a crash. Never let a single malformed snippet body crash the editor; validate and isolate parse errors. _(seen in: Clipy)_
250. **No live preview of placeholder expansion** `[medium]` - The user can't see what a templated snippet will actually produce until they fire it in a real document, so date formats come out wrong, placeholders are misspelled, and they only discover it after sending the email. → **vbuff:** Show a live preview pane in the editor that renders the snippet with sample/last-used variable values and the current date/time, updating as the user types. Flag unknown placeholder tokens and bad date format strings in the preview.
251. **Snippet sync loss / silent overwrite on multi-device edits** `[critical]` - Editing snippets on two devices, or syncing through a cloud-drive folder, produces conflict files, last-writer-wins overwrites, or a wholesale loss of the snippet library. Users lose hand-built templates with no recovery. → **vbuff:** Sync with a real conflict-resolution model (per-snippet versioning / merge, not whole-file overwrite), keep a local change journal and undo-able history, never route the live snippet DB through a naive file-sync folder, and keep timestamped backups so a bad sync is recoverable. _(seen in: Ditto, Paste)_
252. **Sync silently drops rich snippets, images, or placeholders** `[high]` - Sync carries only plain text, so on the second device a rich signature arrives as bare text, images vanish, and placeholder metadata is stripped, leaving the user with broken half-snippets they think synced fine. → **vbuff:** Sync the full snippet model (rich payload, attachments, placeholder definitions, per-snippet settings, folder structure), not just a text field. Verify round-trip fidelity in tests and surface a clear warning if any payload couldn't be transferred rather than dropping it silently. _(seen in: Windows 11 Win+V, 1Clipboard)_
253. **No import/export of the snippet library** `[medium]` - Snippets are trapped in the app. Migrating from another tool, backing up, sharing a team set, or moving to a new machine means re-typing everything by hand, which also locks users in and breeds resentment. → **vbuff:** Provide a documented, human-readable export/import format (e.g. JSON or a folder of files) covering triggers, bodies, placeholders, folders, and settings, plus importers for common formats from popular tools. Make export a first-class backup feature, not an afterthought.
254. **Triggering a snippet leaves the trigger text behind or deletes too much** `[high]` - On expansion the app deletes the wrong number of characters: it leaves the abbreviation in place before the inserted text, eats a preceding character/the user's selection, or removes part of an adjacent word, so output is malformed in subtle ways. → **vbuff:** Track exactly the typed trigger length (accounting for the boundary character) and delete precisely that. Test the delete-and-insert across word-boundary, immediate, and explicit-key modes, and never consume the user's existing selection unless an explicit overtype token says to. _(seen in: ArsClip)_
255. **Date/time and case transforms are rigid or absent** `[medium]` - Users can't get the date in the format they need, can't do relative dates ('next Monday', 'today+3'), and can't auto-match the surrounding case (Title Case vs lowercase), so templated output needs hand-editing and looks inconsistent. → **vbuff:** Support full strftime-style date/time formatting with locale and timezone, relative date arithmetic, and case/format transform functions on variables (upper, lower, title, trim, default-if-empty). Document the format strings and validate them in the editor.
256. **Global expansion has no quick pause / kill switch** `[medium]` - During screen sharing, pair programming, gaming, or password entry the user needs expansion off NOW, but there's no fast toggle, so a stray abbreviation fires in front of an audience or into the wrong field. → **vbuff:** Provide a global pause/resume hotkey and a tray/menubar toggle with a clear on/off indicator, plus an auto-pause option for full-screen apps and for apps on the deny list. Make the current state obvious at a glance.
257. **Sensitive data captured into snippet bodies without protection** `[high]` - Users build snippets containing API keys, license keys, or personal data for convenience, and the tool stores them in plain text and syncs them to the cloud by default, turning a productivity feature into a credential leak. → **vbuff:** Offer per-snippet 'secret' marking that encrypts the body at rest, excludes it from sync (or requires explicit opt-in), masks it in the editor list, and optionally requires re-auth to reveal. Warn when a snippet body looks like a credential.
258. **No per-snippet enable/disable; only all-or-nothing** `[low]` - To stop one misbehaving snippet the user must delete it (losing the content) or turn off the entire engine. There's no way to temporarily retire a snippet while keeping it. → **vbuff:** Add a per-snippet and per-folder enabled toggle that keeps the content but stops the trigger from firing, plus a 'disabled' visual state in the list. Disabling a folder disables its snippets without losing scope/order.
259. **No automatic snippet suggestion / no learning from repeated paste** `[low]` - The tool never helps users discover what should be a snippet. People keep manually re-typing the same paragraph for months because nothing offers to turn frequently repeated text into a snippet. → **vbuff:** Detect frequently repeated clipboard/typed text and offer a non-intrusive 'save this as a snippet?' suggestion with a pre-filled trigger. Keep it opt-in and dismissible, and never auto-create snippets without confirmation.
260. **Trigger only fires with one fixed activation method** `[medium]` - Some users want abbreviation auto-expansion, others want an explicit hotkey-then-pick, others want a typed prefix that opens a menu. A tool that hardwires one method forces a workflow that fights muscle memory and breaks in apps where that method fails. → **vbuff:** Support multiple coexisting activation methods per snippet: inline abbreviation, explicit completion key (Tab/Space/Enter, configurable), global picker hotkey, and menu-by-prefix. Let the user choose defaults globally and override per snippet/folder.
261. **Choice/option placeholders unsupported, forcing many near-duplicate snippets** `[medium]` - To handle small variations (formal vs casual greeting, three product names) the user creates a separate snippet for each combination, multiplying maintenance and trigger collisions. → **vbuff:** Support dropdown/multiple-choice and optional/conditional placeholders within one snippet, so a single template covers the variants. Surface the choices in the fill-in form. Allow a default selection and free-text override.
262. **Reordering, bulk-editing, and duplicating snippets is missing or tedious** `[low]` - Users can't drag to reorder, can't select multiple snippets to move/delete/retag, and can't duplicate a snippet as a starting point, so maintaining a large library is slow and error-prone. → **vbuff:** Support drag-to-reorder, multi-select with bulk move/delete/tag/enable, and 'duplicate snippet' as a one-click action. Make these operations undoable and reflected in sync. _(seen in: Maccy)_

## 8. Transform & action mistakes

263. **No plain-text / strip-formatting paste at all** `[high]` - User copies styled text from a browser or Word and the only paste path re-injects rich text (fonts, colors, background highlight, list markers), so it clashes with the destination document and the user must manually clean it up every time. The single most common everyday clipboard need has no first-class command. → **vbuff:** Ship a built-in 'Paste as plain text' action bound to a distinct hotkey (e.g. a modifier on the paste key) AND a global 'always paste as plain text' mode. Strip all non text/plain representations at paste time without mutating the stored clip, so the rich version stays available. _(seen in: Maccy, Paste, Alfred, Pastebot, Clipman, GPaste, Diodon)_
264. **Plain-text strip path mangles or destroys content** `[high]` - The 'paste without formatting' route does more than drop styling: it strips file:// URLs so a copied file path pastes as nothing, collapses newlines, drops tabs, or turns a copied image into an empty string because only text/plain was retained. The user loses data, not just formatting. → **vbuff:** Define 'plain text' precisely: keep the text/plain (or public.utf8-plain-text) representation verbatim, preserve line endings, tabs and Unicode, and never apply the strip to non-text clips (files, images). If no text representation exists, fall back to pasting the original rather than emptying the clip. _(seen in: Maccy, Paste)_
265. **Transforms mutate the stored clip in place** `[high]` - Applying a transform (lowercase, trim, URL-decode) overwrites the saved history entry, so the original is gone. A reversible-looking action is actually destructive and the user cannot get the source text back. → **vbuff:** Treat every transform as non-destructive: transform a copy at paste time, or write the result as a new history entry while keeping the original. Never overwrite the captured bytes.
266. **No transforms whatsoever** `[medium]` - The manager can only store-and-replay. Users who repeatedly need trim-whitespace, join-lines, change-case, or strip-formatting have to round-trip through an external editor, defeating the point of a power-user clipboard tool. → **vbuff:** Ship a curated set of safe, well-tested built-in transforms (trim, collapse whitespace, change case, strip formatting, join/split lines, URL encode/decode, base64, JSON pretty-print) as preview-able, reversible actions. _(seen in: Maccy, Flycut, Clipy, CopyClip, Ditto)_
267. **Hardcoded, non-extensible action set** `[medium]` - Actions like 'open URL', 'paste', 'delete' are baked in with no way to add your own. A user whose workflow needs 'send to translation' or 'create Jira ticket from clip' is stuck, and the app cannot grow with real usage. → **vbuff:** Expose actions through a declarative plugin/command interface (manifest-described commands the user can enable, reorder, and bind), so the action list is data, not compiled-in code. _(seen in: Maccy, Flycut, Clipy, Windows 11 Win+V)_
268. **No scripting / no user-defined transforms** `[medium]` - Power users want to run their own logic on a clip (regex replace, call an API, reformat a code snippet) but the app offers no scripting hook, so every non-trivial transformation happens outside the tool. -> **vbuff:** Provide a sandboxed native subprocess protocol that receives one bounded clip and returns a typed result, with explicit per-action capability grants for filesystem/network and no plugin code loaded into the resident process. _(seen in: Maccy, Flycut, CopyClip)_
269. **Unsafe script execution with full host privileges** `[critical]` - A 'run shell command on clip' or 'AppleScript action' feature executes user-or-template scripts with the app's full permissions and no sandbox. A malicious snippet, an imported action pack, or a clip containing shell metacharacters can run arbitrary code, exfiltrate the (secret-laden) clipboard, or wipe files. → **vbuff:** Run all transform scripts in a sandbox with no ambient authority: no shell by default, no filesystem/network unless the user granted that specific capability, resource and time limits, and a clear consent prompt before any imported action can execute. _(seen in: ClipboardFusion, CopyQ)_
270. **Clip content interpolated unescaped into a command/template** `[critical]` - Action templates that splice the clip text directly into a shell command, URL, or SQL (e.g. open `https://x/?q={clip}` or run `echo {clip}`) let clip contents inject command separators, quotes, or shell expansion, corrupting the action or executing attacker-controlled commands when the user copies hostile text from a webpage. → **vbuff:** Never string-concatenate clip text into commands. Pass the clip as a separate argument / parameter (argv array, not a shell string), percent-encode for URLs, and bind parameters for any query. Escape by construction, not by hoping the content is benign. _(seen in: ClipboardFusion, CopyQ)_
271. **Auto-running transforms/triggers on every copy by default** `[high]` - A 'triggers' or 'auto-action' feature that fires automatically on each new clip causes random crashes several times a day, surprises users with mutated clips, and can leak data by auto-sending clips somewhere. The vendor of one app eventually disabled it by default after crash reports. → **vbuff:** Make auto-firing actions strictly opt-in per rule, default off, with a visible indicator when an auto-transform changed a clip, a kill switch, and crash isolation so a misbehaving trigger cannot take down capture. _(seen in: ClipboardFusion)_
272. **No chaining of transforms** `[medium]` - Each transform is a dead end. A user who needs 'strip formatting, then trim, then lowercase' has to apply, re-copy, apply, re-copy. Composition, the whole value of transforms, is impossible. → **vbuff:** Allow transforms to be composed into ordered pipelines (clip -> t1 -> t2 -> t3 -> paste), savable as named macros and bindable to a key, with the output of each stage feeding the next. _(seen in: Maccy, Flycut)_
273. **Cannot assemble/paste multiple clips in one action** `[low]` - Users want to concatenate several history items into one paste (gather snippets, build a list) but there is no multi-select-and-merge action, forcing tedious one-at-a-time pasting. → **vbuff:** Support multi-select with a 'paste merged' action and a configurable joiner (newline, comma, space, custom), plus a 'stack/append' mode that accumulates copies into one buffer. _(seen in: Maccy)_
274. **Transform corrupts Unicode / encoding** `[high]` - Case-change, trim, or 'sanitize' transforms operate on bytes or naive char arrays and break multibyte UTF-8, combining characters, emoji (surrogate/grapheme clusters), or right-to-left text, so the pasted result is mojibake or a split emoji. → **vbuff:** Do all text transforms on validated Unicode using grapheme- and locale-aware operations (e.g. Rust's char/grapheme handling, ICU-style case mapping). Never index by byte. Round-trip test transforms against an emoji/CJK/RTL/combining-marks corpus.
275. **Transform silently rewrites line endings** `[medium]` - A transform or even plain capture normalizes CRLF to LF (or strips a trailing newline), so pasting a clip into a Windows tool, a here-doc, or a hash-sensitive field changes the content and breaks diffs, checksums, or required terminators. → **vbuff:** Preserve original line endings by default. Only change them via an explicit, clearly labeled 'normalize line endings' transform that states the target (LF/CRLF), and never apply it implicitly.
276. **Smart-quote / autocorrect mangling on copy or paste** `[high]` - A 'beautify' or 'smart text' transform converts straight quotes to curly quotes, hyphens to dashes, or applies autocorrect, which silently corrupts code, passwords, JSON, and URLs pasted into a terminal or editor. → **vbuff:** Never apply typographic substitution to clips by default. If offered at all, it must be an explicit opt-in transform that is auto-disabled for clips detected as code/URLs/structured data.
277. **Trim/whitespace transform too aggressive** `[medium]` - A 'trim' action strips interior whitespace or significant leading indentation, not just surrounding whitespace, so pasting Python code, YAML, or a Markdown code block loses its indentation and breaks. → **vbuff:** Define trim narrowly as removing only leading/trailing whitespace of the whole clip; offer 'collapse internal runs' as a separate, clearly distinct transform. Preserve indentation; show a before/after preview so the user sees exactly what changes.
278. **No preview before a transform/action is applied** `[medium]` - The user fires a transform blind and only discovers it mangled the clip after pasting into the destination, where the damage is already done. → **vbuff:** Show a live before/after preview for every transform and let the user confirm or cancel. For pipelines, preview the final result. Make all transforms undoable.
279. **Rich-text representation chosen over the better one on paste** `[medium]` - When a clip holds several representations (text/html, RTF, image/png, text/plain) the manager re-pastes the wrong one: an image copied from a browser pastes as an <img> HTML tag or raw HTML instead of pixels, so an image editor receives garbage; or formatted text pastes as bare text when formatting was wanted. → **vbuff:** Preserve all original representations and let the destination negotiate normally. Provide explicit 'paste as image / as HTML / as plain text' actions and pick a sensible default per target rather than dropping representations at capture time. _(seen in: cliphist, clipman, Maccy, CopyQ, wl-clipboard sync tools)_
280. **Capturing only one MIME type, discarding the rest** `[medium]` - The manager reads only text/plain (or only the first offered type) at capture, so on paste all formatting, embedded images, and alternate representations are permanently gone, even though the source app offered them. → **vbuff:** Capture all offered representations/flavors of a clip and store them together. Defer the choice of which to emit until paste time so nothing is lost up front. _(seen in: Clipman, GPaste, Diodon, various GTK managers, Windows 11 Win+V, ClipAngel, Comfort Clipboard)_
281. **Paste action injects a literal keystroke character into the field** `[high]` - The auto-paste action simulates Ctrl+V/Cmd+V by sending key events, but a timing or focus glitch leaks the 'v' keystroke so the destination receives a literal 'v' (or the modifier is dropped) instead of the clip, silently corrupting input. → **vbuff:** Prefer pasting via the platform paste API / synthetic paste event over raw key-event synthesis where possible. If key synthesis is required, verify modifier state, suppress the key character, and confirm focus before sending; never let the activation key bleed through as text. _(seen in: Ditto)_
282. **Paste-back surfaces as a notification instead of inserting** `[critical]` - After an OS update the paste 'action' degrades to posting the clip text into a notification banner rather than typing it into the focused field, so selecting an item appears to do nothing useful and the user must copy out of the notification. → **vbuff:** Make insertion the contract of the paste action with explicit success/failure detection. If insertion cannot be performed (e.g. missing accessibility permission), set the clipboard and tell the user why, rather than silently degrading to a notification that pretends to be a paste. _(seen in: Maccy, Flycut, Clipy)_
283. **Paste action overwrites selection / wrong target instead of inserting at caret** `[medium]` - The paste action replaces an entire field/selection (e.g. a whole filename) or lands in the wrong control instead of inserting at the caret, because it sends edit commands or clicks rather than a clean paste, destroying existing content. → **vbuff:** Insert at the current caret via the standard paste mechanism and never issue select-all/replace as part of paste. Confirm the intended target field has focus before acting; abort with a clear message if it cannot be confirmed. _(seen in: ArsClip)_
284. **'Open URL' / link action auto-fires on copy, leaking or navigating unexpectedly** `[high]` - A clip that looks like a URL triggers an automatic open or prefetch (for preview/unfurl), which navigates the browser without consent, hits attacker-controlled tracking URLs the user merely copied, and can deanonymize or trigger one-click actions. → **vbuff:** Never auto-open or auto-fetch URLs from clips. URL actions must be explicit and user-initiated, and link previews/unfurls (if any) must be opt-in with no network request until the user asks.
285. **Transforms not honoring concealed/transient clips** `[high]` - A transform or auto-action processes a clip the source marked concealed (a password) the same as any other, e.g. running OCR, sending it to a 'translate' web service, or writing the transformed result into persistent history, leaking the secret through the action path even if base capture would have skipped it. → **vbuff:** Propagate the concealed/transient flag through the entire transform/action pipeline: refuse to auto-process, never send concealed clips to network actions, and never persist a transformed derivative of a concealed clip. _(seen in: Maccy, CopyQ, GPaste, Flycut, Clipy)_
286. **Network-backed transforms send clip contents off-device silently** `[high]` - Actions like 'translate', 'shorten URL', 'AI rewrite', or 'OCR via cloud' transmit the clip (which may be a password, token, or private text) to a third-party server without warning, expanding exposure far beyond the local machine. → **vbuff:** Clearly label any action that makes a network call, require explicit per-action consent, show the destination, never send concealed/secret clips, and prefer on-device processing (local OCR/translation) where feasible.
287. **OCR-on-everything as an automatic transform** `[medium]` - Auto-running text recognition on every copied image turns photos of IDs, cards, and receipts into searchable plaintext stored in history, broadening what is at risk and burning CPU on every image copy. → **vbuff:** Make OCR an explicit on-demand action per image, off by default, run locally, and exclude its output from persistent indexing unless the user opts in. Never OCR concealed clips. _(seen in: KeepClips, Clipory, ClipZ, Alfred, Raycast, Maccy)_
288. **No way to disable / strip auto-applied formatting fix-ups** `[medium]` - The manager always re-emits whatever it captured with its own 're-encoding', and there is no toggle to get a clean plain default, so users who paste into code editors all day fight formatting on every paste with no global escape hatch. → **vbuff:** Provide a global default-paste-mode setting (rich / plain / match-destination) plus a per-clip override, and a quick toggle. Respect it consistently across the picker, hotkey paste, and drag-out. _(seen in: Maccy, Paste, Alfred, Pastebot)_
289. **Snippet/template expansion with unescaped or recursive placeholders** `[medium]` - Snippet templates with placeholders ({date}, {clipboard}, {cursor}) expand naively: an embedded placeholder inside pasted content re-expands (injection), recursive snippet references loop, or a malformed token leaves literal garbage in the output. → **vbuff:** Expand placeholders in a single non-recursive pass over the template only (never over the substituted values), validate tokens, cap expansion depth, and treat clip-derived content as inert data that is not re-scanned for placeholders.
290. **Transform crashes or hangs on large clips with no isolation** `[high]` - Running a transform (JSON pretty-print, OCR, regex) on a hundreds-of-MB clip pins a core, balloons RAM, and crashes the whole app, taking capture down with it, because the transform runs on the main thread with no limits. → **vbuff:** Run transforms off the UI thread with size guards, time and memory limits, cancellation, and crash isolation so a failed transform reports an error and leaves the original clip and the rest of the app intact. _(seen in: CopyQ, GPaste)_
291. **No 'paste original' escape after a transform pipeline** `[medium]` - Once a transform or auto-action has run, the user has no quick way to get the untouched original clip back into the destination, so a wrong transform forces a re-copy from the source (which may no longer exist). → **vbuff:** Always keep the untouched original alongside any derived result and bind a one-key 'paste original / revert' action. Make the original recoverable from history even after a transform was applied.
292. **Destructive 'sanitize' transform silently drops characters** `[medium]` - A 'remove special characters' or 'clean' action strips characters it deems unsafe (control chars, certain Unicode, currency symbols), so legitimate content is silently truncated or altered and the clip never round-trips faithfully. → **vbuff:** Default to lossless. Any sanitizing transform must enumerate exactly what it removes, show a preview, be opt-in, and never be applied as part of normal capture/paste. _(seen in: GPaste)_
293. **Drag-out / quick-paste action emits a different representation than the picker paste** `[low]` - Dragging a clip out, hotkey paste, and the picker each pick different representations or apply different formatting, so the same clip yields inconsistent results depending on how it was inserted, confusing users. → **vbuff:** Route all output paths (picker, hotkey, drag-out, action) through one paste engine with one representation-selection policy and one plain/rich setting, so behavior is identical regardless of entry point.
294. **Transform/action errors fail silently** `[medium]` - When an action cannot run (script error, missing permission, unsupported clip type) it does nothing and gives no feedback, so the user thinks the clip pasted and only discovers the empty/unchanged result downstream. → **vbuff:** Surface every transform/action failure inline with a clear reason and leave the original clip on the clipboard so a manual paste still works. Distinguish 'did nothing' from 'failed'.

## 9. Sync & multi-device mistakes

295. **Plaintext sync over the wire** `[critical]` - Clipboard contents (passwords, 2FA codes, card numbers, private URLs) travel between devices unencrypted, so anyone on the same LAN, a coffee-shop Wi-Fi, or a MITM position can read every copy. → **vbuff:** Encrypt all sync traffic in transit by default with modern TLS or an authenticated end-to-end channel (e.g. Noise/libp2p with mutual authentication). Never offer a 'plain' transport; if a peer can't negotiate encryption, refuse to sync rather than downgrade silently. _(seen in: Ditto (network sync), LAN-sync managers, various vendor clipboard apps with sync)_
296. **Cloud relay can read your clipboard (no E2E encryption)** `[critical]` - Even when transport is TLS, a vendor relay server decrypts and stores clip contents, so the user must trust the provider's servers, staff, breach posture, and legal jurisdiction with their most sensitive copies. → **vbuff:** Use true end-to-end encryption where keys never leave the user's devices; the relay only forwards opaque ciphertext it cannot decrypt. Derive the sync key from a device-pairing secret, not from the account password on the server. _(seen in: Windows clipboard cloud sync (Microsoft account), 1Clipboard (Google Drive), Paste, PastePal, PasteNow)_
297. **Sensitive items synced off-device by default or via easily-missed toggle** `[critical]` - Passwords and card numbers leave the machine the moment they're copied because cloud sync is on by default or the opt-in is buried, expanding exposure far beyond the local disk without the user realizing it. → **vbuff:** Default sync to OFF, or default to LAN-only. Never sync items flagged sensitive/concealed at all. Make the sync scope explicit on first run and surface a persistent indicator showing what is leaving the device. _(seen in: Windows 11 Clipboard History (Win+V), 1Clipboard, ClipboardFusion)_
298. **Vendor lock-in: sync only through the vendor's proprietary cloud** `[high]` - Sync is welded to one company's servers and account; users can't self-host, can't point it at their own infrastructure, and lose all sync if the vendor shuts down, raises prices, or gets acquired. → **vbuff:** Make the sync transport pluggable and documented: ship a built-in LAN/peer transport plus an optional self-hostable relay with an open protocol. Never make cloud the only path. _(seen in: Paste, Windows 11 Clipboard History (Win+V), 1Clipboard)_
299. **No LAN-only / peer-to-peer option** `[high]` - Users on the same network are forced to round-trip every clip through the internet and a third-party cloud even when both devices are sitting on the same desk, adding latency, an outage dependency, and needless cloud exposure. → **vbuff:** Implement direct device-to-device sync over the local network with auto-discovery (mDNS/DNS-SD) and an encrypted P2P channel. Prefer LAN when peers are reachable; fall back to relay only when they aren't. _(seen in: Paste, Windows 11 Clipboard History (Win+V), 1Clipboard)_
300. **Sync requires a mandatory account** `[high]` - Users must create and sign in to a vendor account just to move a clip from laptop to phone, tying a purely local convenience to an identity, an email harvest, and a server-side profile they never wanted. → **vbuff:** Support account-free pairing: scan a QR code or enter a short code on both devices to establish a shared key. Reserve accounts as an optional convenience for cross-network relay, never a precondition for LAN sync. _(seen in: Windows 11 Clipboard History (Win+V), Paste, 1Clipboard)_
301. **Sync silently stalls or stops with no error** `[high]` - Items just stop appearing on the other device; the user keeps copying expecting it to work, discovers hours later it broke, and has no log, status, or error to diagnose why. → **vbuff:** Show explicit per-device sync status (connected / last-synced timestamp / pending count / error). Surface failures as actionable notifications, keep a sync log, and auto-retry with backoff plus a manual 'force resync' that reports what it did. _(seen in: Windows 11 Clipboard History (Win+V), Copied, 1Clipboard, Ditto, Paste)_
302. **Last-writer-wins overwrites concurrent clips silently** `[high]` - Two devices copy at nearly the same time; the sync layer resolves the 'conflict' by keeping one and discarding the other, so a clip the user copied on device A vanishes without trace. → **vbuff:** Model clipboard history as an append-only, per-device log, not a single mutable 'current clipboard' cell. Merge histories by union with stable IDs and timestamps so concurrent copies all survive; never silently drop one.
303. **Routing the DB through a file-sync folder (Dropbox/Drive/iCloud)** `[high]` - Putting the SQLite history file inside a cloud-synced folder produces conflict copies ('file (conflicted copy)'), DB locking, and outright corruption, because file sync tools are not safe for live multi-writer databases. → **vbuff:** Never sync the raw database file. Sync at the record level over a dedicated protocol with per-item IDs and explicit merge logic. If a user points the DB at a synced folder, detect it and warn. _(seen in: Ditto, CopyQ)_
304. **No conflict resolution / merge corrupts the history DB** `[high]` - When the same store is written from two devices, the manager has no merge strategy, so the database ends up corrupted or inconsistent and the user must delete it and lose all history to recover. → **vbuff:** Use conflict-free replicated data structures (CRDT/OR-Set semantics) or vector clocks for the history set so independent edits on multiple devices always converge deterministically without corruption. _(seen in: Ditto)_
305. **Deletes don't propagate (tombstone-less sync)** `[high]` - The user deletes a sensitive clip on one device, but it lives on (or resurrects) on every other device, so a 'deleted' secret is still recoverable across the fleet and may even sync back. → **vbuff:** Replicate deletions as first-class tombstone records with their own IDs and timestamps so a delete on any device removes the item everywhere and cannot be undone by a stale peer re-adding it.
306. **Resurrection of deleted items from a stale peer** `[high]` - A device that was offline when a clip was deleted comes back online and re-pushes the old item, undoing the deletion across all devices, which is especially dangerous for secrets the user thought were purged. → **vbuff:** Track per-item version vectors so a delete dominates any older add of the same item; require peers to reconcile tombstones before re-adding, and never let an add with an older version overwrite a newer delete.
307. **Laggy sync: copy on A, long wait before it lands on B** `[high]` - Multi-second or minutes-long propagation delay defeats the purpose of cross-device clipboard; the user gives up and emails the snippet to themselves instead. → **vbuff:** Use an event-driven push channel (a persistent encrypted connection / push notification), not periodic polling. Target sub-second LAN propagation and show a 'syncing now' indicator so the user trusts it's working. _(seen in: Paste, Windows 11 Clipboard History (Win+V))_
308. **Initial sync requires keeping the app open for minutes** `[high]` - On first pairing the user must leave the app foregrounded for several minutes or the sync never finishes, and items fail to appear until they manually reset cloud data. → **vbuff:** Make initial sync resumable and incremental in the background; checkpoint progress so a closed app continues where it left off. Never require a foreground babysitting session, and never demand a full cloud-data reset to fix a stuck sync. _(seen in: Paste)_
309. **Polling the sync source instead of event notifications** `[high]` - The app wakes on a timer (e.g. every ~5s) and scans a synced folder or queries the server, burning CPU and battery, scaling worse as history grows, while still adding propagation latency. → **vbuff:** Subscribe to change events (filesystem watch, server push, P2P notify). Reserve polling as a slow keepalive fallback only, with exponential backoff when idle. _(seen in: CopyQ, Ditto)_
310. **Peer sync silently dies when another process fights the clipboard** `[high]` - RDP, VMware/VirtualBox, AutoHotkey, or a second clipboard tool grabs the clipboard chain and the sync agent quietly records and propagates nothing, while the tray icon still looks healthy. → **vbuff:** Detect when capture is failing (heartbeat on expected clipboard events, ownership-loss detection) and raise a visible 'capture stopped, sync paused' warning instead of failing silently. Use modern listener APIs that coexist with other clipboard consumers. _(seen in: Ditto, CopyQ)_
311. **No selective sync (all-or-nothing)** `[high]` - Sync is a single global on/off, so the user can't keep secrets, huge images, or work clips local while syncing only the snippets they actually want on other devices. → **vbuff:** Provide per-item, per-type, per-source-app, and per-device sync rules: a 'don't sync this' toggle, type filters (skip images/large items), app exclusion lists, and a 'push this one to device X' one-shot action. _(seen in: Windows 11 Clipboard History (Win+V), Paste, 1Clipboard)_
312. **Sync ignores the concealed/sensitive pasteboard flag** `[critical]` - Password managers mark a copy as concealed/transient, but the sync layer uploads it anyway, so a credential the OS asked to keep ephemeral gets replicated and persisted across every device. → **vbuff:** Honor OS sensitivity hints (macOS org.nspasteboard.ConcealedType / TransientType, KDE x-kde-passwordManagerHint, Windows ExcludeClipboardContentFromMonitorProcessing) by refusing to store AND refusing to sync flagged items, end to end. _(seen in: Windows 11 Clipboard History (Win+V), Flycut, CopyQ, Ditto)_
313. **Undocumented sync limitation: only plain text crosses devices** `[medium]` - Users copy a formatted snippet or an image expecting it on the other device, but only text/plain syncs and the rest is silently dropped, with no documentation or indication of the limit. → **vbuff:** Sync all captured representations (plain text, rich text, images, files) as a single multi-format item, or clearly state and surface in the UI which formats do and don't sync, with a visible badge on partially-synced items. _(seen in: Windows 11 Clipboard History (Win+V))_
314. **Syncing huge images/files over a metered or constrained link** `[medium]` - A single Retina screenshot can be tens of MB; auto-syncing every large clip saturates the user's mobile data, drains battery, and clogs the relay, sometimes for content they never needed elsewhere. → **vbuff:** Sync small items eagerly; for large items send a lightweight metadata/preview record and fetch the full payload on demand ('download to this device'). Respect metered-connection settings and a configurable per-item size threshold for auto-sync. _(seen in: Ditto, CopyQ, Windows 11 Clipboard History (Win+V))_
315. **No end-to-end integrity / authentication of synced items** `[critical]` - Without per-item authentication, a compromised relay or LAN attacker can inject or tamper with a synced clip (e.g. swap a crypto address), and the receiving device pastes attacker-controlled content. → **vbuff:** Authenticate every synced record with an AEAD/MAC keyed to the shared device secret; reject items that fail verification. Pin device identities so a relay cannot impersonate a paired device.
316. **Weak or no device-pairing authentication** `[high]` - Anyone who can reach the sync port or guess a short code can pair a rogue device and start receiving every copied clip, or push malicious clips into the user's history. → **vbuff:** Require an out-of-band pairing step (QR code or short code shown on one device, entered on the other) that establishes a mutually-authenticated key. Show a list of paired devices with names and last-seen, and allow instant revocation.
317. **No way to revoke a lost or stolen device from sync** `[high]` - A stolen laptop or phone keeps receiving the user's clipboard forever because there's no device list and no revocation, so every future copy leaks to the thief. → **vbuff:** Maintain an explicit paired-device roster with per-device revocation that rotates the shared sync key so a removed device can no longer decrypt new items. Confirm revocation took effect on the remaining devices.
318. **Sync loop / echo storms between devices** `[high]` - Device A sets the clipboard from a synced item, which the local watcher treats as a new copy and re-syncs back to B, which echoes it again, causing duplicate floods, wasted bandwidth, and reordered history. → **vbuff:** Tag every synced item with an origin device ID and a stable content/event ID; suppress re-capture of clipboard changes the app itself injected from sync, and dedupe on the shared ID so an item never bounces.
319. **Cross-device clock skew corrupts ordering and merge** `[medium]` - Merge and conflict resolution rely on wall-clock timestamps, so a device with a wrong clock makes its clips jump to the top, hide newer items, or win conflicts they shouldn't, scrambling history across the fleet. → **vbuff:** Order and reconcile using logical clocks (Lamport timestamps / version vectors) rather than raw wall time; use wall time only for display, and detect/flag implausible skew.
320. **Offline edits lost instead of queued** `[high]` - Copies made while a device is offline are dropped or never propagate once it reconnects, so the user's airplane-mode clips silently never reach their other devices. → **vbuff:** Queue all local changes durably while offline and replay them on reconnect with idempotent, ID-keyed merge so nothing is lost and nothing is duplicated.
321. **Persistent residual synced data after disabling sync or uninstall** `[high]` - Turning sync off or uninstalling leaves the user's clips sitting in the vendor cloud / relay and on paired devices, so the secret-laden synced history survives what the user believed was a wipe. → **vbuff:** On disable/uninstall, offer to purge server-side and remote-device copies, confirm completion, and document exactly what remains where. Provide a 'delete all my synced data everywhere' action. _(seen in: Ditto, SQLite-based managers, various vendor cloud managers)_
322. **Closed-source, unauditable sync crypto claims** `[medium]` - Marketing says 'private, end-to-end, we never see your data', but the sync code and key handling are closed and unverifiable, so users must take the security of their entire clipboard on faith. → **vbuff:** Open-source the sync and crypto layer (vbuff is Rust/open), use standard well-reviewed primitives, document the protocol and key model, and ideally publish a third-party audit so the E2E claim is verifiable, not marketing. _(seen in: Paste, PastePal, PasteNow, other commercial sync managers)_
323. **Sync requires the proprietary mobile app to exist and stay maintained** `[medium]` - Cross-device sync only works through a vendor's companion app; when that app is dropped, never ships for a platform, or breaks on OS updates, the whole sync promise evaporates and clips strand on one device. → **vbuff:** Build sync on an open, documented protocol so any platform client (including third-party or community ones) can participate, and ship first-class clients per platform rather than betting sync on a single fragile companion app. _(seen in: Pastebot, Maccy, CopyClip, Flycut)_
324. **Relay metadata leakage (traffic analysis even with E2E)** `[medium]` - Even when payloads are encrypted, the relay logs which devices talk, when, how often, and item sizes, building a behavioral profile and timing fingerprint of the user's copy activity. → **vbuff:** Minimize and avoid logging metadata on the relay, support padding/batching to blur sizes and timing, allow direct P2P that bypasses the relay entirely, and document the relay's data-retention policy.
325. **No per-item TTL / auto-expiry on synced sensitive clips** `[high]` - A synced 2FA code or password lingers in every device's history (and the relay queue) forever, so a one-time secret stays recoverable fleet-wide long after it was needed. → **vbuff:** Support per-type and per-item time-to-live that propagates with the item, so a sensitive clip auto-expires and is purged from all devices and the relay queue at the same deadline. _(seen in: Windows clipboard cloud sync (Microsoft account), various history-keeping sync managers)_
326. **Sync state can't be reset without nuking all history** `[medium]` - When sync gets wedged the only fix offered is to delete the entire iCloud/cloud data set, which also destroys the user's accumulated clipboard history. → **vbuff:** Separate the sync transport/replication state from the stored history. Provide a 'reset sync (keep history)' action that re-handshakes peers and replays, without forcing the user to lose their clips. _(seen in: Paste)_

## 10. Security & privacy mistakes

327. **Plaintext on-disk history database** `[critical]` - Every copied password, API key, card number, and 2FA code is written to an unencrypted SQLite/JSON file. Any local process running as the user, any backup, and any malware can read the full clipboard history off disk indefinitely. → **vbuff:** vbuff encrypts the history store at rest with an authenticated cipher (e.g. XChaCha20-Poly1305 via the `ring`/`age` crates). Derive the key from an OS keychain-stored secret (macOS Keychain, Windows DPAPI/Credential Manager, Linux Secret Service/kernel keyring), never a hardcoded or file-stored key. Encrypt per-record so nothing readable ever hits disk. _(seen in: Ditto, CopyQ, Windows Win+V, Maccy, Samsung Clipboard, Flycut, Clipy)_
328. **Capturing OS concealed/transient clipboard flags ignored** `[critical]` - Password managers mark copied secrets with platform hints (macOS org.nspasteboard.ConcealedType / org.nspasteboard.TransientType, Windows ExcludeClipboardContentFromMonitorProcessing / CanIncludeInClipboardHistory=0, KDE x-kde-passwordManagerHint, X11 password-manager hints). Naive managers ignore them and store the secret anyway. → **vbuff:** Before storing any item, vbuff inspects every available concealment/transience marker on all platforms and skips capture entirely (no record, no log) when any is present. Treat unknown/private MIME types conservatively. Add CI tests that copy via known password managers and assert nothing is stored. _(seen in: Maccy, CopyQ, Ditto, GPaste, Flycut, Clipy, Jumpcut, Klipper)_
329. **No auto-clear / auto-expiry for sensitive items** `[high]` - Secrets, OTP codes, and banking data persist in history forever with no time bound, so anything ever copied stays recoverable on disk long after it was needed. → **vbuff:** vbuff supports a global TTL and a stricter TTL for entries classified sensitive (auto-expire after N seconds/minutes, default on for detected secrets). Securely overwrite the in-memory buffer and delete the encrypted record + indexes on expiry, not just hide it from the UI. _(seen in: Maccy, Ditto, CopyQ, Flycut, Clipy)_
330. **Deleted clips not actually purged (SQLite freelist/WAL residue)** `[high]` - User deletes a secret, but the bytes survive in SQLite freelist pages, the WAL/journal, and unallocated pages, recoverable with a hex editor or forensic tool. 'Clear all' is cosmetic. → **vbuff:** Because data is encrypted per-record (see encryption-at-rest), deletion drops the key/record so residue is ciphertext. Additionally run `PRAGMA secure_delete=ON`, VACUUM after bulk deletes, use `PRAGMA journal_mode` carefully, and shred WAL/temp files. Provide a verifiable 'purge' that proves the plaintext is unrecoverable. _(seen in: Ditto, CopyQ, SQLite-based managers generally)_
331. **No content classifier for secrets (passwords/OTP/cards/keys)** `[high]` - The manager treats all clipboard content identically, so high-entropy strings, OTP codes, credit card numbers (Luhn), private keys, and JWTs land in plain history with no special handling. → **vbuff:** vbuff runs a local heuristic classifier on capture: Luhn check for cards, regexes for AWS/GCP/GitHub tokens, PEM/SSH key markers, BIP39/crypto addresses, 6-8 digit OTP patterns, and Shannon-entropy thresholds. Classified-sensitive items get short TTL, are excluded from sync/search-by-default, and never logged. All classification is on-device.
332. **No exclusion list / per-app source filtering** `[medium]` - The manager captures from every source, including password managers, banking apps, terminals, and secret-handling tools, unless the user discovers and manually configures ignore rules they often don't know exist. → **vbuff:** vbuff ships a sane default deny-list (known password managers, common banking/auth apps) plus user-editable allow/deny by app bundle id / executable / window class. Capture is suppressed at the source level so excluded apps never produce a record. _(seen in: Ditto, Flycut, many basic managers)_
333. **Telemetry/analytics enabled by default** `[medium]` - An app sitting over the most sensitive data path on the machine phones home, with consent opt-out rather than opt-in. Even 'anonymous' metrics are unacceptable from a clipboard tool and erode trust. → **vbuff:** vbuff ships with zero telemetry by default and is fully functional offline. Any optional diagnostics are strictly opt-in, documented, local-first, and provably exclude clipboard content. Make the no-telemetry guarantee auditable (open source, no network calls in the capture/storage path). _(seen in: ClipboardFusion, CopyClip, closed-source vendor managers)_
334. **Secrets written into application logs / crash reports** `[high]` - Debug logging, error messages, or crash dumps include the clipboard payload, so secrets leak into log files, stderr, system logs, or uploaded crash reports - bypassing the encrypted store entirely. → **vbuff:** vbuff never logs clipboard content at any level. Wrap payloads in a `Secret`/`Zeroizing` newtype whose Debug/Display redacts to a length+type summary. Strip payloads from panic messages and crash handlers; scrub any backtrace. Add a lint/test that fails if a payload type is formatted into a log macro.
335. **No lock / authentication gate on history** `[high]` - Anyone at an unlocked machine, a shoulder-surfer, or someone with a brief unattended session can open the picker and read the entire copy history, including recently copied secrets. → **vbuff:** vbuff supports an optional lock requiring OS biometric/credential auth (Touch ID, Windows Hello, polkit) to open history or reveal sensitive items, plus auto-lock on screen lock / idle timeout. Sensitive-classified items stay masked until authenticated. _(seen in: Ditto, CopyQ, Maccy, Flycut, Windows Win+V)_
336. **Sync uploads clipboard to a vendor/platform cloud** `[high]` - Sensitive copies leave the device to servers the user must trust, sometimes via a default-on or easily-missed toggle (Microsoft account cloud clipboard, Google Drive backing store). Exposure expands far beyond the local machine. → **vbuff:** vbuff defaults to local-only, no cloud. If sync exists it is opt-in, peer-to-peer or self-hostable, with no vendor-readable copy. Sensitive-classified items are excluded from sync by default. Be explicit in the UI about exactly what would leave the device. _(seen in: Windows Win+V cloud sync, 1Clipboard, ClipboardFusion)_
337. **Sync without end-to-end encryption** `[high]` - Cross-device sync that trusts the server or transport means the provider (or anyone who compromises it) can read every copied secret. 'Private cloud' marketing claims are unverifiable. → **vbuff:** vbuff's sync (if present) is end-to-end encrypted with keys that never leave the user's devices; the server/relay sees only ciphertext. Use a vetted protocol (e.g. Noise/age-style key agreement), document the threat model, and make the implementation auditable rather than relying on trust-us claims. _(seen in: Paste, PastePal, PasteNow, commercial sync managers)_
338. **Network/LAN sync unencrypted in transit** `[medium]` - LAN or peer sync sent in cleartext exposes copied data to anyone sniffing the network or performing a MITM, turning convenience into a leak channel. → **vbuff:** vbuff requires TLS/Noise for all transport with certificate/key pinning between paired devices and authenticated pairing (out-of-band code). No cleartext fallback. Reject downgrade. _(seen in: Ditto network sync, LAN-sync managers)_
339. **Routing the history DB through cloud-synced folders** `[medium]` - Placing the DB in Dropbox/Google Drive/iCloud Drive uploads the plaintext history to a third party and causes DB locking/corruption and conflict files, leaking secrets while also breaking reliability. → **vbuff:** vbuff stores its DB in a per-OS app-private data dir (never a synced folder), refuses to run with its store in a known cloud-sync path (warn the user), and provides its own E2E sync rather than file-level cloud sync. _(seen in: Ditto)_
340. **Residual data left on disk after uninstall / 'clear all'** `[medium]` - After the user uninstalls or clears history, the DB, WAL files, caches, and config dirs survive, so a secret-laden history remains on disk past what the user believes was a wipe. → **vbuff:** vbuff provides a verified 'wipe and uninstall' that securely deletes the store, WAL/temp/index/cache files, keychain entries, and config. Document exact file locations. Because records are encrypted, orphaned files are useless without the keychain key, which is also destroyed. _(seen in: Ditto, SQLite-based managers)_
341. **OCR-on-everything turns images into searchable secret text** `[medium]` - Running text recognition over every copied image/screenshot converts a photo of a card, ID, or receipt into searchable plaintext stored in history, broadening what is at risk and what malware can scrape. → **vbuff:** vbuff keeps OCR off by default, runs it strictly on-device (no cloud OCR), and applies the same secret classifier + TTL + exclusion rules to OCR output. Never persist OCR text for items classified sensitive. Make per-source OCR opt-in. _(seen in: KeepClips, Clipory, ClipZ)_
342. **Storing secrets unprotected in process memory** `[medium]` - Even with an encrypted disk store, plaintext clipboard buffers live in heap memory, can be swapped to disk, captured in a core dump, or read by a debugger/other process, exposing secrets at runtime. → **vbuff:** vbuff wraps secret payloads in zeroize-on-drop buffers (`zeroize`/`secrecy` crates), minimizes plaintext lifetime, avoids unnecessary clones, and where the OS allows locks pages (mlock/VirtualLock) and disables core dumps for the process handling sensitive content.
343. **Hardcoded or file-stored encryption key** `[critical]` - An encryption key compiled into the binary or stored in a config file next to the DB provides no real protection: anyone with the binary or disk access decrypts everything. 'Encrypted' becomes theater. → **vbuff:** vbuff derives/stores the master key only in the OS secure enclave/keychain (Keychain, DPAPI, Secret Service/kernel keyring). The key is never in source, never beside the DB, and is unlocked via OS user auth. Use per-install random keys.
344. **World-readable file/dir permissions on store and config** `[high]` - The history DB or config is created with loose permissions, so other local users (or other-uid daemons) can read clipboard history directly, sidestepping any in-app gate. → **vbuff:** vbuff creates its data dir and files with owner-only permissions (0700 dir / 0600 files on Unix, restricted ACL on Windows), verifies/tightens permissions on startup, and refuses to operate from a world-writable/world-readable location.
345. **Insecure local IPC between daemon and UI/picker** `[high]` - A capture daemon and UI that talk over an unauthenticated TCP port, a predictable named pipe, or a world-accessible Unix socket let any local process inject paste commands or siphon the entire history. → **vbuff:** vbuff uses an OS-private IPC channel (Unix domain socket in a 0700 dir with SO_PEERCRED uid check, Windows named pipe with a restrictive SDDL/SID check) and authenticates both ends. Never bind a network socket. Validate the peer's identity on every connection.
346. **No defense against clipboard hijacking (clipper malware)** `[high]` - Clipper malware silently replaces copied crypto addresses or other data with attacker lookalikes before paste; a history manager that just records changes amplifies exposure and gives the user no warning. → **vbuff:** vbuff detects suspicious clipboard mutations: flag when clipboard content changes to a different crypto-address/IBAN within milliseconds of a copy without a user copy action, warn before pasting an address that differs from what was copied, and offer address-format verification. Surface unexpected owner changes. _(seen in: Android clipper malware, Laplas Clipper, desktop clipper malware)_
347. **Centralized searchable store of everything copied (Recall-style risk)** `[high]` - A single, indexed, plaintext store of everything ever copied is a prime infostealer target - the same criticism leveled at Microsoft Recall. One read of that store dumps the user's entire secret history. → **vbuff:** vbuff minimizes the blast radius: encrypt at rest, default-exclude/short-TTL sensitive items, cap retention, keep a bounded history rather than forever, and offer an incognito/pause mode. Make 'don't remember this' a one-keystroke action. _(seen in: Windows clipboard history, any persistent history manager)_
348. **No incognito / pause-capture mode** `[medium]` - Users handling secrets for a stretch (entering banking, working with credentials) have no way to temporarily stop recording, so everything they copy is captured even when they know it's sensitive. → **vbuff:** vbuff offers a global pause/incognito toggle (hotkey + menu) that stops all capture and a per-window 'don't record from this app right now' option. Show a clear indicator while paused/recording so state is never ambiguous.
349. **Search index leaks plaintext of encrypted items** `[high]` - Even if the main store is encrypted, a full-text search index (FTS table, separate index file) often holds plaintext tokens of every item, including secrets, creating a second readable copy on disk. → **vbuff:** vbuff encrypts the search index too, or builds it from a keyed/blind index (encrypted tokens) so the index never contains plaintext. Sensitive-classified items are excluded from indexing by default. Apply the same secure-delete to index entries.
350. **Unbounded retention of sensitive history** `[medium]` - With no cap on count or age, secrets copied months ago remain on disk and in the picker, maximizing what an attacker recovers and what a forensic exam finds. → **vbuff:** vbuff enforces a configurable max item count and max age with sane defaults, and a separate stricter cap for sensitive-classified items. Eviction securely deletes (encrypted record + index). Surface current retention clearly so users aren't surprised. _(seen in: Windows ActivitiesCache.db, history-keeping managers generally)_
351. **Clearing history doesn't clear the live clipboard (and vice versa)** `[high]` - User purges history believing the secret is gone, but the live system clipboard still holds it and remains pasteable; or clears the live clipboard while the manager's history retains the secret. → **vbuff:** vbuff's 'clear' clears both the persisted history and the live system clipboard atomically (and offers each independently with explicit labels). After clearing a sensitive item, overwrite the live clipboard with empty/neutral content so it can't be pasted. _(seen in: Klipper, Plasma clipboard, KeePassXC interaction)_
352. **Defeating password managers' clipboard auto-clear** `[high]` - KeePassXC/Bitwarden wipe a copied credential from the live clipboard after N seconds, but the manager already snapshotted it into history, so the secret survives the very auto-clear meant to protect it. → **vbuff:** vbuff honors concealment hints so password-manager copies aren't stored at all; if a credential is detected, it does not capture, and it never resurrects the live clipboard after a password manager clears it. Coordinate with the OS clear rather than re-asserting old content. _(seen in: Klipper, Plasma clipboard, KeePassXC, CopyQ)_
353. **Re-asserting clipboard ownership exposes stale secrets** `[medium]` - Managers that re-claim clipboard ownership to 'persist' content (a Wayland/X11 workaround) can re-publish an already-cleared secret back into the live clipboard, undoing a password manager's clear and leaking it to the next reader. → **vbuff:** vbuff never re-publishes content the user/source cleared or that is classified sensitive. Persistence helpers must respect concealment + clear events and drop sensitive payloads from any in-memory persistence buffer immediately. _(seen in: wl-clip-persist, Firefox Wayland, sync tools)_
354. **Misses copies during rapid succession (gap captures secrets inconsistently)** `[medium]` - While reliability is a separate domain, in the security context inconsistent capture means concealment/exclusion checks can be skipped under load, letting a secret slip into history when copies fire quickly. → **vbuff:** vbuff applies concealment/exclusion/classification checks atomically per capture even under burst load, and fails closed (skip storage) rather than open (store unchecked) if a check cannot complete in time.
355. **Backups/Time Machine/snapshots include plaintext history** `[medium]` - An unencrypted history DB gets swept into Time Machine, Windows File History, or filesystem snapshots, persisting copied secrets in backups the user can't easily scrub. → **vbuff:** vbuff encrypts at rest (so backups capture only ciphertext) and sets the platform 'exclude from backup' attribute on its store where appropriate (e.g. macOS NSURLIsExcludedFromBackupKey), documenting that the keychain key is the protection boundary.
356. **Auto-update channel without signature verification** `[high]` - An updater that fetches and runs binaries over plain HTTP or without verifying signatures lets a MITM ship malware into a tool that already has full clipboard access - a catastrophic supply-chain foothold. → **vbuff:** vbuff updates only over HTTPS with mandatory code-signing/notarization and an independent update-signature check (e.g. minisign/Ed25519 over the artifact) before applying. No silent unsigned binary execution. Pin the update key.
357. **Storing secrets in shell/sync history or temp files during paste** `[high]` - Some managers shell out (xdotool, pbpaste pipelines, temp files) to perform paste, leaking the secret into shell history, process argv (visible via ps to other users), or world-readable /tmp files. → **vbuff:** vbuff never passes clipboard content as command-line arguments (visible in process listings) and never writes secrets to temp files. It uses in-process platform APIs for injection; if an external tool is unavoidable, pass data via an authenticated pipe/fd, not argv or files.
358. **Closed-source / unverifiable handling of sensitive data** `[medium]` - A closed-source manager asks users to trust marketing claims ('private', 'we never see your data') over the most sensitive data path on the machine, with no way to audit encryption, key handling, or network behavior. → **vbuff:** vbuff is open source so its capture, storage, encryption, and (no-)network behavior are independently auditable. Publish the threat model and reproducible builds so the no-leak guarantee is verifiable, not asserted. _(seen in: Paste, PastePal, PasteNow, closed-source commercial managers)_
359. **No way to delete a single sensitive item quickly / forget-last** `[medium]` - After accidentally copying a secret, the user has no fast path to remove just that entry and its traces, so they either leave it or nuke everything; in practice the secret lingers. → **vbuff:** vbuff offers a one-keystroke 'forget last copy' and per-item secure delete (record + index + live clipboard if it matches), with immediate secure overwrite rather than soft-hide.
360. **Exposing history over an accessibility/automation surface** `[medium]` - To enable auto-paste, managers expose history through accessibility APIs or scriptable interfaces (AppleScript, AT-SPI, UI Automation) that other apps can query, turning the secret store into an automation-readable resource. → **vbuff:** vbuff does not expose clipboard history through scriptable/accessibility surfaces; it requests the minimum permission needed only for keystroke injection at paste time and keeps the history store out of any queryable automation API. Audit what the accessibility entitlement actually exposes.
361. **Image/binary payloads bypass the secret pipeline** `[medium]` - Secret classification, exclusion, and TTL are applied only to text, so a copied QR code of a seed phrase, a screenshot of a password field, or an image of a card is stored unprotected forever. → **vbuff:** vbuff routes all content types (text, RTF/HTML, images, files) through the same concealment + exclusion + retention pipeline. Images get optional on-device OCR-classification before storage decisions, and file paths to sensitive locations are treated cautiously.
362. **No protection against screen-capture of the picker** `[medium]` - The history picker renders secrets in plaintext on screen; screen-recording malware or a remote-control session captures the entire history visually, even if the disk store is encrypted. → **vbuff:** vbuff masks sensitive-classified items in the picker by default (reveal on auth), and where the OS supports it marks the picker window as excluded from screen capture/recording (e.g. macOS sharingType / Windows SetWindowDisplayAffinity WDA_EXCLUDEFROMCAPTURE).
363. **Trusting clipboard MIME content type without sanitization** `[medium]` - Treating clipboard HTML/RTF as trusted and rendering it in the picker (e.g. via an embedded webview) can execute injected markup/script or load remote resources, leaking that the user copied something or enabling injection. → **vbuff:** vbuff renders previews as inert sanitized text, never executing HTML/JS or loading remote resources (no auto-fetched images) from clipboard content. Strip active content and remote references before any preview.
364. **Pasting wrong/stale content into security-sensitive fields** `[high]` - Focus/timing bugs cause the manager to inject a previously copied secret into the wrong field or app (e.g. a password ends up typed into a chat box or URL bar), directly disclosing it. → **vbuff:** vbuff verifies the intended target window/field has focus immediately before injection, fails closed if focus is ambiguous, and warns before auto-pasting a sensitive-classified item. Never auto-type into a field of a different app than the one that had focus at trigger time. _(seen in: Ditto (literal 'v' / wrong target), ArsClip, CopyQ)_
365. **Default-broad capture with no first-run privacy disclosure** `[medium]` - On install the manager silently begins recording everything copied with no explanation of what's stored, where, for how long, or how to exclude sources, so users handle secrets unaware they're being persisted. → **vbuff:** vbuff shows a clear first-run privacy summary (what is captured, where it's stored, encryption, retention, how to pause/exclude) and ships privacy-protective defaults (encryption on, telemetry off, sensitive auto-skip, bounded retention) rather than burying them in settings.

## 11. UI / UX mistakes

366. **Popup steals focus from the active app** `[critical]` - When the picker grabs keyboard focus, the OS no longer knows which field was the paste target, so after selecting a clip the user has to click back into their document and Ctrl/Cmd+V manually, defeating the one-keystroke promise of a clipboard manager. → **vbuff:** Show the picker as a non-activating panel (NSPanel with nonactivatingPanel style on macOS, WS_EX_NOACTIVATE on Windows, layer-shell overlay without focus grab where possible on Wayland) so the previously focused app stays the key window. Capture the focused-window handle at summon time and restore it before injecting the paste, exactly like Spotlight keeps the underlying app active. _(seen in: Maccy, ClipMenu, CopyQ, Ditto)_
367. **Picker opens behind other windows or on the wrong monitor** `[high]` - The global hotkey fires but the panel renders behind the active window or on a different display than the cursor, so the user sees nothing happen, presses the hotkey again, and ends up toggling it open/closed or hunting across monitors. → **vbuff:** Always raise the panel to the top window level (floating/utility level) on summon, and position it on the screen that currently contains the mouse cursor (or the focused window), reading live multi-monitor geometry each time rather than caching a single screen. _(seen in: Ditto, CopyQ)_
368. **Slow summon: history fully loaded into the view on every open** `[critical]` - Loading the entire history (tens of thousands of rows, with image thumbnails) into the list before the window can paint makes the picker take seconds to appear, so it feels crashed and users give up waiting for clips they need instantly. → **vbuff:** Render a virtualized/lazy list that materializes only the visible rows plus a small buffer, query the store paginated, and decode thumbnails off the main thread. Target sub-100ms time-to-first-paint regardless of history size; never block window display on full-history load. _(seen in: Maccy, Pastebot, CopyQ)_
369. **Janky scrolling and laggy typeahead with large history** `[high]` - Once history grows, scrolling stutters and there is a visible delay between each keystroke in the search box and the list filtering down, making the picker feel broken precisely when the user has the most clips to sift through. → **vbuff:** Filter incrementally against an in-memory index on a background task, debounce input lightly, and keep row rendering cheap (no per-row layout thrash, no synchronous image decode). Keep the visible frame at 60fps by recycling row widgets. _(seen in: Maccy, CopyQ)_
370. **No keyboard-only flow (mouse required to pick or paste)** `[critical]` - Power users live on the keyboard; if you must reach for the mouse to click a clip, or arrow-keys/Enter do not select-and-paste, the tool is slower than just re-copying the original, which kills daily adoption. → **vbuff:** Make the whole loop keyboard-driven: global hotkey opens with the search field focused, arrow keys/Ctrl+J-K navigate, number keys (1-9) jump to top items, Enter pastes-and-dismisses, Esc closes. Mouse is optional, never required. _(seen in: Diodon)_
371. **Arrow-key navigation and Esc-to-close do not work in the popup** `[high]` - On some platforms (notably Wayland) the popup cannot take keyboard focus, so arrow keys do nothing and Esc will not dismiss it, trapping the user in a window they can only close with the mouse. → **vbuff:** Ensure the popup is a focusable surface on every supported platform; verify on Wayland via the layer-shell keyboard-interactivity flag. Add an automated smoke test that the window receives key events, and always wire Esc to close as a hard guarantee. _(seen in: Diodon)_
372. **Hotkey + Enter opens picker but never actually pastes** `[high]` - The user does the full keyboard motion (summon, select, Enter) and the window closes, but nothing lands in the target field, so the interaction looks complete yet silently fails and the user loses trust in every paste. → **vbuff:** Treat paste as the committed action of Enter: restore focus to the captured target, set the clipboard, inject the paste keystroke, and confirm. If injection cannot be guaranteed (e.g. no accessibility permission), do not silently swallow it; surface an inline status and fall back to copy-to-clipboard with a clear hint. _(seen in: Maccy, CopyQ, Flycut)_
373. **Hotkey collides with built-in OS shortcuts** `[medium]` - Default or user-set hotkeys overlap system shortcuts (e.g. macOS Cmd+Shift+3/4/5 screenshots), so summoning history instead fires a screenshot or some other system action, corrupting the user's intent. → **vbuff:** Ship a default hotkey known to be free of common OS/screenshot/IME conflicts, validate the chosen combo at set-time against a known-collision list and warn, and detect at runtime when registration fails so the user is told rather than left with a dead key. _(seen in: Maccy, CopyClip 2)_
374. **No visible feedback when the global hotkey fails to register** `[medium]` - Another app grabbed the shortcut, so the picker silently never appears; the user presses the key, nothing happens, and they cannot tell whether the app is running, broken, or the key is wrong. → **vbuff:** Detect failed hotkey registration on startup and on conflict, show a clear, non-blocking notice in the UI (and tray menu) naming the conflict and offering a one-click rebind, instead of a generic 'Failed to set hotkey' on every launch or silence. _(seen in: ArsClip, Ditto, ClipboardFusion)_
375. **Window is tiny, fixed-size, and non-resizable** `[medium]` - A cramped fixed window truncates long clips and shows only a handful of rows, forcing endless scrolling and making multi-line snippets or code impossible to read or verify before pasting. → **vbuff:** Make the window resizable, remember the user's last size/position, and let row count and preview pane scale with it. Provide sensible min/default sizes that show ~10 rows and a readable preview without configuration. _(seen in: Paste)_
376. **No content previews for clips** `[high]` - When list rows show only a tiny snippet or a generic label, the user cannot tell similar clips apart (two URLs, two code blocks, two addresses) and pastes the wrong one, then has to undo and retry. → **vbuff:** Show a rich preview: multi-line text preview with whitespace made visible, syntax-tinted code where detectable, image thumbnails for image clips, and a detail/preview pane for the highlighted item. Make the preview the primary disambiguator before paste. _(seen in: Alfred, Raycast, Maccy)_
377. **Images stored with no thumbnail or preview** `[medium]` - Image clips appear as blank rows or opaque labels with no thumbnail, so the user cannot find the screenshot they copied and must paste blindly to check, slowing every image workflow. → **vbuff:** Generate and cache a small downscaled thumbnail per image clip for the list, show a larger preview in the detail pane, and label with dimensions/format. Decode thumbnails lazily off the main thread so they never block list rendering. _(seen in: Alfred, Raycast, Maccy)_
378. **Items render blank or empty in the list** `[medium]` - A clip was captured but the row shows nothing visible, so it looks un-pasteable and confusing; the user cannot tell whether the content is empty or just unrendered, and skips items they actually need. → **vbuff:** Never render a row as visually empty: if text is whitespace-only or a format failed to render, show an explicit label (e.g. '3 spaces', 'whitespace', 'image 1200x800', 'unsupported preview') plus a type icon, so every row communicates what it holds. _(seen in: CopyQ, Maccy)_
379. **No dark mode / does not follow system appearance** `[medium]` - A blinding white panel summoned over a dark editor at night is jarring and hard to read, and a manager that ignores the OS theme looks foreign and dated next to every other app. → **vbuff:** Support light and dark themes that follow the OS appearance setting by default and switch live when the system toggles, with an optional manual override. Respect the system accent color where the platform exposes it.
380. **Cluttered, heavy UI for a tool that should be fast and out of the way** `[medium]` - Toolbars, panels, chrome, and decorative elements crowd out the actual clips, slow comprehension, and make a utility meant to disappear feel like a heavyweight app the user has to manage. → **vbuff:** Default to a minimal, single-column list with search at top and a preview pane; keep chrome to a minimum. Put advanced features behind a clearly labeled affordance, not in the user's face during the core copy-paste loop. _(seen in: Paste)_
381. **Confusing or undifferentiated type icons** `[low]` - If text, rich text, links, images, files, and code all look the same (or use cryptic icons), the user cannot scan the list by type and has to read every row, slowing recall. → **vbuff:** Use clear, distinct, conventional glyphs per content type (plain text, rich text, URL/link, image, file, color, code), shown consistently at a fixed position in each row, with a tooltip naming the type. Avoid ambiguous or novel iconography.
382. **No empty state or onboarding for a brand-new install** `[medium]` - On first run the history is empty by definition, so the user opens the picker, sees a blank window, and concludes the app is broken or not capturing, with no idea what to do next. → **vbuff:** Show a friendly empty state explaining 'Copy something and it appears here', the summon hotkey, and any permissions still needed (with a button to grant them). Provide a short first-run walkthrough of the hotkey, search, and paste flow.
383. **No 'no results' state when search matches nothing** `[low]` - Typing a query that matches nothing leaves a blank list identical to an error or empty history, so the user cannot tell if the item is gone, the search is broken, or they mistyped. → **vbuff:** Render a distinct 'No matching clips' state that echoes the query and offers to clear the search or search older items, visually different from the first-run empty state.
384. **Plain-text paste is buried or unreachable from the picker** `[high]` - Most paste-from-history is into plain contexts, but if 'paste as plain text' requires digging through menus or settings, users get unwanted fonts/colors/styles pasted into their document every time. → **vbuff:** Expose paste-as-plain-text as a first-class modifier directly in the picker (e.g. hold a modifier while pressing Enter, or a visible toggle), and let the user set plain-text as the default paste mode. Show in the preview which mode will be used. _(seen in: Maccy, Paste, Alfred, Pastebot)_
385. **Menus flicker open and shut; hover/selection states break after OS updates** `[high]` - The popup opens and immediately closes, items cannot be highlighted or selected, and hover states stop responding after an OS upgrade, making the tool unusable until a fix ships months later. → **vbuff:** Build the UI on a stable, well-supported toolkit rather than fragile platform menu internals, add integration tests that exercise open/hover/select on each target OS, and treat each OS beta cycle as a release gate to catch breakage before users do. _(seen in: ClipMenu, Clipy)_
386. **Paste surfaces as a notification instead of inserting into the field** `[high]` - Instead of typing the clip into the focused field, the action posts a system notification containing the text, so the user has to read it out of a banner and copy it again, which is the opposite of what they asked for. → **vbuff:** Never route paste output to notifications. If keystroke injection is unavailable, fall back to placing the content on the clipboard and showing a quiet inline hint to press the OS paste shortcut, not a notification dump of the content. _(seen in: Maccy)_
387. **Search box not focused on open / search is hidden** `[medium]` - If the user must click into or tab to the search field before typing, the fast 'summon then type to filter' flow is broken and every recall costs extra keystrokes. → **vbuff:** Focus the search field the instant the picker opens so the first keystroke filters immediately, and keep search always visible at a fixed location. Allow typing to filter even when a list row is highlighted.
388. **Pinned/favorite items cannot be reordered or are visually buried** `[medium]` - Users pin frequently used snippets but cannot drag them into order, and pins are mixed in or hard to find, so the feature meant to speed access actually slows it. → **vbuff:** Give pinned items a clearly separated, persistent section (or a dedicated tab) at the top, support drag-and-drop and keyboard reordering, and keep their order stable across restarts. _(seen in: Maccy, CopyQ)_
389. **No multi-select or bulk actions in the list** `[low]` - Cleaning up clutter or assembling several snippets one at a time is tedious; without multi-select the user deletes or copies items individually, which is slow and discourages housekeeping. → **vbuff:** Support range/multi-select (Shift/Ctrl+click and Shift+arrow) with bulk delete, bulk copy, and bulk pin, plus a select-all in the current filter. Confirm destructive bulk deletes. _(seen in: Maccy)_
390. **Duplicate rows clutter the list instead of deduping** `[low]` - Copying the same text repeatedly produces many identical rows, pushing useful older clips off-screen and making the list noisy and harder to scan. → **vbuff:** De-duplicate by content: when an identical clip is copied again, bump the existing entry to the top (updating its timestamp) rather than inserting a new row. Offer a setting for users who genuinely want every event. _(seen in: Maccy)_
391. **Picker does not auto-dismiss after paste** `[medium]` - If the window stays open after a clip is pasted, it lingers over the user's document and they must press Esc or click away every time, adding friction to a flow meant to be instantaneous. → **vbuff:** Dismiss the picker automatically the moment a paste is committed (unless the user holds a 'keep open' modifier for multi-paste), and restore focus cleanly to the target app.
392. **Hover popovers and overlays that will not dismiss** `[low]` - Preview popovers or tooltips that get stuck on screen obstruct the list and other windows, forcing the user to click around or relaunch the app to clear them. → **vbuff:** Tie every transient overlay to a clear dismissal rule (mouse-leave, focus change, Esc, or selection change), and ensure overlays never outlive the row or window they belong to. Add a test that overlays clear on window close. _(seen in: CopyClip 2)_
393. **No truncation/whitespace indication for long or whitespace-heavy clips** `[low]` - A long single-line clip silently truncates with no indication, and leading/trailing whitespace or tabs are invisible, so the user pastes something subtly different from what the row implied. → **vbuff:** Show an explicit truncation indicator with full length on hover/preview, render whitespace and line endings visibly in the preview pane, and display character/line counts so the user knows exactly what they are about to paste.
394. **Tray/menubar UI freezes and becomes unresponsive** `[medium]` - Opening settings or the tray menu hangs for many seconds (or indefinitely), forcing a force-quit and relaunch, which feels like the app crashed and erodes confidence in an always-on utility. → **vbuff:** Keep the tray/menubar UI thread free of blocking work: load settings and history asynchronously, never run DB or file scans on the UI thread, and add a watchdog so the menu always opens instantly even while background work proceeds. _(seen in: ClipboardFusion)_
395. **Clip ordering corrupts (newest not on top)** `[medium]` - When pinning or settings glitch, the list stops being chronological and newest clips no longer appear at the top, so the user's mental model ('my last copy is right there') breaks and recall becomes guesswork. → **vbuff:** Define one canonical sort (most-recent-first within unpinned, pinned in their own ordered section) and enforce it in the view layer independent of storage quirks. Add a test that verifies ordering after pin/unpin and restart. _(seen in: Ditto)_
396. **No visible indication of which app/field paste will target** `[medium]` - On multi-window setups the user cannot tell whether paste will go to the editor they meant or some background window, so they paste hopefully and sometimes drop content into the wrong place. → **vbuff:** Show the captured target (app name and optionally window title) in the picker header at summon time, so the user can confirm the destination before pressing Enter, and refuse to paste if the target is gone. _(seen in: CopyQ, ArsClip)_
397. **Summon hotkey toggles unpredictably (double-press opens then closes)** `[low]` - If summon and dismiss share the hotkey but timing is loose, a quick double tap (or a key repeat) opens and instantly closes the window, leaving the user unsure whether it is open. → **vbuff:** Debounce the toggle, make the open/close transition deterministic against rapid presses, and prefer a model where the summon hotkey always shows (and re-focuses search) rather than toggling, with Esc as the dismiss.
398. **Cursor-relative vs fixed positioning ignores where the user is looking** `[low]` - A picker that always opens dead-center or in a corner forces the user's eyes and attention away from the caret they were typing at, slowing the find-and-paste loop and feeling disconnected from context. → **vbuff:** Offer a positioning mode that opens near the text caret/cursor when its location is known (falling back to the active screen center), and let the user choose between cursor-anchored and fixed placement. Keep placement stable within a session.
399. **No accessibility support (screen reader, contrast, scaling, reduced motion)** `[medium]` - List rows without accessible labels, low-contrast text, fixed font sizes that ignore OS scaling, and animations that ignore reduce-motion exclude users who rely on assistive tech and strain everyone else. → **vbuff:** Label every row and control for the platform accessibility API, meet WCAG AA contrast in both themes, honor OS text-size/scaling and reduce-motion settings, and ensure full screen-reader navigation of the list and search.
400. **Window does not remember size, position, or last state** `[low]` - If the picker resets to a default size and place on every summon, users who resized or moved it must redo it constantly, and the tool feels like it forgets them. → **vbuff:** Persist and restore window size and position (per display where sensible) across summons and restarts, while still clamping to the currently active screen so a remembered position on a now-disconnected monitor does not send the window off-screen.
401. **No quick way to inspect/expand a clip before pasting** `[low]` - For long text, code, or rich content, a one-line row is not enough to verify the clip, but if there is no expand/quick-look the user must paste it somewhere just to read it. → **vbuff:** Provide a quick-expand action (e.g. Space or a peek key) that opens a full read-only view of the highlighted clip, including formatting and metadata, without leaving the picker or committing a paste.
402. **Destructive actions (clear all) have no confirmation or undo** `[medium]` - A misclick on 'Clear history' wipes everything irreversibly, and without confirmation or undo the user loses pinned snippets and recent work in one stroke. → **vbuff:** Require confirmation for clear-all and bulk deletes, exempt pinned items by default (with an explicit option to include them), and offer a short-lived undo for the last destructive action.
403. **Loading/transition state missing on first paint** `[low]` - If there is any unavoidable delay (cold start, large store migration), showing a blank or frozen window makes the app look hung; users force-quit before it would have finished. → **vbuff:** Show an immediate lightweight skeleton/spinner frame the instant the hotkey fires, then stream rows in as they load, so the user always gets instant visual acknowledgement that the summon registered.

## 12. Hotkey & input mistakes

404. **No global hotkey at all** `[critical]` - User must click a tray/menu-bar icon or alt-tab to the app every time, which defeats the speed advantage of a clipboard manager and makes recall slower than just retyping. → **vbuff:** Ship a working global hotkey out of the box (sane default per OS, e.g. Cmd+Shift+V on macOS, Win+Shift+V on Windows, Super+V on Linux) that summons the picker from any focused app without the app needing foreground focus first.
405. **Hotkey not customizable** `[high]` - The fixed default clashes with the user's existing muscle memory or another app's binding and there is no escape hatch, so the feature is unusable for that user. → **vbuff:** Expose a full hotkey-capture UI that records any modifier+key combo (including F-keys and media keys), validates it, persists it, and lets the user clear/reset to default.
406. **Hotkey conflicts with OS reserved shortcuts** `[medium]` - Choosing a combo the OS owns (Cmd+Shift+3/4/5 screenshots on macOS, Win+L lock, etc.) means the OS action fires instead of the picker, e.g. taking a screenshot when the user wanted to paste. → **vbuff:** Maintain a per-OS deny/known-conflict list, refuse or warn when the user picks a reserved combo, and never ship a default that overlaps a documented system shortcut. _(seen in: Maccy)_
407. **Hotkey conflicts with other apps with no detection** `[high]` - Another app (terminal, IDE, screenshot tool, second clipboard manager) already grabbed the combo, so the picker silently never appears and the user assumes the app is broken. → **vbuff:** Detect failed hotkey registration at startup, surface a clear, persistent in-app warning naming the conflict, and offer one-click rebinding instead of failing silently. _(seen in: Maccy, Flycut, CopyQ)_
408. **Silent 'failed to set hotkey' on every startup** `[medium]` - When registration fails (combo taken), some apps either show a cryptic error every launch or silently give up, leaving the user without a working summon and no actionable next step. → **vbuff:** On registration failure show a single actionable dialog/banner (which key, which app likely holds it, button to pick a new one), retry registration after rebind, and remember the resolution so it does not nag every boot. _(seen in: ArsClip, Ditto, ClipboardFusion)_
409. **Global hotkey broken on Wayland** `[critical]` - On Wayland an ordinary app cannot grab global hotkeys, so the summon shortcut never fires and the entire shortcut-driven workflow silently dies even though the app looks like it is running. → **vbuff:** Register global shortcuts through the XDG GlobalShortcuts portal (org.freedesktop.portal.GlobalShortcuts) where available, fall back to wlroots/compositor-native binding, and when neither exists, show in-app instructions to bind the picker via the DE settings rather than pretending the hotkey works. _(seen in: CopyQ, Diodon, GPaste)_
410. **Hotkey stops working after sleep/wake** `[high]` - After the machine sleeps and wakes, the global hook is silently dropped and the shortcut no longer summons the picker until the app is restarted, so the user thinks it died at random. → **vbuff:** Subscribe to system sleep/wake (and session lock/unlock) notifications and re-register the global hotkey on resume; add a periodic self-check that re-registers if the binding is found unregistered.
411. **Hotkey stops working after OS/app update or permission revocation** `[high]` - An OS upgrade or app update silently drops the Accessibility/Input-Monitoring grant or the registered hook, so the hotkey and auto-paste quietly stop and the user has no idea why. → **vbuff:** Verify required input/accessibility permissions and hook registration on every launch and on permission-change events, show a clear in-app status indicator when they are missing, and deep-link to the exact OS settings pane to re-grant. _(seen in: Maccy, Paste, Pastebot, Flycut, Clipy)_
412. **Modifier-only bindings (e.g. double-Cmd)** `[medium]` - Binding the picker to a bare modifier or modifier-only chord causes accidental triggers during normal typing and is impossible to enter reliably, so the picker pops up unexpectedly or not at all. → **vbuff:** Require at least one non-modifier key in custom bindings by default; if offering modifier-only/double-tap triggers, gate them behind an opt-in and a tunable timing threshold, and exclude them from the recommended defaults.
413. **No number/quick-select keys** `[high]` - Without 1-9 (or 0) shortcuts to instantly paste the Nth recent item, every recall requires arrowing or mousing through the list, which is slow for the common case of grabbing one of the last few clips. → **vbuff:** Bind digit keys 1-9 (and 0) to paste the corresponding visible item immediately, show the number badge next to each row, and let the user paste-the-Nth even without opening the full UI via an optional direct combo.
414. **No type-to-filter / search-as-you-type** `[high]` - The picker forces scrolling instead of letting the user just start typing to narrow the list, so finding a clip among hundreds is tedious and the keyboard-only flow breaks. → **vbuff:** Make the search field focused on open so typing immediately filters fuzzily across content (and OCR text), with the first match auto-highlighted so Enter pastes it; keep filtering responsive even on large histories.
415. **Picker steals focus from the target app** `[high]` - Opening the picker takes keyboard focus away from the field the user was in, so after selecting a clip the paste lands nowhere or in the wrong app and the user must click back manually. → **vbuff:** Open the picker as a non-activating/accessory panel that overlays without deactivating the previous app (like Spotlight), remember the prior frontmost window/field, and restore focus to it before injecting the paste. _(seen in: Maccy, ClipMenu, CopyQ)_
416. **Picker opens behind other windows** `[high]` - The summoned panel appears under the active window or off to the side, so the hotkey seems to do nothing and the user cannot find or type into the picker. → **vbuff:** Always present the picker as a topmost floating panel at the cursor/active-screen, raise and key-focus it on summon, and verify it is on the screen that currently has focus in multi-monitor setups. _(seen in: Ditto, CopyQ)_
417. **Hotkey opens picker but Enter never pastes** `[high]` - The summon works and an item can be highlighted, but pressing Enter selects without injecting the paste, so the round trip silently fails and the user has to Ctrl/Cmd+V manually anyway. → **vbuff:** Make Enter (and digit/quick-select) perform select-then-paste atomically as the default action, with auto-paste on by default and a clearly-labeled fallback that copies-only when injection is blocked. _(seen in: Maccy, CopyQ)_
418. **Wrong keystroke leaks into the target (literal 'v')** `[high]` - When simulating Ctrl/Cmd+V the app releases the modifier at the wrong time and the bare 'v' (or other key) gets typed into the field, silently corrupting the user's input. → **vbuff:** Synthesize the paste with correct key-down/key-up ordering and modifier flags, verify the modifier is actually held during the key event, and prefer the OS paste action/services API over raw keystroke injection where available. _(seen in: Ditto)_
419. **Auto-paste keystroke does not reach Electron/secure/Spotlight fields** `[medium]` - Injected paste keystrokes silently fail in certain targets (Electron apps, secure/password fields, Spotlight), so selecting a clip does nothing and the user cannot tell why it works elsewhere but not here. → **vbuff:** Detect when injection fails or the target is a secure field, fall back to leaving the clip on the system clipboard plus a clear 'press Cmd/Ctrl+V to paste' hint, and document per-app limitations instead of silently no-op-ing. _(seen in: Maccy)_
420. **Paste targets the wrong window after summon (Wayland/multi-window)** `[high]` - After the picker grabs focus it cannot reliably refocus the original window, so the paste goes to the previously-active or background window instead of where the user wanted it. → **vbuff:** Capture the target window handle at summon time, restore focus to that exact window before injecting, and on Wayland use the portal RemoteDesktop/clipboard handoff rather than guessing the active window. _(seen in: CopyQ, cliphist)_
421. **No keyboard navigation of the list** `[high]` - If arrow keys, Home/End, PgUp/PgDn, and Tab do not move the selection, the picker forces mouse use, which is slow and breaks the keyboard-only workflow the hotkey was meant to enable. → **vbuff:** Provide full keyboard navigation (arrows, Home/End, page keys, Tab between sections), keep the selected row visible/auto-scrolled, and ensure focus stays inside the list so every action is reachable without the mouse. _(seen in: Diodon)_
422. **Esc does not close / focus cannot reach the picker (Wayland)** `[medium]` - On Wayland the popup sometimes cannot take keyboard focus, so arrow keys do nothing and Esc will not dismiss it, trapping the user with a panel they can only close with the mouse. → **vbuff:** Request keyboard focus via layer-shell/xdg keyboard-interactivity on open, bind Esc to dismiss reliably, and if focus cannot be obtained, surface a visible warning rather than presenting a dead panel. _(seen in: Diodon)_
423. **Single hotkey for everything (no dedicated actions)** `[medium]` - Offering only one summon shortcut means common variants (paste-as-plain-text, paste-and-keep-open, paste-last-item directly) all require extra navigation, slowing the frequent flows. → **vbuff:** Allow multiple independently-bindable global actions: open picker, paste plain text, paste last item, toggle clipboard capture, and quick-select Nth, each separately customizable and individually clearable.
424. **No paste-as-plain-text shortcut** `[medium]` - Reaching plain-text paste is buried in menus or requires a modifier combo the user must remember, so the very common need to strip formatting becomes a chore. → **vbuff:** Provide a first-class, separately-bindable plain-text-paste action (e.g. a modifier held during Enter, and an optional dedicated global hotkey) and make it discoverable in the picker. _(seen in: Maccy, Paste, Alfred, Pastebot)_
425. **Same hotkey to open and close behaves inconsistently** `[low]` - If pressing the summon key again does not toggle the picker shut (or worse, re-opens a second instance), the user cannot quickly dismiss it and the interaction feels unpredictable. → **vbuff:** Make the global hotkey a clean toggle: pressing it while the picker is open closes it and restores focus to the prior target; guard against duplicate windows from rapid repeats.
426. **Hotkey auto-repeat causes runaway opens** `[low]` - Holding the combo a beat too long fires key-repeat, opening/closing the picker rapidly or queuing multiple pastes, which is jarring and can paste the wrong thing. → **vbuff:** Trigger on key-down once and debounce until key-up, ignore auto-repeat events for the global binding, and rate-limit the summon/paste actions.
427. **No way to dismiss/cancel without pasting** `[medium]` - If there is no clear cancel (Esc or click-away) that closes the picker without inserting anything, an accidental summon risks pasting an unwanted clip into the document. → **vbuff:** Bind Esc and outside-click to close-without-paste, never auto-paste on blur, and require an explicit confirm key (Enter/digit/click) before any injection happens.
428. **Hotkey hardcoded to one keyboard layout** `[medium]` - Binding by physical scancode-only or by character-only breaks for non-US layouts and multi-layout users, so the combo fires on the wrong physical key or not at all after a layout switch. → **vbuff:** Register hotkeys against keycodes resolved through the active layout, re-resolve on layout-change events, and let the user capture the binding by pressing the actual physical keys they intend.
429. **Numeric quick-select tied to physical layout, not the visible list** `[medium]` - If digit shortcuts map to fixed slots rather than the currently-filtered/visible rows, pressing 1 after typing a filter pastes an unexpected item, breaking the type-then-number flow. → **vbuff:** Make 1-9 always refer to the currently-displayed (filtered, scrolled) rows in visual order, update the number badges live as the filter changes, and never paste from a stale pre-filter index.
430. **Hotkey grabbed exclusively, breaking the underlying key for other apps** `[medium]` - Registering the combo with an exclusive/blocking grab swallows the keystroke so apps that legitimately use the unmodified key (or the combo) stop receiving it system-wide. → **vbuff:** Grab only the exact modifier+key combo, pass through everything else, and prefer non-exclusive OS hotkey registration so the binding never starves other apps of input.
431. **No discoverability of shortcuts in the picker** `[low]` - Even when number-select, plain-paste, and navigation keys exist, users never learn them because nothing in the UI hints at them, so power features go unused. → **vbuff:** Show number badges on rows, a compact key-hint footer in the picker, and a discoverable shortcuts/help overlay (e.g. '?'), so the available keys are visible at the moment of use.
432. **Hotkey settings silently reset or lost across restarts/updates** `[medium]` - A custom binding is forgotten after reboot or update and silently reverts to default (or to nothing), so the user's summon stops working and they must reconfigure it. → **vbuff:** Persist hotkey config in durable, versioned storage written atomically, migrate it across app updates, validate on load, and never overwrite a user binding with a default without explicit reset. _(seen in: CopyQ, CopyClip 2 (FIPLAB))_
433. **Search field captures navigation keys, blocking quick-select** `[medium]` - When the focused search box swallows digit keys and arrows for text entry, the user cannot use 1-9 quick-select or arrow navigation while typing a filter, forcing a mode switch. → **vbuff:** Reserve a clear modifier (or context) so digits and arrows drive selection even while the search box is focused, or auto-route plain arrows/Enter to the list and only treat digits as filter text when explicitly escaped.
434. **Injecting paste before the clipboard write has propagated** `[high]` - The app sets the clipboard then immediately fires Ctrl/Cmd+V, but on slow/delayed-render targets the paste reads the previous clipboard content, so the wrong (old) item is pasted. → **vbuff:** Confirm the clipboard owner/content is the intended item (or wait for the change to be observable) before synthesizing the paste keystroke, with a short bounded retry rather than a blind fixed delay.
435. **Hotkey only works when the app window has focus** `[critical]` - If the 'global' shortcut actually only fires while the app is frontmost, it is not global at all and is useless from inside other apps, which is the only place a clipboard picker is needed. → **vbuff:** Register a true system-wide hotkey via the OS global-hotkey API (and the GlobalShortcuts portal on Wayland), and explicitly test that it fires while a different app is focused.

## 13. Extensibility & integration mistakes

436. **No CLI at all** `[high]` - Power users and scripts cannot push, pull, list, or clear clips from a terminal. Every workflow that wants to feed the clipboard programmatically (CI output, git hooks, shell aliases) is impossible, forcing users back to platform primitives like pbcopy/xclip/clip.exe and abandoning the manager for automation. → **vbuff:** Ship a single self-contained `vbuff` binary that is both the daemon and a CLI. Provide subcommands: `vbuff copy [-]`, `vbuff paste [--index N]`, `vbuff list --json`, `vbuff search QUERY --json`, `vbuff get ID`, `vbuff delete ID`, `vbuff clear`, `vbuff pin ID`. Read stdin and write stdout so it composes in pipes. Exit codes must be meaningful (0 success, non-zero on miss). _(seen in: Maccy, Paste, Pastebot, Windows 11 Win+V)_
437. **No machine-readable output (human-only formatting)** `[high]` - When a CLI or query interface exists but only prints decorated, columnar, human-formatted text, scripts must parse fragile layouts that break on multi-line clips, tabs, or unicode. cliphist's `id<TAB>preview` list format collides with fzf/rofi/fuzzel column splitting, so naive configs select the wrong column and recall the wrong item. → **vbuff:** Offer a stable `--json` (and `--json-lines` / NDJSON for streaming) output mode on every read command, with a documented, versioned schema (id, content type, byte length, preview, timestamps, pinned flag). Keep IDs opaque and stable. Never embed user content in a delimiter-ambiguous single-line format as the only machine option. _(seen in: cliphist, clipcat)_
438. **No stdin/stdout pipe support** `[high]` - Even with a CLI, if commands cannot read content from stdin or emit a clip to stdout, the tool cannot participate in Unix pipelines (`cmd | vbuff copy`, `vbuff get ID | other`). Users cannot pipe command output into history or pipe a stored clip into another program, defeating scriptability. → **vbuff:** Make `vbuff copy` read raw bytes from stdin when given `-` or no argument, preserving the declared MIME type via `--type`. Make `vbuff get`/`vbuff paste --stdout` write the raw bytes (not a decorated preview) to stdout. Never inject ANSI color or trailing newlines into piped output. _(seen in: cliphist)_
439. **No import from other clipboard managers** `[high]` - Users with years of pinned snippets, templates, and history in Maccy/Ditto/CopyQ/GPaste cannot migrate without losing everything, so switching cost is effectively their entire history. This locks them into the incumbent and means they never try vbuff at all. → **vbuff:** Provide `vbuff import` with format adapters for the common managers' on-disk stores: Maccy/Flycut (Core Data/SQLite), Ditto (SQLite), CopyQ (its tab data), GPaste (history files), cliphist. Map content, type, pinned/favorite flag, and timestamps. Document the source paths and ship a dry-run mode that reports counts before writing. _(seen in: Maccy, Ditto, CopyQ, GPaste, Flycut)_
440. **No export of history** `[high]` - If history can only live inside the app, users cannot back it up, migrate to a new machine, version-control snippets, or leave the app without abandoning their data. This is the mirror of no-import: it makes vbuff itself a lock-in trap and erodes trust for a tool holding important data. → **vbuff:** Provide `vbuff export` to a documented, open, self-describing format (JSON Lines or a zip with a manifest plus binary blobs referenced by hash for images). Round-trip guarantee: `export` then `import` reproduces content, types, pins, and timestamps exactly. Include a schema version field. _(seen in: Maccy, Pastebot, CopyClip)_
441. **Proprietary undocumented on-disk format** `[medium]` - When the history store is an opaque or undocumented schema, users and third-party tools cannot inspect, back up, repair, or migrate it; when the app breaks (corruption, abandonment) the data is unrecoverable. Ditto users resorted to external SQLite tools precisely because the format and maintenance were opaque. → **vbuff:** Use an open, documented storage format (plain SQLite with a published schema, or files on disk) and publish the schema in the repo. Keep large blobs as external files referenced by content hash so users can inspect and back them up with ordinary tools. Treat the format as a public contract with migration notes. _(seen in: Ditto, CopyQ)_
442. **No URL scheme / deep links** `[medium]` - Other apps, Shortcuts/Automator, browser bookmarklets, and documentation links cannot trigger vbuff actions (open picker, paste item N, run a snippet). Integrations that the platform makes trivial via custom URL schemes are simply unavailable, so vbuff cannot be wired into other tools' automation. → **vbuff:** Register a `vbuff://` URL scheme on each platform with a small, documented, allow-listed action set: `vbuff://open`, `vbuff://paste?id=...`, `vbuff://search?q=...`. Require the target to be local-only and validate/escape parameters. Document the scheme so other apps can call it.
443. **No local IPC socket / control API** `[high]` - Without a stable local control channel, the CLI, launchers, and scripts each have to reimplement how they talk to the running daemon, and there is no way to subscribe to events (new clip captured). Integrations become brittle reimplementations and there is no single source of truth for state. → **vbuff:** Expose a documented local-only IPC endpoint (Unix domain socket on macOS/Linux, named pipe on Windows) with a small request/response protocol and an event-subscription channel for 'clip added/removed/pinned'. The CLI is just a thin client of this socket. Protect it with filesystem permissions; never bind a TCP port by default.
444. **No launcher integration (Alfred/Raycast/Rofi/PowerToys Run)** `[medium]` - Users live in their launcher. If vbuff can't surface history as a launcher source or script filter, they must context-switch to vbuff's own window, losing the keep-focus, type-to-filter speed they expect. Maccy's popup even behaves unlike Spotlight, breaking that muscle memory. → **vbuff:** Ship first-class launcher adapters built on the CLI/IPC: an Alfred Workflow and Raycast extension on macOS, a rofi/fuzzel/wofi/dmenu recipe on Linux, and a PowerToys Run / Flow Launcher plugin on Windows. Each consumes `vbuff search --json` and invokes paste via IPC. Document and maintain them in-repo. _(seen in: Maccy, ClipMenu)_
445. **Brittle scripting via keystroke simulation** `[high]` - When the only way to script vbuff is to simulate global hotkeys and screen-scrape its UI, automations break whenever the window layout, focus, or hotkey changes, and they race against UI timing. This is the same fragility that makes auto-paste itself unreliable across apps and OS updates. → **vbuff:** Make every user-facing action reachable through the CLI/IPC API, not just the GUI. Document that scripting should target the API, not the UI. Never require AppleScript-UI-poking, key-injection, or fixed sleep delays as the supported automation path. _(seen in: ArsClip, Ditto)_
446. **No plugin / extension system** `[medium]` - Every transform a user wants (trim whitespace, format JSON, base64 decode, run a regex, OCR, translate) has to be begged for from the maintainer and shipped in core, bloating the app and leaving niche needs unmet. CopyQ's scriptable commands are popular precisely because static apps can't anticipate every workflow. → **vbuff:** Provide a documented plugin interface: at minimum 'command' plugins (external executables that receive a clip on stdin and return transformed bytes on stdout, declared in a manifest) for safe out-of-process extensibility, with an optional richer embedded scripting layer later. Run plugins out-of-process so a bad plugin can't crash the daemon. _(seen in: CopyQ)_
447. **Plugins run in-process with full trust** `[medium]` - A plugin/scripting system that executes third-party code in the daemon's address space means one buggy or malicious extension can crash the whole manager, leak the entire history, or hang capture. CopyQ's Triggers feature caused crashes severe enough that the vendor disabled it by default. → **vbuff:** Run extensions out-of-process with least privilege: pass only the specific clip(s) they need over a pipe, time-box and resource-cap execution, and surface failures without taking down the daemon. Require explicit user opt-in per plugin and show what data each plugin can see. _(seen in: ClipboardFusion, CopyQ)_
448. **No snippet / template engine for integration** `[low]` - Users want reusable snippets with placeholders (date, clipboard, cursor position, prompted variables) to integrate vbuff into text-expansion workflows. Without it they bolt on a separate text expander, and the two tools fight over the clipboard and hotkeys, corrupting each other's behavior. → **vbuff:** Support saved snippets with a documented, escapable placeholder syntax (e.g. {{date:FORMAT}}, {{clipboard}}, {{cursor}}, {{input:label}}) and expose snippet expansion through the CLI/IPC so other tools can invoke it. Make placeholder syntax literal-escapable so snippet content containing braces is safe. _(seen in: Clipy)_
449. **No documented exit codes / error contract for the CLI** `[medium]` - If the CLI returns 0 on failure, writes errors to stdout, or uses inconsistent codes, scripts cannot branch reliably ('did the search find anything?', 'did the daemon respond?'). Automation silently does the wrong thing and users distrust it. → **vbuff:** Define and document a stable exit-code contract (0 success, distinct non-zero codes for not-found, daemon-unreachable, permission-denied, bad-args). Write all diagnostics to stderr, data to stdout. Keep the contract part of the public versioned interface and don't change codes across minor versions.
450. **Ignoring platform clipboard primitives for interop** `[medium]` - If vbuff doesn't cooperate with pbcopy/pbpaste, xclip/xsel/wl-copy/wl-paste, and clip.exe, existing scripts that target those tools never reach vbuff's history, and users can't gradually adopt it. On Linux, ignoring the wl-clipboard/X11 ecosystem means existing pipelines bypass the manager entirely. → **vbuff:** Integrate with, rather than fight, the native primitives: capture whatever lands on the system clipboard (so pbcopy/wl-copy output is recorded) and document drop-in wrappers/aliases. On Linux, support both X11 and Wayland data-control protocols so existing tooling continues to work and feeds history. _(seen in: GPaste, Clipman, Diodon)_
451. **No headless / daemon mode for servers and automation** `[medium]` - On a remote box, CI runner, or tiling-WM setup with no tray, a GUI-coupled manager can't run at all, so automation that wants a clipboard history on headless or minimal environments is impossible. cliphist exists precisely because people need a no-GUI option. → **vbuff:** Make the daemon runnable headless with no GUI dependency, controllable entirely via CLI/IPC, and shippable as a systemd user unit / launchd agent / Windows service. The GUI is an optional separate front-end that attaches to the daemon, not a hard requirement. _(seen in: cliphist)_
452. **Schema/version drift with no compatibility guarantees** `[medium]` - If the storage schema, JSON output, or IPC protocol changes shape between releases without versioning, every script, launcher plugin, and importer silently breaks on update. This is the integration analog of Ditto's 'an update silently broke things' class of failures. → **vbuff:** Version every external contract: put a `schema_version`/`api_version` field in JSON output, IPC handshakes, and the export format. Maintain backward compatibility within a major version, document breaking changes, and let clients negotiate or detect the version before parsing. _(seen in: Ditto, Maccy)_
453. **No way to programmatically exclude apps / sources** `[medium]` - Integration with password managers, terminals, and banking apps requires that exclusion rules be set in config or via API, not buried in a GUI checkbox the user never finds. Without a scriptable/config-file exclusion mechanism, users can't provision policy across machines or via dotfiles. → **vbuff:** Drive source-exclusion from a human-editable, documented config file (and a CLI: `vbuff exclude add com.example.app`) in addition to the GUI, so rules are version-controllable and deployable. Honor the same rules whether set via GUI, CLI, or config. (Capture/secret-handling policy itself is a security-domain concern; this entry is only about the integration surface for setting it.) _(seen in: Ditto, Flycut)_
454. **No global-hotkey rebinding via config / API** `[medium]` - When the open-picker hotkey can only be set in the GUI and isn't exposed to config or the platform's own shortcut system, conflicts ('Failed to set hotkey' because another app grabbed it) can't be resolved programmatically, and Wayland users (where apps can't grab global hotkeys at all) have no path to bind it in the DE. → **vbuff:** Expose hotkey configuration in the documented config file and detect/report conflicts clearly. On Wayland, do not try to grab global hotkeys yourself: document and support DE-level binding that invokes `vbuff open` via the CLI/IPC, and ship example bindings for common compositors. _(seen in: ArsClip, Ditto, ClipboardFusion, CopyQ)_
455. **No content-type / MIME contract in the API** `[medium]` - If the API exposes clips only as 'text', integrations can't request the image bytes, the HTML representation, or a file list, and can't tell what a clip actually is. Tools that want the raw image/png (not the text/html offer) can't get it, mirroring the Linux 'image pasted as HTML' breakage at the integration layer. → **vbuff:** Model each clip as a set of typed representations (MIME type plus bytes) in the API and storage. Let callers enumerate available types (`vbuff get ID --types`) and request a specific one (`--type image/png`). Preserve all offered representations on capture so integrations can choose the right one. _(seen in: cliphist, clipman)_
456. **No event hooks / watch interface for automation** `[medium]` - Users want to run an action whenever something is copied (auto-format code, strip tracking params from URLs, log to a file). Without a watch/hook mechanism they must poll the history on a timer, which is laggy and wastes CPU, the same polling antipattern that makes some managers burn cores. → **vbuff:** Provide `vbuff watch` that streams new-clip events as JSON Lines over the IPC channel, and a config-declared hook mechanism that runs a user command on capture (out-of-process, with the clip on stdin). Event-driven, not polling. Let hooks optionally transform-and-replace the clip with explicit opt-in.
457. **Single-instance / second-manager hostility** `[medium]` - If vbuff assumes it is the only clipboard tool and fights any other manager for clipboard ownership, running it alongside an existing tool (during migration, or with a password manager's own clipboard handling) causes double-capture, wrong content, or dropped copies. Maccy + Pure Paste and Ditto + AutoHotkey show this corruption. → **vbuff:** Be a well-behaved clipboard citizen: observe changes passively rather than aggressively re-asserting ownership, debounce to avoid feedback loops, and detect when another manager is active and warn the user instead of silently corrupting state. Make capture purely observational by default. _(seen in: Maccy, Pure Paste, Ditto)_
458. **No stable item IDs for external reference** `[high]` - If scripts and launchers must refer to clips by volatile position index ('item 3'), any new copy shifts every reference, so automations paste the wrong thing. This is the integration root cause behind 'pasted into the wrong target / wrong item' for any index-based workflow. → **vbuff:** Assign each clip a stable, opaque, durable ID (e.g. content-hash or monotonic UUID) that survives reordering, dedup, and restarts, and accept that ID everywhere in the CLI/IPC/URL-scheme. Allow position-based access as a convenience but make ID-based the canonical, scriptable contract. _(seen in: ArsClip, Ditto)_
459. **No man page / shell completions / discoverable help** `[low]` - A CLI that ships without `--help` for every subcommand, a man page, or shell completions is effectively undocumented; users can't discover the integration surface and resort to guessing or giving up. Poor discoverability is why many managers' scripting features go unused. → **vbuff:** Generate per-subcommand `--help`, a man page, and shell completions (bash/zsh/fish/PowerShell) from the same source of truth as the argument parser, and ship them in the package. Include copy-pasteable examples for the common integrations (pipe, launcher, hook) in the help and README.
460. **No package-manager / standard distribution channel** `[low]` - If vbuff is only a hand-downloaded binary, it can't be installed or updated via Homebrew, winget/Scoop, or Linux package repos and AUR, so it can't be provisioned in dotfiles or fleet scripts, and users miss updates, the same staleness that leaves abandoned managers broken on new OS releases. → **vbuff:** Publish to the ecosystems users automate against: Homebrew (formula + cask), winget and Scoop on Windows, and Linux packages plus an AUR entry, in addition to GitHub release binaries with checksums. Keep a stable package name so scripts and CI can depend on it. _(seen in: Clipy, Flycut, 1Clipboard)_
461. **Config only editable through the GUI** `[medium]` - When all settings live behind GUI dialogs with no plain-text config file, users can't version-control, diff, or deploy their configuration across machines, and any preference reset (a known failure mode) silently wipes their setup with no committed copy to restore from. → **vbuff:** Back every setting with a documented, human-readable, hand-editable config file (TOML/YAML) in a standard location (XDG / Application Support / AppData). The GUI edits the same file. Support an env var or flag to point at an alternate config path for testing and provisioning. _(seen in: CopyQ, Flycut)_
462. **No automation-friendly clear/purge command** `[medium]` - Integrations and security-conscious users want to script 'clear history now' (e.g. before screen-sharing, or on lock) but if clear is GUI-only, it can't be wired to a hotkey daemon, a lock-screen hook, or a panic script, so the action that matters most in a hurry is the slowest to reach. → **vbuff:** Expose `vbuff clear` (all) and `vbuff delete ID`/`vbuff clear --older-than 1h` via CLI/IPC so they can be bound to system events (screen lock, session end) and panic scripts. Make clear synchronous and report what was removed, so callers know it completed before they proceed. (Whether clear also wipes the live system clipboard and on-disk remnants is a security-domain concern.) _(seen in: Klipper, Ditto)_
463. **Unstable / dependency-fragile scripting runtime** `[medium]` - If the only scripting path depends on a separately-installed interpreter, a specific shell, or DE-specific glue that breaks on upgrade (the GPaste GNOME-extension and 'run gpaste-client dr after upgrades' treadmill), users' integrations rot with every OS update and need manual repair. → **vbuff:** Keep the integration surface self-contained in the vbuff binary (CLI + IPC) with no required external interpreter for core scripting. Where DE/launcher glue is unavoidable, version it, test it against current OS releases in CI, and document a single recovery command rather than a manual ritual. _(seen in: GPaste)_

## 14. Settings & management mistakes

464. **No launch-at-login, or it silently unchecks itself after OS updates** `[high]` - An always-on utility that doesn't auto-start records nothing until the user manually relaunches it after every reboot; worse, when the login-item flag silently clears after an OS upgrade the user keeps working for days believing capture is running while their history quietly stops growing, and only notices when a clip they expected is missing. → **vbuff:** Ship launch-at-login ON by default via the OS-native mechanism (SMAppService/launchd on macOS, registered Startup app on Windows, XDG autostart .desktop on Linux). Re-verify the registration on every app start, repair it if missing, and surface a persistent in-app indicator showing 'starts at login: yes/no' plus the last successful registration check so silent breakage is visible. _(seen in: Flycut)_
465. **Settings/history don't persist across restart unless manually enabled** `[high]` - The user copies all day, reboots, and loses everything because persistence was an off-by-default toggle buried in preferences they never opened. They assumed a 'clipboard manager' kept history by definition. → **vbuff:** Persist history and preferences to disk by default with an explicit, atomic write on every change (write-temp-then-rename). Make 'keep history across restarts' the default state, and if the user opts into volatile/RAM-only mode, label it clearly and confirm the trade-off. _(seen in: Flycut, Windows 11 Win+V)_
466. **No per-app behavior rules (ignore-list, plain-text-only, capture mode)** `[high]` - A single global policy can't express 'never record from my password manager or banking app, paste plain text into Slack, but keep formatting from my editor'. Without per-app rules the user is forced into all-or-nothing and either leaks secrets or loses convenience. → **vbuff:** Provide a per-application rules table keyed by app bundle id / executable path / Wayland app_id, with actions: never-capture, capture-but-mark-sensitive, force-plain-text, and custom hotkey. Ship sane built-in rules for common password managers and banking apps, and make the matching key visible so users can verify which rule fired. _(seen in: Ditto, Flycut)_
467. **Bad default history limit: too low silently drops, or unlimited bloats** `[high]` - A hardcoded low cap (e.g. 25 items) silently deletes clips the user expected to still be there, while an unbounded default lets the store grow to gigabytes and rots performance. Either default surprises the user because the value isn't shown or explained. → **vbuff:** Default to a sensible bounded limit (e.g. a few thousand items AND a total-bytes budget, whichever hits first), expose both limits in settings with the current usage shown live, and warn before eviction policy changes take effect. Never hardcode a silent cap. _(seen in: Maccy, Windows 11 Win+V)_
468. **Hidden critical options buried behind advanced panels or only in config files** `[high]` - Settings that materially affect safety or correctness (ignore-list, encryption toggle, auto-clear timer, plain-text default) are hidden so deeply, or only editable in a hand-written config file, that ordinary users never discover them and run with unsafe defaults. → **vbuff:** Surface safety-critical settings on the first/top settings page, not behind an 'Advanced' wall. Anything important enough to affect data exposure must be reachable in the GUI with a plain-language description; config-file-only options are acceptable only for genuinely niche tuning and must be documented. _(seen in: Ditto, Flycut)_
469. **No 'reset to defaults' / no recovery from a corrupted config** `[high]` - When a user breaks their settings or a config file gets corrupted, there is no in-app way to restore known-good defaults, so the app may refuse to launch or behave erratically and the only fix is deleting opaque files in hidden directories the user can't find. → **vbuff:** Provide an explicit 'Reset all settings to defaults' button (with confirmation and an automatic backup of the prior config). On startup, validate the config against a schema; if invalid, fall back to defaults, keep the broken file as .bak, and tell the user what happened instead of crashing. _(seen in: CopyQ)_
470. **Settings silently reset or corrupted by app updates / rename operations** `[high]` - After an update or an innocuous action (e.g. renaming a tab/profile) the user's customization vanishes with no warning, erasing carefully tuned hotkeys, ignore rules, and limits. Trust in the app evaporates. → **vbuff:** Version the settings schema and write explicit, tested migrations for every release. Never destructively rewrite settings on rename; treat config as append-or-migrate, back up the previous version before any migration, and show a one-time 'your settings were migrated' note with a rollback option. _(seen in: CopyQ)_
471. **Settings not portable: no export/import of configuration** `[medium]` - A user on a new machine, or maintaining several machines, must re-create every preference, hotkey, ignore rule, and profile by hand because there's no way to export and re-import configuration. Setups drift between machines. → **vbuff:** Support exporting all settings (excluding clipboard contents and secrets) to a single human-readable file (e.g. TOML/JSON) and importing it on another machine. Keep the on-disk config in a documented, plain, diffable format so it can be version-controlled and synced by the user's own tools.
472. **Config tied to OS-specific opaque store, breaking cross-platform parity** `[medium]` - Storing settings in platform-specific opaque blobs (macOS UserDefaults plist, Windows registry, GSettings dconf) means a cross-platform app behaves differently per OS, can't share a config, and the user can't inspect or back up their own preferences. → **vbuff:** Keep a single canonical, platform-neutral config file (TOML) in an XDG-style location per platform, with identical keys and semantics everywhere. Do not scatter settings across registry/plist/dconf; if OS integration requires a registration (login item), derive it from the canonical config rather than storing the source of truth there.
473. **No profiles / context switching (work vs personal vs presenting)** `[medium]` - Users want distinct setups: an isolated work history, a 'presenting' mode that pauses capture and hides history during screen-share, and a personal context. With one global profile they manually toggle settings constantly or accidentally expose private clips on a shared screen. → **vbuff:** Support named profiles, each with its own history store, ignore-list, and limits, plus a quick switcher and a one-key 'pause capture / privacy mode' for screen-sharing. Make the active profile and capture state visible in the tray/menubar at all times.
474. **No pause/snooze control for capture** `[medium]` - Before pasting passwords, doing sensitive work, or sharing a screen, users have no quick way to temporarily stop recording. Quitting the app entirely is the only option, which they forget to undo, losing capture later. → **vbuff:** Offer an explicit pause toggle (with optional auto-resume after N minutes) accessible from the tray menu and a global hotkey, with an unmistakable visual state change while paused so the user can't forget capture is off.
475. **Encryption / auto-clear of sensitive items not exposed as a setting (off by default)** `[high]` - The single most safety-relevant management options (encrypt the store at rest, auto-expire sensitive clips after N minutes) are either absent or default-off, so the user runs unprotected without knowing the option even existed. → **vbuff:** Expose 'encrypt history at rest' and 'auto-clear sensitive clips after N minutes' as prominent settings, default encryption ON, and make the sensitivity/auto-clear behavior configurable per content-type. Treating these as first-class settings is the management responsibility; the cryptographic implementation is out of this domain's scope. _(seen in: Maccy, Ditto, CopyQ)_
476. **No de-duplication option / dedup behavior not configurable** `[low]` - Copying the same text repeatedly creates dozens of identical rows, cluttering history and pushing out useful older items; there's no setting to either dedup-and-bump or keep all, so the user can't choose the behavior they want. → **vbuff:** Default to de-duplicating identical clips by bumping the existing entry to the top, and expose a setting to switch between dedup-bump, dedup-keep-position, and keep-all. Make the chosen policy explicit so behavior is predictable. _(seen in: Maccy)_
477. **Global hotkeys default to OS-reserved combos and can't be remapped per action** `[medium]` - Default shortcuts collide with system actions (e.g. macOS Cmd+Shift+3/4/5 screenshots, Cmd-X) so the user triggers a screenshot or cut instead of opening history, and there's no per-action remapping to escape the clash. → **vbuff:** Choose defaults that avoid known OS-reserved combos per platform, validate the chosen hotkey against a built-in list of system shortcuts and warn on conflict, and let every action (open picker, paste plain, pause, switch profile) be independently rebound. Detect already-taken hotkeys at registration and surface a clear conflict message rather than failing silently. _(seen in: Maccy, CopyClip 2 (FIPLAB), CopyQ)_
478. **'Failed to set hotkey' on startup with no resolution path** `[medium]` - Another app grabbed the global shortcut, so the manager throws a 'failed to set hotkey' error on every launch (or the hotkey just stops firing) and the user has no guidance on which app conflicts or how to pick a working binding. → **vbuff:** On registration failure, report which combo failed and (where the OS allows) which process holds it, offer a one-click 'pick a different shortcut' flow, retry registration when focus returns, and persist a clear 'hotkey not active' indicator until resolved. _(seen in: ArsClip, Ditto, ClipboardFusion)_
479. **No first-run setup / onboarding for required permissions and defaults** `[medium]` - The app launches into a blank tray icon with no guidance, so the user never grants required permissions, never sets an ignore-list, and never learns the hotkey. Critical safety choices are skipped by default. → **vbuff:** Provide a short first-run wizard that requests required OS permissions, proposes a default ignore-list (password managers, banking apps), confirms the open-history hotkey with a live conflict check, and explains where history is stored and how to clear it.
480. **No visible permission/health status in settings; silent permission drops invisible** `[high]` - When an OS permission (Accessibility on macOS, clipboard access under Wayland) is revoked after an update, the settings UI shows nothing wrong, so the user can't tell why the app stopped working and burns time re-toggling random options. → **vbuff:** Add a 'Status / Diagnostics' settings page that live-checks every dependency (permissions granted, login item registered, hotkey active, store writable, compositor protocol available) with green/red indicators and a one-click 'fix' or deep-link to the relevant OS settings pane. _(seen in: Maccy, Flycut, Clipy)_
481. **Telemetry/analytics on by default with no opt-out in settings** `[medium]` - A tool sitting over the most sensitive data path on the machine phones home by default, and the user finds no setting to disable it (or it's opt-out, buried). Even 'anonymous' telemetry is unacceptable to many for this class of app. → **vbuff:** Default telemetry OFF and opt-in only, with a clearly labeled toggle on the main settings page and a precise description of exactly what (if anything) would be sent. For a privacy-sensitive utility, prefer no network calls at all unless the user explicitly enables a feature that needs them. _(seen in: ClipboardFusion, CopyClip)_
482. **In-app ads / cross-promotion of the developer's other apps in settings** `[low]` - Promotional banners and 'try our other apps' upsells inside a utility holding sensitive clipboard data feel cheap and untrustworthy, and clutter the very settings UI the user opened to configure safety options. → **vbuff:** Ship zero ads and zero cross-promotion in the app or its settings. If the project needs funding, use a clearly-separated optional donate/sponsor link, never injected promos in the management UI. _(seen in: CopyClip)_
483. **History storage location is fixed/hidden and not configurable** `[medium]` - The user can't see or choose where history is stored, can't move it off a synced/cloud folder, and can't point it at an encrypted volume. They may unknowingly have their clipboard DB inside Dropbox/iCloud, expanding exposure and causing DB lock/corruption. → **vbuff:** Show the exact storage path in settings, allow relocating it to a user-chosen directory (with migration), and warn if the chosen path is inside a known cloud-sync folder. Default to a local, app-private, non-synced location. _(seen in: Ditto)_
484. **No clear/wipe controls (no 'clear all', no 'delete this app's clips', no clear-on-quit)** `[high]` - Users have no easy way to purge history when they need to (before handing over a machine, after copying a secret), or to scope a clear to one app's clips, so sensitive data lingers and the only recourse is hunting for hidden files. → **vbuff:** Provide prominent 'Clear all history', 'Clear last item', and 'Clear all clips from app X' actions, plus optional 'clear history on quit' and 'clear on lock/sleep' settings. Confirm destructive actions and report what was removed. (Whether the underlying bytes are truly purged is a security-domain concern; this is about the management controls existing and being reachable.) _(seen in: Ditto, Maccy)_
485. **Leftover config/data on uninstall, with no documented removal** `[medium]` - After uninstalling, config directories and history stores remain in hidden locations the user can't find, so 'removing' the app leaves a sensitive history behind and re-installing resurrects stale, possibly incompatible settings. → **vbuff:** Document every file/directory the app creates, keep them in standard discoverable locations, and offer a 'remove all vbuff data' option (in-app and as an uninstall helper) that deletes config and history. Don't reuse stale config silently on reinstall; offer to start fresh or import. _(seen in: Ditto)_
486. **Confusing, overstuffed, or mislabeled preferences UI** `[medium]` - Dozens of cryptic toggles with no descriptions, ambiguous labels, and no search make it impossible for users to find or understand the setting they need, so they leave unsafe defaults in place or change the wrong thing. → **vbuff:** Group settings into a few clear sections (General, Capture, Privacy, Shortcuts, Storage, Status), give every toggle a one-line plain-language explanation and its default value, add a settings search box, and avoid jargon. Each setting should state its consequence, not just its name. _(seen in: Paste, CopyQ)_
487. **No system-managed/enterprise config (machine-wide policy lock)** `[low]` - In managed environments, admins can't enforce safe defaults (mandatory ignore-list, disabled cloud sync, forced encryption) because there's no machine-wide policy layer, so every user can disable protections and the org can't standardize behavior. → **vbuff:** Support an optional admin-supplied policy file in a system location that sets and optionally locks specific settings, layered under the user config (system policy > user config > built-in defaults). Clearly indicate in the UI which settings are administrator-locked.
488. **Settings changes require restart or don't apply until relaunch** `[medium]` - The user toggles an ignore rule or lowers the history limit expecting it to take effect immediately, but the app keeps capturing the excluded app or keeping extra items until a manual restart, so the protection they just enabled isn't actually active. → **vbuff:** Apply all settings live without restart: re-evaluate capture rules, limits, and hotkey bindings on save, and reflect the new state immediately. If any setting genuinely needs a restart, mark it explicitly and offer an in-app 'apply and restart' button.
489. **No default ignore-list for known sensitive sources; capture-everything by default** `[high]` - Out of the box the manager records from password managers, banking apps, terminals, and 2FA prompts because there's no shipped exclusion list, so secrets land in history before the user even learns the ignore feature exists. → **vbuff:** Ship a curated default ignore-list covering common password managers and sensitive apps (overridable by the user), honor OS concealed/transient pasteboard hints as an additional default, and make the active ignore-list visible and editable on the main Privacy page. _(seen in: Ditto, Flycut)_
490. **No per-content-type policy (text vs image vs files) in settings** `[medium]` - Users want different handling per type, e.g. don't store images at all, cap image size, never keep file paths, but a single global switch forces one policy on everything, so they either bloat the store with images or lose text history to disable images. → **vbuff:** Expose per-content-type settings: enable/disable capture per type (plain text, rich text, image, files), a per-type size cap, and a per-type retention limit. Show current per-type usage so the user can tune deliberately.
491. **Cloud/sync of settings or history enabled by default without explicit consent** `[high]` - An easily-missed default uploads configuration or clipboard contents to a vendor or platform cloud, expanding exposure off-device and surprising users who assumed a local-only tool. Consent is opt-out instead of opt-in. → **vbuff:** Keep all sync OFF until the user explicitly enables it, with a distinct opt-in for syncing settings vs syncing clipboard contents, a clear statement of where data goes, and a permanent local-only mode. Never flip sync on via an update. (Sync transport/encryption correctness belongs to the sync domain; this is about the default and the consent gate.) _(seen in: Windows 11 Win+V, 1Clipboard, ClipboardFusion)_
492. **Pinned/favorite item configuration is fragile and order is non-deterministic** `[medium]` - Pins are part of how users curate their history, but they disappear after updates/reboots, can't be reordered, and silently change order on restart, so the user's deliberately managed set of kept items becomes unreliable. → **vbuff:** Store pinned items separately from the rolling history with explicit, persisted ordering the user controls (drag to reorder), exclude pins from eviction by default, and round-trip pins through settings export/import so curation survives migrations and reinstalls. _(seen in: Maccy, CopyQ)_
493. **No way to disable update checks / no control over update behavior for a sensitive tool** `[low]` - An app over the most sensitive data path either auto-updates with no user control, or phones an update server the user can't disable, and there's no setting to choose manual updates, undermining trust and adding an unwanted network dependency. → **vbuff:** Provide explicit update settings: auto / notify-only / manual, with a visible last-checked time and changelog, and the ability to fully disable network update checks. Default to notify-only so the user stays informed without surprise changes to a privacy-sensitive tool.
494. **Defaults optimized for demo flash, not safe/quiet operation** `[medium]` - Out of the box the app enables OCR-on-every-image, aggressive previews, sounds, and notifications to look feature-rich, turning copied IDs and receipts into searchable plaintext and spamming the user, who must hunt to turn it all off. → **vbuff:** Default to quiet, minimal, privacy-preserving behavior: heavy features (OCR, content scanning, network calls, notifications) OFF by default and opt-in, with each clearly explaining its data/cost implications before the user enables it. _(seen in: KeepClips, Clipory, ClipZ)_

## 15. Platform-specific mistakes

495. **No wlr-data-control / ext-data-control support on Wayland** `[critical]` - On wlroots/Sway/Hyprland the only sanctioned way to monitor the clipboard is the wlr-data-control (now ext-data-control) protocol. A manager that polls or relies on focus instead captures nothing in the background, so history stays empty and the app looks broken even though it is running. → **vbuff:** Implement an ext-data-control-v1 (with wlr-data-control-unstable-v1 fallback) client that registers a data-control device and reads every new selection offer without needing focus. Detect protocol availability at startup and surface a clear capability banner if absent. _(seen in: cliphist, clipman, wl-clipboard, wl-clip-persist, CopyQ)_
496. **Treating GNOME Mutter Wayland as if data-control exists** `[critical]` - Mutter deliberately refuses to implement wlr/ext-data-control on security grounds. A manager that assumes the protocol is present silently records nothing on the single most common Linux desktop, and users blame the app rather than the compositor. → **vbuff:** Probe for the protocol; if running under GNOME Wayland, fall back to a documented path (a GNOME Shell extension talking to the app over D-Bus, or focus-based capture via the portal) and tell the user explicitly which mode is active and its limits, instead of pretending to work. _(seen in: cliphist, clipman, wl-clip-persist, GNOME Mutter)_
497. **Clipboard data dies when source app closes (no persistence helper)** `[critical]` - On X11 the server only holds a reference; on Wayland the data offer is destroyed with the window. If the manager does not eagerly read and own the bytes at copy time, closing the source app (or terminal) loses the clip. This is the most-reported decades-old Linux clipboard frustration. → **vbuff:** On every selection change, immediately read ALL offered MIME types into the manager's own storage and take clipboard ownership (re-serve the data) so the clip survives the source app exiting. Do not lazily fetch at paste time. _(seen in: X11 core, Wayland core, GPaste, Klipper, Diodon, Clipman)_
498. **Confusing PRIMARY selection with CLIPBOARD on X11/Wayland** `[high]` - Linux has two independent buffers: PRIMARY (select-to-copy, middle-click paste) and CLIPBOARD (Ctrl+C). Mixing them means Ctrl+C lands only in the middle-click buffer, or a paste retrieves the wrong selection, corrupting the user's workflow. → **vbuff:** Track PRIMARY and CLIPBOARD as separate sources with explicit per-source settings. Default to capturing CLIPBOARD; make PRIMARY capture opt-in. Never write a CLIPBOARD copy back into PRIMARY (or vice versa) unless the user asked. _(seen in: X11 core, GPaste, Klipper, CopyQ)_
499. **Breaking middle-click PRIMARY paste / ignoring its conventions** `[medium]` - Middle-click PRIMARY paste is deep muscle memory on Linux. Managers that hijack PRIMARY, clear it, or constantly overwrite it break a workflow users rely on dozens of times an hour. GNOME's default for middle-click-paste also flipped, so assumptions about the default are unsafe. → **vbuff:** Leave PRIMARY behavior to the OS by default; only touch PRIMARY when the user opts in. Read the desktop's middle-click-paste gsetting rather than assuming a value, and never clear PRIMARY as a side effect of capturing CLIPBOARD. _(seen in: GNOME, GTK apps, Fedora 44)_
500. **X11/Wayland/XWayland clipboard split not bridged** `[high]` - Copies do not automatically propagate between Wayland-native apps and XWayland apps. A manager running in only one world cannot see the other's clipboard, so content copied in a GTK4 app is unpastable in an X11 app and history is half-empty depending on toolkit. → **vbuff:** Run capture in both worlds: a Wayland data-control client AND an X11/XWayland selection owner, then mirror clips between them (respecting an anti-loop guard). Document that bridging is active so users understand cross-toolkit behavior. _(seen in: Diodon, CopyQ, XWayland bridge)_
501. **Auto-paste keystroke injection on Wayland (impossible without a backend)** `[high]` - Wayland forbids synthetic global input, so a Ctrl+V injection approach simply does nothing. Users pick a clip and nothing is typed, defeating the picker. CopyQ auto-paste does not work on Wayland at all for this reason. → **vbuff:** Use the supported path per compositor: wlr/ext virtual-keyboard or the input-method protocol where available, ydotool/uinput as an opt-in fallback, otherwise set the clipboard and tell the user to press Ctrl+V. Never silently no-op; show the chosen paste strategy. _(seen in: CopyQ, Diodon)_
502. **Global hotkey grabbed by the app instead of via the DE/portal on Wayland** `[high]` - Wayland apps cannot grab global hotkeys directly. A manager that tries silently fails to open via its shortcut, and many shortcut-driven workflows break with no error. → **vbuff:** Register global shortcuts through the org.freedesktop.portal.GlobalShortcuts portal where supported, and otherwise provide clear instructions plus a ready-made command for the user to bind in their DE settings. Detect failure and show it rather than appearing dead. _(seen in: CopyQ, Diodon, GPaste)_
503. **Picker steals focus and cannot refocus the real target (Wayland)** `[high]` - Showing the picker via shortcut takes focus from the previously active window; on Wayland the manager cannot reliably restore it, so paste goes to the wrong window or nowhere, and arrow-key navigation may not even work in the popup. → **vbuff:** Use a layer-shell overlay (wlr-layer-shell) that takes keyboard focus for the picker but records the previously focused surface, then on selection set the clipboard and trigger paste into the remembered target via the virtual-keyboard/input-method path. Test ESC and arrow keys explicitly under Wayland. _(seen in: CopyQ, cliphist, Diodon)_
504. **Invoking the picker crashes the receiving app or session on Wayland** `[critical]` - Buggy focus-protocol / IPC handling can crash the target application or the whole session when the paste window is invoked, which is catastrophic for a tool meant to be invisible and always-on. → **vbuff:** Isolate compositor IPC in a supervised subprocess with timeouts and graceful degradation; never block the compositor's event loop. Fuzz-test focus and layer-shell transitions and fail soft (set clipboard, no crash) on protocol errors. _(seen in: CopyQ)_
505. **5-second Wayland clipboard activation timeouts / null data** `[high]` - Activating the Wayland clipboard synchronously can hang for seconds ('Activating Wayland clipboard took 5000 ms', 'Null data in clipboard'), freezing capture and producing empty or missing items, while the same machine works on Xorg. → **vbuff:** Do all clipboard offer reads asynchronously off the UI thread with bounded timeouts and retries; handle the empty-offer race (data not yet ready) by re-requesting on the next frame rather than recording a null item. _(seen in: CopyQ)_
506. **Only saving the last item during rapid successive copies (Wayland offer race)** `[high]` - Because Wayland offers are transient and the manager only reads on focus or lazily, copying several things in a row saves just the last one, so users lose intermediate clips and must re-copy. → **vbuff:** Subscribe to every selection event via data-control and synchronously snapshot each offer's bytes before the next offer arrives; queue reads so no selection change is dropped under bursty copying. _(seen in: CopyQ, cliphist, GPaste, clipcat)_
507. **wl-clip-persist-style ownership fights producing empty pastes** `[medium]` - Re-asserting clipboard ownership to persist data can fight with apps (Firefox on Wayland, sync tools), causing intermittent empty pastes; users disable persistence to regain reliability, trading the feature away. → **vbuff:** Re-serve clips with correct seat/serial handling and only re-assert ownership when no other client currently owns the selection; add an anti-feedback guard and back off when another app claims ownership instead of fighting in a loop. _(seen in: wl-clip-persist, Firefox (Wayland), Hyprland/Omarchy setups)_
508. **Requiring users to stack a compositor + manager + persistence daemon** `[medium]` - On Sway/wlroots the standard advice is to chain yet another daemon (e.g. wl-clip-persist) plus a separate image watcher just to get reliable history; any link breaking breaks the clipboard, and the burden falls on the user. → **vbuff:** Ship one binary that handles capture, persistence, and image MIME watching together (text and image watchers built in), so users do not have to assemble and maintain a fragile daemon stack. _(seen in: Sway, wlroots compositors, clipman, wl-clip-persist)_
509. **Needing a separate watcher to keep images on Wayland** `[medium]` - Tools like cliphist need a distinct 'wl-paste --type image --watch' invocation to capture images; a single text watcher silently drops image/png offers, so pasting into an image editor fails. → **vbuff:** On each selection, enumerate ALL advertised MIME types and store the richest set (image/png, image/svg+xml, text/html, text/plain together) in one capture path; never hardcode a single watched type. _(seen in: cliphist, clipman, wl-clipboard sync tools)_
510. **Storing browser image copies as text/html instead of pixels** `[medium]` - Browsers offer both text/html and image/png for a copied image; grabbing the html offer means pasting into an image editor yields markup, not an image, so the copy is effectively useless for its intended target. → **vbuff:** When multiple representations exist, prefer the one matching the user's paste intent and keep image/png as the canonical image payload; let paste choose the representation rather than committing to text/html at capture time. _(seen in: cliphist, clipman, wl-clipboard sync tools)_
511. **macOS Accessibility permission silently revoked after updates** `[high]` - Auto-paste needs Accessibility permission, but after app updates, code-signing/notarization changes, or OS upgrades the TCC grant silently drops (or the DB is corrupted), so paste stops with no in-app indication of why; users remove and re-add the app repeatedly. → **vbuff:** Check AXIsProcessTrusted() at launch and before every auto-paste; if not trusted, show an actionable banner with a deep link to the exact Settings pane and fall back to clipboard-set + manual Cmd+V instead of failing silently. Keep a stable bundle id / Developer ID so the grant persists across updates. _(seen in: Maccy, Paste, Pastebot, Flycut, Clipy)_
512. **Auto-paste produces a notification instead of pasting (macOS)** `[critical]` - On recent macOS, broken keystroke synthesis surfaces the clip as a notification banner containing the text rather than inserting it into the focused field, so the core action visibly fails after an OS update. → **vbuff:** Drive paste via CGEvent Cmd+V into the frontmost app's focused element after verifying Accessibility trust; detect when synthesis is blocked and degrade to leaving the item on the clipboard with a clear hint, never to a notification masquerading as a paste. _(seen in: Maccy, Flycut, Clipy)_
513. **Picker steals focus on macOS instead of behaving like Spotlight** `[high]` - The popup grabs focus from the active app, breaking the paste target and forcing extra clicks; users expect Spotlight-like behavior where the previous app stays the paste destination. → **vbuff:** Show the picker as a non-activating panel (NSPanel with .nonactivatingPanel, canBecomeKey for typing but not stealing app activation), remember the previously frontmost app, and re-activate it before issuing the paste keystroke. _(seen in: Maccy, ClipMenu)_
514. **Default hotkeys colliding with macOS screenshot shortcuts** `[medium]` - Defaults near Cmd+Shift+3/4/5 collide with built-in macOS screenshot capture, so summoning a clip triggers a screenshot instead, surprising the user and littering the desktop with screenshots. → **vbuff:** Ship defaults that avoid reserved system combinations, detect conflicts with registered system shortcuts at first run, and warn plus offer a safe rebind if the chosen hotkey is already claimed by the OS. _(seen in: Maccy)_
515. **Pasting into Spotlight / Electron / secure fields silently fails (macOS)** `[medium]` - Synthetic Cmd+V does not reach Spotlight, some Electron apps, or secure/password fields, so the chosen item is never inserted and the user cannot tell why. → **vbuff:** Detect the focused app/field class; for secure or non-cooperating targets, set the clipboard and prompt manual paste rather than silently injecting into the void. Maintain a per-app quirks table for known-problem targets. _(seen in: Maccy)_
516. **Ignoring macOS concealed/transient pasteboard hints** `[high]` - Password managers set org.nspasteboard.ConcealedType / TransientType (and NSPasteboardTypeAutoGenerated). Managers that ignore these capture credentials and one-time fields into persistent history that the OS asked them not to store. → **vbuff:** Before storing, inspect the NSPasteboard for concealed/transient/auto-generated markers and the legacy org.nspasteboard.ConcealedType, and skip or auto-expire those items by default. Make the exclusion the default, not an opt-in. _(seen in: Maccy, Paste, Raycast/Alfred, CopyClip, Flycut, Jumpcut)_
517. **Polling NSPasteboard changeCount on a tight timer** `[medium]` - macOS has no clipboard-change notification, so naive managers spin a tight timer reading changeCount, wasting CPU/battery and still missing rapid bursts between ticks. → **vbuff:** Poll changeCount on an efficient interval coalesced with run-loop idle, throttle when the app is backgrounded or on battery, and snapshot immediately on each detected change so rapid successive copies are not missed. _(seen in: CopyClip, Maccy)_
518. **Not honoring x-kde-passwordManagerHint / KDE secret hints on Linux** `[high]` - KeePassXC and KDE set x-kde-passwordManagerHint=secret so managers skip the clip. Managers that ignore it store passwords in searchable history; KeePassXC's auto-clear then wipes the live clipboard but the secret remains in history. → **vbuff:** Honor x-kde-passwordManagerHint (and equivalent secret MIME hints) by never persisting those clips, and when a source clears the live clipboard, propagate the clear to any matching history entry. _(seen in: Clipman, GPaste, Klipper, CopyQ, KeePassXC)_
519. **Clearing history does not clear the live clipboard (and vice versa)** `[high]` - On Klipper/Plasma, clearing history leaves the live clipboard still pasteable (or clearing the clipboard leaves history intact), so a user who 'purged' a secret can still paste it - a false sense of security tied to platform clipboard ownership. → **vbuff:** Make a single clear action atomically wipe both the history store and the OS-owned live selection (take ownership and serve empty / call the platform clear), and verify on each platform that both are actually empty afterward. _(seen in: Klipper, Plasma clipboard)_
520. **No native OS keychain / secret-store integration for the DB key** `[high]` - Encryption keys or sync tokens stored in a plaintext config file or hardcoded defeat the purpose; the platform already offers a vetted secret store and not using it leaves keys readable by any local process. → **vbuff:** Store any encryption key / sync credential in the OS secret store: macOS Keychain, Windows DPAPI / Credential Manager, and the freedesktop Secret Service (libsecret / KWallet / GNOME Keyring) on Linux. Never write secrets to the app's own config or DB.
521. **Win+V native history limits leak into a third-party app's expectations** `[high]` - The Windows native history caps at 25 items, 4MB/item, drops oldest silently, wipes on reboot unless pinned, and only syncs plain text. A manager that mirrors or relies on these limits inherits silent data loss users blame on it. → **vbuff:** Run an independent capture pipeline rather than wrapping Win+V; set generous configurable caps, persist across reboot by default, and if integrating with Win+V at all, document that the OS feature's limits are separate from vbuff's storage. _(seen in: Windows 11 Win+V)_
522. **Win+V misses delayed-rendering / complex-format sources (Excel)** `[high]` - Apps like Excel use delayed rendering and offer many formats; the OS history and naive managers miss these on fast copies or fail to materialize the rich data, so a copied range never lands in history. → **vbuff:** On Windows, request the formats lazily but render delayed-render formats by calling the source before it exits (handle WM_RENDERALLFORMATS), and enumerate the full format list rather than grabbing only CF_UNICODETEXT. _(seen in: Windows 11 Win+V)_
523. **Injecting a literal 'v' instead of pasting (Windows keystroke synthesis bug)** `[high]` - A broken simulated Ctrl+V can send the 'v' character half without the Ctrl modifier, so instead of the chosen clip the app types a literal 'v' into the field, silently corrupting input. → **vbuff:** Synthesize paste with correct SendInput key-down/up ordering (Ctrl down, V down, V up, Ctrl up) and verify modifier state; prefer setting the clipboard and letting the user paste when the target rejects synthetic modifiers. _(seen in: Ditto)_
524. **Picker opens behind other windows / off the focused monitor (Windows)** `[high]` - The picker summoned by hotkey opens behind the active window or loses focus, so the user gets a panel they cannot type into or that hides the target app, breaking the pick-and-paste flow. → **vbuff:** Create the picker as a topmost tool window, foreground it with the proper SetForegroundWindow/AttachThreadInput dance, and position it on the monitor and near the caret of the previously focused window, restoring that window as paste target. _(seen in: Ditto)_
525. **'Failed to set hotkey' when another app owns the global shortcut** `[medium]` - RegisterHotKey fails if another app already grabbed the combo, and naive managers either crash, show the error every launch, or silently have a dead shortcut, leaving the picker unreachable. → **vbuff:** On registration failure, fall back gracefully (offer an alternative default, allow rebinding inline), report which combo failed, and re-try registration on focus/session changes rather than only at startup. _(seen in: ArsClip, Ditto, ClipboardFusion)_
526. **Paste overwrites the wrong selection / inserts at wrong control (Windows)** `[medium]` - Fragile focus detection makes paste replace an entire filename or selection instead of inserting at the caret, or land in the wrong control, because the tool cannot reliably identify the focused field. → **vbuff:** Resolve the focused element via UI Automation (GetFocusedElement) before pasting, insert at the caret rather than replacing selection unless intended, and keep a per-app behavior table for known-fragile targets (browsers, specific editors). _(seen in: ArsClip)_
527. **Capture dies when another process owns the clipboard chain (RDP/VM/AHK)** `[high]` - RDP, VMware/VirtualBox, AutoHotkey, or a second manager fight over the clipboard chain; the tray app stays alive but silently records nothing, so users think it works while history is empty. → **vbuff:** Use AddClipboardFormatListener (not the legacy SetClipboardViewer chain) so co-existing listeners do not break each other, detect prolonged capture gaps, and surface a 'capture stalled' indicator instead of failing silently. _(seen in: Ditto, CopyQ)_
528. **Default-on cloud clipboard sync uploading secrets off-device** `[high]` - Windows Cloud Clipboard (Microsoft account) and apps syncing via Google Drive upload copied passwords and card numbers off-device by a default or easily-missed toggle, expanding exposure beyond the local machine - a platform convention users often do not realize is on. → **vbuff:** Keep all sync opt-in and off by default, never route through a vendor cloud or cloud-synced folder implicitly, and clearly label what leaves the device; respect the OS 'do not roam' clipboard flag where the platform sets it. _(seen in: Windows 11 Win+V, 1Clipboard, ClipboardFusion)_
529. **Rich-text/HTML/RTF formats dropped or mangled on platform round-trip** `[medium]` - Storing only text/plain (or only CF_UNICODETEXT) means a clip captured with formatting pastes as bare text or broken styling, and file URLs / image references get stripped, because the manager did not preserve the platform's full format set. → **vbuff:** Capture and re-offer the complete set of representations each platform provides (CF_HTML, RTF, file descriptors on Windows; public.html, public.rtf, file URLs on macOS; text/html, image/png on Linux) and let paste pick the right one, with an explicit 'paste as plain text' that is easy to reach. _(seen in: ClipAngel, Comfort Clipboard, Windows 11 Win+V, Maccy, Paste)_
530. **Retina/HiDPI screenshots stored as raw TIFF (platform pixel-format ignorance)** `[medium]` - macOS puts huge Retina TIFFs on the pasteboard; storing the raw representation means a single screenshot is 10-30 MB and history bloats to gigabytes, a platform-specific pixel-format trap distinct from generic image handling. → **vbuff:** Normalize pasteboard images to an efficient encoding (PNG) for storage while retaining the original representation only when needed for fidelity, and account for backing-scale so HiDPI captures are stored once at proper resolution, not re-expanded. _(seen in: Ditto, Maccy)_
531. **Picker positioned wrong on multi-monitor / mixed-DPI setups** `[medium]` - The picker appears on the wrong monitor, at the wrong scale, or offscreen when monitors have different DPI, forcing the user to hunt for it and breaking the fast summon-and-pick flow; CopyQ even shows huge memory growth tied to a second monitor. → **vbuff:** Be per-monitor-DPI aware (Windows PerMonitorV2 manifest, macOS backing-scale, Wayland fractional-scale / wl_output scale), and position the picker on the monitor containing the focused window's caret, clamped to that monitor's work area. _(seen in: CopyQ)_
532. **Blurry UI from not declaring HiDPI / per-monitor DPI awareness** `[medium]` - Without proper DPI-awareness manifests the OS bitmap-scales the picker, producing blurry text and icons on HiDPI displays and misplaced hit areas when dragging between monitors of different scale. → **vbuff:** Declare PerMonitorV2 DPI awareness on Windows and handle WM_DPICHANGED; respond to NSWindow backingScaleFactor changes on macOS and wl_surface scale / fractional-scale events on Wayland so the picker renders crisply per monitor.
533. **macOS menu-bar / tray item that breaks across OS releases** `[high]` - Status-item hover/selection states stop working, menus flicker open and shut, or items become unselectable after a macOS upgrade, leaving the only entry point to the app broken. → **vbuff:** Use the current NSStatusItem / menu APIs (not deprecated carbon/legacy paths), test the menu on each new OS beta, and provide a non-menu fallback entry (hotkey + window) so the app is still usable if the status item misbehaves. _(seen in: ClipMenu, Clipy)_
534. **No tray/menu-bar capability detection (tray-less DEs)** `[medium]` - Some Linux desktops (GNOME without an extension) have no system tray at all; a manager whose only UI is a tray icon becomes invisible and unreachable, and on GNOME the recommended Shell extension breaks every version bump. → **vbuff:** Detect StatusNotifierItem / AppIndicator availability; when absent, provide an alternative surface (a windowed launcher, a global-shortcut popup, or a maintained GNOME Shell extension) rather than assuming a tray exists. _(seen in: GNOME Shell, GPaste, Diodon)_
535. **GNOME Shell extension dependency that rots on every GNOME release** `[medium]` - Panel integration delivered as a Shell extension breaks, shows only one entry, or goes unmaintained after each GNOME update, and users must manually re-enable functionality - a perpetual maintenance treadmill tied to the platform's unstable extension API. → **vbuff:** Keep the core capture/storage in a stable D-Bus daemon and treat the Shell extension as a thin, separately-versioned front end pinned to supported GNOME versions, so a broken extension never takes down capture and can be updated independently. _(seen in: GPaste, Clipboard Indicator, Clipboard History)_
536. **App reports 'damaged' / cannot open due to signing & notarization gaps** `[high]` - Missing macOS notarization/Gatekeeper signing (or unsigned Windows binaries) makes the app report itself as damaged or blocked on first launch, so users cannot even start the clipboard manager. → **vbuff:** Ship Developer-ID-signed and notarized macOS builds (with hardened runtime), Authenticode-signed Windows binaries, and a stable signing identity so TCC grants and SmartScreen reputation persist across updates. _(seen in: CopyQ)_
537. **No Apple-Silicon native build (Rosetta-only)** `[medium]` - An x86-only build runs under Rosetta, wastes battery, will stop working on future macOS, and breaks on OS updates - a platform-architecture mistake that quietly degrades and then dies. → **vbuff:** Ship a universal2 (arm64 + x86_64) macOS binary, test on Apple Silicon, and keep the minimum-macOS target current so the app does not rot on the platform it runs on. _(seen in: Clipy, Flycut)_
538. **Login-item / autostart silently fails after OS upgrade** `[low]` - The 'start at login' checkbox silently unchecks after a macOS upgrade (or the systemd/autostart unit is misregistered on Linux, or the Run key is wrong on Windows), so the manager stops capturing at boot and users lose history without noticing. → **vbuff:** Register autostart via the modern per-platform API (SMAppService on macOS, a proper systemd user service / XDG autostart on Linux, registry Run or Task Scheduler on Windows), verify it is actually enabled on each launch, and warn if it was reset. _(seen in: Flycut)_
539. **Multi-minute CPU spike at OS boot/login pegging a core** `[high]` - Doing heavy work (DB migration, full-history load, format scanning) at login pegs a core for minutes and delays the whole desktop startup, a platform-startup mistake that makes the always-on tool a boot tax. → **vbuff:** Defer non-essential startup work, lazy-load history, and start capture immediately with a lightweight path; schedule migrations/compaction for idle time and respect the OS's startup-throttling so login stays fast. _(seen in: ClipboardFusion)_
540. **Holding a delayed-render lock on the OS clipboard causing global paste lag** `[high]` - A manager that takes a long time to serve requested formats (delayed render) or holds the clipboard locked stalls every app's copy/paste, freezing typing for up to a second system-wide. → **vbuff:** Serve clipboard format requests immediately from pre-materialized data, never block in the render callback (handle WM_RENDERFORMAT / pasteboard provider callbacks synchronously and fast), and pre-encode representations at capture time so paste never waits on the manager. _(seen in: Windows Clipboard History)_
541. **Source-app attribution broken on Wayland (no window/app info)** `[medium]` - Wayland intentionally hides which window owns a selection, so managers that rely on X11-style active-window queries record clips with wrong or missing source-app labels, and features keyed on source app silently misbehave. → **vbuff:** Do not depend on cross-client window introspection under Wayland; obtain source identity only through sanctioned channels (the data-control offer, foreign-toplevel-management where available, or app-provided hints) and gracefully omit attribution when the platform does not provide it. _(seen in: Diodon)_

## 16. Performance & reliability mistakes

542. **Tight clipboard polling loop (timer-based change detection)** `[high]` - Waking the CPU every few hundred ms to read the pasteboard change-count keeps a core partly busy even when nothing is copied, draining battery and showing up as a top energy consumer in Activity Monitor / Task Manager. Polling at high frequency also worsens as history grows. → **vbuff:** Prefer OS change notifications where they exist (Windows AddClipboardFormatListener / WM_CLIPBOARDUPDATE; X11 XFixes selection-owner-change events; Wayland wlr-data-control 'selection' events). Only fall back to polling on macOS (which has no native notification), and there poll at a coarse interval (>=300-500ms), check the cheap NSPasteboard.changeCount integer first, and back off when the machine is idle or on battery. _(seen in: Maccy, CopyQ)_
543. **Polling a cloud-synced folder instead of the OS clipboard** `[high]` - Some managers detect changes by re-scanning a Dropbox/Drive folder every ~5 seconds, burning ~1.5 cores; cost grows linearly with item count and the scan happens forever in the background. → **vbuff:** Never use filesystem scanning as the change-detection mechanism. Use OS clipboard events for capture, and for cross-device sync use a push/event-driven channel or filesystem-watch (inotify/FSEvents/ReadDirectoryChangesW) rather than periodic full directory scans. _(seen in: CopyQ)_
544. **Loading the entire history into the UI on every open** `[high]` - With 30k-50k items the popup freezes, scrolling is sluggish, and summon feels like the app crashed, even though the user only wants the last few clips. → **vbuff:** Virtualize the list (render only visible rows) and page/lazy-load from storage with LIMIT/OFFSET or keyset pagination. The summon path should query just the first screenful (e.g. 50-100 rows) and fetch more on scroll. Never materialize the whole table into view models at once. _(seen in: Maccy, Pastebot)_
545. **Storing raw images (full-size TIFF/PNG) inline in the history DB** `[high]` - A single Retina screenshot is 10-30 MB of TIFF; storing raw clipboard data hits hundreds of MB/day and gigabytes/week, bloating disk and ballooning memory when rows are loaded. → **vbuff:** Store image blobs as external files (or external BLOB storage), keep only a path + small downsampled thumbnail in the DB, transcode to a compressed format (PNG/WebP) on capture, and enforce a per-item size budget. Generate previews lazily; never load full-resolution images just to render a list row. _(seen in: Ditto, Maccy)_
546. **Image-history memory bloat from caching full-size decoded bitmaps** `[high]` - Keeping decoded image history resident grows RAM without bound; users report 1-3 GB resident after a long session because every captured image stays in memory. → **vbuff:** Cache only thumbnails in memory with an LRU bound; decode full images on demand and drop them after display. Cap total in-memory image cache by bytes, not by count, and evict aggressively. _(seen in: ClipboardFusion, CopyQ, GPaste)_
547. **Classic memory leak: RAM grows for the lifetime of the process** `[high]` - Memory climbs the longer the app runs and is only released by quitting and relaunching; an always-on utility silently becomes a multi-GB resource hog. → **vbuff:** In Rust, avoid retain cycles via Rc/Arc by using Weak for back-references; bound every cache; run under a leak/heap profiler (valgrind/heaptrack, macOS Instruments leaks, Rust dhat) in CI on a soak test. Add a long-running soak test that copies thousands of items and asserts RSS stays flat. _(seen in: Maccy, ClipboardFusion, CopyClip)_
548. **Clearing history does not free the memory it consumed** `[high]` - User clears history to reclaim RAM, but resident memory stays elevated and keeps climbing, so the clear button is useless for its main purpose. → **vbuff:** On clear, actually drop and deallocate the in-memory structures (replace with empty collections, shrink_to_fit / drop caches), release image caches, and return freed pages to the OS (e.g. malloc_trim on glibc, or use an allocator that returns memory). Verify with a before/after RSS assertion in tests. _(seen in: Maccy)_
549. **Idle CPU spike to ~45% / 1.5 cores while doing nothing** `[high]` - The app pegs over a core at idle, drains battery, and shows as a top 12-hour energy consumer; only a restart temporarily helps. → **vbuff:** Eliminate busy-wait loops and runaway timers; block on event sources rather than spin. Add an idle watchdog metric in debug builds that logs if the main loop wakes more than N times/sec with no clipboard change. Profile with sampling profilers to find the spinning code path. _(seen in: Maccy, CopyQ)_
550. **Background CPU runaway: jumps to 100% indefinitely until killed** `[high]` - While idle the process occasionally enters a 100% CPU state and stays there until manually killed, with no self-recovery. → **vbuff:** Make the monitor loop interruptible and time-bounded; never loop forever on a transient failure. Wrap the capture/monitor task in a supervisor that detects a hung/spinning state (no progress for N seconds) and restarts the task with backoff. Add a circuit breaker around clipboard reads that can fail or block. _(seen in: CopyQ)_
551. **Infinite re-add loop on a single clipboard item** `[high]` - An image (or any item) gets re-added to history repeatedly in a loop, pinning a CPU core and exploding the DB/memory with duplicate entries. → **vbuff:** De-duplicate by content hash and by the OS change-count/serial before inserting; ignore clipboard-change events that the manager itself caused (track 'we just wrote this' ownership). Add a guard that suppresses re-capture of content identical to the most recent N items. _(seen in: GPaste, Clipy)_
552. **Blocking the main/UI thread on clipboard reads, hashing, or DB writes** `[critical]` - Reading large clipboard formats, hashing big blobs, or writing to SQLite on the UI thread freezes the picker and stalls the whole app for up to a second on each copy/paste. → **vbuff:** Do all capture, format conversion, hashing, image transcoding, and persistence on background threads/async tasks; keep the UI thread for rendering only. Use an async runtime (tokio) with a dedicated blocking pool for SQLite (or a writer thread + channel), and never await DB work on the render thread. _(seen in: CopyQ, ClipboardFusion)_
553. **Holding a delayed-render lock on the system clipboard causing global paste lag** `[high]` - A manager that registers as a delayed-render source or keeps re-asserting clipboard ownership makes every app's copy/paste stall up to a second, degrading typing system-wide. → **vbuff:** Capture eagerly into the history store on change rather than offering delayed-render data that other apps must wait on. Avoid re-asserting ownership unless implementing a deliberate persistence feature, and when doing so respond to render requests instantly from a pre-materialized buffer. _(seen in: Windows Clipboard History)_
554. **Degrading the source app after a copy by re-requesting/transforming all formats** `[medium]` - Eagerly pulling and converting every offered MIME/pasteboard type (RTF, HTML, image, file URLs) on each copy slows the source app and raises its memory; a copy becomes expensive. → **vbuff:** Capture the format list lazily: store a cheap reference and only materialize heavy representations (images, RTF) on demand or in a deferred background pass. Cap how many formats and bytes are pulled synchronously at copy time. _(seen in: generic clipboard managers)_
555. **Unbounded SQLite growth: deleted clips never reclaimed** `[critical]` - Deleted rows linger as free pages so the DB file bloats into gigabytes (sometimes tens/hundreds of GB), slowing every query and eventually the whole app. → **vbuff:** Run incremental auto-vacuum (PRAGMA auto_vacuum=INCREMENTAL + periodic incremental_vacuum) or schedule background VACUUM during idle; store large blobs externally so deletes free real files immediately. Track DB size and trigger compaction past a threshold without user action. _(seen in: Ditto)_
556. **Requiring manual Compact & Repair / VACUUM to shrink the DB** `[high]` - Users must run a manual compaction (or external SQLite tools) to reclaim space, and the built-in repair is inconsistent, so most users never do it and the DB rots. → **vbuff:** Make reclamation fully automatic and incremental (no full-rewrite VACUUM that locks the DB for minutes). Use WAL with periodic checkpointing and incremental vacuum on idle. Never expose 'compact' as a required maintenance chore. _(seen in: Ditto)_
557. **DB corruption / freeze when the file grows too large, recovery only by deletion** `[high]` - Past a size threshold the DB corrupts or locks up and the only recovery is renaming/deleting it, which loses all history. → **vbuff:** Cap total store size with a retention policy (evict oldest beyond N items / M MB / D days), enable WAL + synchronous=NORMAL for durability, run PRAGMA integrity_check on startup, and keep blobs external so the index DB stays small and resilient. Provide automatic repair/rebuild that preserves recoverable rows. _(seen in: Ditto)_
558. **Copying very large text/data pins a core and crashes with bad-alloc** `[high]` - Copying hundreds of MB makes the monitor subprocess pin 100% CPU, balloon to ~1.7 GB RAM, then crash with 'bad allocation', taking capture down. → **vbuff:** Enforce a hard per-item byte cap; for oversized content, store a truncated preview + reference rather than the whole blob, or skip capture with a visible note. Stream/chunk reads instead of allocating the entire payload, and wrap allocations so failure degrades gracefully instead of aborting. _(seen in: CopyQ)_
559. **Massive memory blowup tied to environment (multi-monitor / multi-display)** `[high]` - On multi-monitor setups the app consumes 7-8 GiB within seconds of starting, or grows ~1 GiB/s, because per-display monitoring multiplies work or leaks. → **vbuff:** Use a single shared clipboard monitor independent of display count; never instantiate per-monitor capture loops. Test under multi-display and headless/remote configurations in CI, and bound any per-display caches. _(seen in: CopyQ)_
560. **Multi-minute CPU spike at boot/login that delays startup** `[high]` - On launch the app pegs a core for minutes (e.g. eagerly loading/indexing the whole history), delaying the user's login and the app becoming usable. → **vbuff:** Defer heavy work: lazy-load history, build search indexes incrementally in the background, and make the UI responsive before indexing finishes. Persist a ready-to-use index so startup is read-only, and stagger startup tasks off the critical path. _(seen in: ClipboardFusion)_
561. **Slow cold start from schema migration on a large store** `[medium]` - ORM-backed (SwiftData/Core Data style) managers run schema migration over a big store at launch, delaying startup, and inline image blobs make every later save slow too. → **vbuff:** Use lightweight, append-only or versioned migrations that run incrementally/in background; avoid rewriting the whole table on launch. Keep the schema narrow (metadata only) and store blobs externally so migrations touch small rows. Measure cold-start time as a tracked regression metric. _(seen in: Maccy, SwiftData/Core Data managers)_
562. **Search/filter is laggy at large history sizes** `[medium]` - Visible delay between typing and results once history reaches thousands of items, especially with large single-line items, because filtering scans all rows in memory or does unindexed LIKE queries. → **vbuff:** Back search with a proper full-text index (SQLite FTS5) over a normalized text column; debounce input (~100-150ms), run queries off the UI thread, and cancel in-flight queries on new keystrokes. Cap the indexed prefix length of huge single-line items so they don't dominate. _(seen in: Maccy, CopyQ)_
563. **Search/keyboard selection freezes once history hits ~1k-3k items** `[high]` - Search, keyboard navigation, and clearing the search box freeze the UI once history grows to a few thousand items or contains very large items. → **vbuff:** Combine list virtualization with indexed/paged queries so selection never iterates the full set; precompute lightweight display rows. Keep per-keystroke work O(visible rows), not O(history). Stress-test the UI with 50k synthetic items as a gate. _(seen in: CopyQ)_
564. **Settings/tray UI hangs for 20+ seconds requiring force-quit** `[medium]` - Opening settings or the tray menu becomes unresponsive for tens of seconds and the app must be force-closed and relaunched. → **vbuff:** Never do synchronous I/O, DB queries, or large enumerations on the UI/menu-open path; load settings lazily and asynchronously with a responsive placeholder. Add a watchdog that logs/recovers if the UI thread is blocked beyond a few hundred ms. _(seen in: ClipboardFusion)_
565. **Crashes lose data because writes aren't durable / committed** `[critical]` - A crash, kill, or power loss discards recently captured clips (or all history) because data was buffered in memory or written without a durable commit, and there is no recovery path. → **vbuff:** Persist each capture immediately in its own transaction with WAL + synchronous=NORMAL (or FULL for critical writes) so committed data survives a crash. Treat capture as write-through, not write-behind. On startup, run integrity_check and recover from WAL automatically. _(seen in: Maccy, CopyQ)_
566. **No crash recovery / atomic-write strategy for the store and config** `[high]` - A crash mid-write corrupts the DB or settings file, leaving the user with a blank/inaccessible history or reset preferences and no automatic repair. → **vbuff:** Write config/state via atomic write-temp-then-rename; keep a rolling backup of the DB and settings; on corruption detection auto-restore from the last good backup and rebuild indexes. Never overwrite the live file in place. _(seen in: Maccy, CopyQ, Flycut)_
567. **Update silently empties history / breaks capture** `[critical]` - After an app update the entire saved history vanishes or nothing new is recorded, because a migration failed or wiped data without a backup. → **vbuff:** Before any migration, snapshot the existing store; run migrations in a transaction that rolls back on failure; verify capture works post-migration with a self-check and surface an error if it doesn't. Keep the pre-migration backup until the new version is confirmed healthy. _(seen in: Ditto, Maccy)_
568. **Startup crash loop: app crashes and relaunches several times per second** `[high]` - A bad state/config or corrupt DB causes the app to crash on launch and auto-relaunch in a tight loop, making it unusable and burning CPU. → **vbuff:** Detect rapid repeated crashes (crash counter with timestamps) and enter a safe mode that disables the offending feature and quarantines the suspect DB/config instead of looping. Validate config/DB on load and fall back to defaults rather than crashing. _(seen in: CopyQ)_
569. **A specific feature crashes the app several times a day** `[high]` - An optional feature (e.g. triggers/automation) causes random multiple-times-a-day crashes that take the whole manager down, losing in-flight captures. → **vbuff:** Isolate risky/optional features behind a sandbox or out-of-process worker so their failure can't crash the core capture+store. Ship risky features off by default, add per-feature kill switches, and log crash signatures for diagnosis. The core monitor must keep running even if a feature panics. _(seen in: ClipboardFusion)_
570. **Single process: a panic anywhere kills clipboard capture** `[high]` - When UI, capture, sync, and features all run in one process/thread, any panic or hang silently stops capture while the tray icon still shows as 'running', so the user loses history without noticing. → **vbuff:** Run the clipboard monitor as a supervised task (or separate process) decoupled from the UI; catch panics (catch_unwind / restart policy) and auto-restart the monitor. Surface a clear 'capture stopped' indicator instead of pretending to work. _(seen in: CopyQ, Ditto)_
571. **Missing copies during rapid successive copying** `[medium]` - Items copied quickly in a row don't all land in history; fast bursts outrun the capture path so intermediate copies are silently lost. → **vbuff:** Make capture event-driven (not a slow poll) and enqueue every change-count increment into a fast in-memory queue drained by a background writer, so no event is dropped under burst. Avoid debouncing that collapses distinct copies; key de-dup on content, not timing. _(seen in: CopyClip, Maccy, Windows Clipboard History)_
572. **Duplicate rows from no de-duplication bloat the store and slow it** `[low]` - Copying the same text repeatedly creates many identical rows, inflating the DB and list and slowing queries/scrolling, instead of bumping the existing entry. → **vbuff:** De-duplicate on insert by content hash: if the incoming content matches an existing recent entry, update its timestamp/position instead of inserting. Index the hash column so the lookup is O(log n). _(seen in: Maccy)_
573. **Login-item / autostart silently fails so capture stops without warning** `[low]` - After an OS upgrade the launch-at-login registration is dropped (checkbox unchecks), so the manager isn't running and history capture silently stops; the user only finds out when a clip is missing. → **vbuff:** Register autostart via the supported per-OS mechanism (macOS SMAppService, Windows registry Run key/Task Scheduler, Linux XDG autostart), verify on each launch that registration is intact and re-assert it, and show a clear in-UI warning if it's missing. _(seen in: Flycut)_
574. **x86-only build runs under emulation, slow and future-broken** `[medium]` - No native Apple-Silicon (or no native ARM/Windows-on-ARM) build means the app runs under Rosetta/emulation: slower, higher energy, and at risk of stopping entirely on future OS versions. → **vbuff:** Ship native builds for every target architecture (universal2 on macOS, native arm64 on Windows/Linux). For Rust, set up cross-compilation and CI matrices per arch and publish multi-arch artifacts; treat 'native on all supported arches' as a release gate. _(seen in: Clipy, Flycut)_
575. **Synchronous network/cross-device sync on the capture path** `[high]` - Doing sync (peer or cloud) inline with capture or paste makes the app stall, freeze, or drop captures when the network is slow or a peer is unreachable, and can wedge the UI. → **vbuff:** Run sync fully asynchronously and decoupled from capture: capture writes locally first (offline-first), then a background task reconciles/uploads with retries and backoff. A network failure must never block a local copy/paste or the UI. _(seen in: Ditto, Windows 11 Win+V)_
576. **Routing the live DB through a cloud-synced folder** `[high]` - Putting the SQLite DB in Dropbox/Drive produces conflict files, lock contention, and DB corruption as the sync client and the app fight over the file. → **vbuff:** Never store an active SQLite DB in a cloud-synced folder. Keep the DB in app-local storage and implement sync at the record/row level (export deltas, sync those), so the cloud client never touches an open DB file. Document this and refuse to run from a known sync path. _(seen in: Ditto)_
577. **Capture silently stops when another process fights for the clipboard chain** `[high]` - RDP, VMs (VMware/VirtualBox), AutoHotkey, or a second clipboard manager grab the clipboard chain/ownership and the manager silently records nothing while the tray app stays alive. → **vbuff:** Prefer the modern listener APIs (AddClipboardFormatListener) over the legacy clipboard-viewer chain that breaks when a link drops out; detect when capture has gone silent (no events for an unexpectedly long window despite activity), self-heal by re-registering the listener, and surface a 'capture not working' state. _(seen in: Ditto, CopyQ)_
578. **~1s UI freeze when accessing the manager's own clipboard data** `[medium]` - Pasting an item right after selecting it freezes the UI ~1s (notably on Wayland) because reading back the clipboard the app just set blocks on a slow round-trip. → **vbuff:** Serve the chosen item from the in-memory/stored copy rather than round-tripping through the OS clipboard synchronously; if the OS must be queried, do it async with a timeout and a non-blocking UI. Cache the just-written content to answer immediate paste reads instantly. _(seen in: CopyQ)_
579. **Hard low history cap silently drops data users expected to keep** `[medium]` - A hardcoded/low item cap (e.g. 25 items, or a low default) silently evicts older clips, so users lose items they assumed were still there. → **vbuff:** Make limits generous and configurable, with retention by count, total bytes, and age; warn before eviction of pinned/old items and never drop pinned content. Because storage is virtualized and indexed, large histories should stay fast, so caps exist for hygiene, not as a performance crutch. _(seen in: Windows 11 Win+V, Maccy)_
580. **Pinned/favorite items lost or reordered on reboot/update** `[high]` - Pins disappear after an OS update or reboot, silently change order, or pasting a tab of all-pinned items shuffles or crashes the app, so users lose curated content. → **vbuff:** Persist pins and their explicit order durably (separate, backed-up, with stable sort keys) and apply the same crash-recovery/atomic-write guarantees as the main store. Treat pinned data as the highest-durability tier; never let eviction or migration touch it. _(seen in: Maccy, CopyQ)_
581. **Items show up blank/empty and are un-pasteable** `[medium]` - Entries appear in history with no visible content (capture got an empty or wrong representation), making them useless and confusing, and indicating a silent capture failure. → **vbuff:** Validate captured content before committing (non-empty for the chosen representation); if the preferred type is empty, fall back to another offered type or skip the entry rather than storing a blank row. Log capture-source and offered types for diagnosis, and verify round-trip on insert. _(seen in: CopyQ, Maccy)_
582. **No backpressure on the capture queue under sustained load** `[medium]` - A flood of large clipboard changes (e.g. an automation copying repeatedly) can grow an unbounded in-memory work queue until the app OOMs, because there's no limit on pending captures. → **vbuff:** Use a bounded queue with an explicit overflow policy (drop-oldest-pending or coalesce duplicates), apply byte-based backpressure, and coalesce rapid identical events. Monitor queue depth and shed load gracefully rather than allocating without bound.

## 17. Accessibility & i18n mistakes

583. **Clipboard list items have no screen-reader labels** `[critical]` - VoiceOver / NVDA / Orca announce each history row as a blank, generic 'group' or 'cell' with no readable content, so a blind user cannot tell what any clip contains and cannot pick one. The core feature is unusable non-visually. → **vbuff:** Expose each item with a proper accessible name/role: on macOS set NSAccessibility label/role on the cell; on Windows use a real list/listitem (UIA) with Name = a sanitized preview; on Linux expose ATK/AT-SPI roles. Include type (text/image/file), a truncated text preview, and position ('item 3 of 50'). Never rely on visual layout alone.
584. **Popup picker is mouse-only / not keyboard navigable** `[critical]` - Users who cannot or prefer not to use a mouse (motor-impaired, RSI, power users) cannot move through history with arrow keys, page up/down, Home/End, or type-to-search; they must point and click each item, defeating a keyboard-centric utility. → **vbuff:** Make every action reachable from the keyboard: arrow/Home/End/PageUp-Down to move, Enter to paste, Delete to remove, type-ahead search focused on open, Tab order that covers pins and toolbar buttons. Test the whole flow with the mouse physically unplugged. _(seen in: Diodon)_
585. **No visible keyboard focus indicator** `[high]` - Low-vision and keyboard users cannot tell which history row or button currently has focus because focus is shown only by a subtle background tint (or nothing), so they paste the wrong item or get lost in the list. → **vbuff:** Render a high-contrast focus ring (>=2px, >=3:1 contrast against adjacent colors per WCAG 2.4.11/1.4.11) that follows keyboard focus, is distinct from the mouse-hover state, and is never removed via 'outline:none' without a replacement. Make it visible in both light and dark themes.
586. **English-only UI with no localization framework** `[high]` - Non-English speakers cannot understand menus, settings, permission prompts, or error messages, and the app cannot ever be translated because strings are hardcoded in source. A global utility excludes most of the world. → **vbuff:** Externalize all user-facing strings from day one (e.g. Fluent/ICU MessageFormat via fluent-rs or gettext), key-based not English-key based, and load the OS locale by default with a manual override. Ship at least a few locales and accept community translations. Never concatenate sentence fragments. _(seen in: ClipMenu, Flycut, Diodon, cliphist)_
587. **No right-to-left (RTL) layout support** `[high]` - Arabic, Hebrew, Farsi, and Urdu users get a mirror-wrong UI: toolbar buttons on the wrong side, text truncation/ellipsis on the wrong edge, search icon misplaced, and a jarring left-aligned chrome that is hard to use and looks broken. → **vbuff:** Drive layout direction from the UI locale (and per-item content direction). Use logical start/end properties instead of left/right, mirror the whole chrome under RTL, and let the toolkit (GTK/AppKit/Win RTL mirroring or a logical-layout flag) flip automatically. Test with an RTL locale.
588. **Mixed/bidirectional clip text rendered without bidi handling** `[medium]` - A clip mixing RTL and LTR (e.g. Arabic with an English URL or numbers) shows in scrambled visual order, and naive truncation cuts mid-bidi-run producing nonsense previews, so users cannot recognize the clip they copied. → **vbuff:** Render previews through the Unicode Bidirectional Algorithm (the toolkit text engine, or unicode-bidi crate) and isolate each clip's content with first-strong/auto direction (or bidi isolates) so neighboring LTR UI text and RTL content do not contaminate each other. Truncate on grapheme/run boundaries, not bytes.
589. **Fixed tiny font size that ignores OS text-scaling** `[high]` - Low-vision and older users cannot read the history; the app uses a hardcoded small point size and ignores Windows 'Make text bigger', macOS Dynamic Type, GNOME text-scaling-factor, so the only escape is OS-wide zoom which breaks everything else. → **vbuff:** Use relative/DPI-aware units and honor the platform text-scale setting; provide an in-app font-size control that persists. Layout must reflow (no clipping) up to at least 200% per WCAG 1.4.4. Never bake px sizes that cannot grow.
590. **Ignores prefers-reduced-motion for animations** `[medium]` - Users with vestibular disorders, migraine, or motion sensitivity get nauseated or distracted by slide-in panels, animated reordering, and fade transitions that fire on every summon - and these also delay the paste for everyone. → **vbuff:** Query the OS reduced-motion signal (macOS NSWorkspace accessibilityDisplayShouldReduceMotion, Windows SPI_GETCLIENTAREAANIMATION, GNOME gtk-enable-animations) and disable or instantly-complete non-essential animation. Offer an explicit 'reduce animations' toggle too.
591. **No high-contrast / forced-colors support** `[high]` - Users running Windows High Contrast or macOS Increase Contrast see a UI that ignores their theme: low-contrast custom colors persist, text becomes unreadable, focus and selection vanish, because the app paints its own palette regardless of the OS accessibility setting. → **vbuff:** Detect and honor forced-colors / high-contrast modes: use system color roles, respect macOS NSWorkspace accessibilityDisplayShouldIncreaseContrast and Windows high-contrast theme, never hardcode foreground/background pairs without a high-contrast fallback, and ensure selection/focus stay visible.
592. **Insufficient color contrast for text and UI** `[medium]` - Light-gray-on-white timestamps, muted placeholder/search text, and pale separators fall below WCAG contrast minimums, so low-vision users and anyone in bright sunlight cannot read previews, type counts, or metadata. → **vbuff:** Meet WCAG 2.2 AA: >=4.5:1 for normal text, >=3:1 for large text and meaningful UI/graphical elements. Verify both themes with a contrast checker; do not encode state by color alone (also use icon/shape/text).
593. **Color is the only channel for meaning (pinned, type, secret)** `[medium]` - Colorblind users (~8% of men) cannot distinguish 'pinned' vs normal, 'image' vs 'text', or a flagged-sensitive row when the only cue is a colored dot or tint, so they act on the wrong item. → **vbuff:** Pair every color cue with a redundant non-color cue: an icon, a text label, a shape, or position. Ensure the screen-reader name also states the state ('pinned', 'image', 'sensitive'). Test with a colorblindness simulator.
594. **Global hotkey assumes a US QWERTY physical layout** `[medium]` - Users on AZERTY, QWERTZ, Dvorak, or non-Latin layouts find the documented shortcut lands on a different physical key or is impossible to type, and the picker either never opens or fires on the wrong key, silently breaking the primary entry point. → **vbuff:** Bind hotkeys by physical key code (or let the OS shortcut layer handle layout), show the binding using the user's current layout, and provide a 'press the keys' rebind UI that captures the actual scancode rather than a hardcoded character. Validate the binding works across layout switches.
595. **Search/filter is ASCII-only or accent-sensitive** `[medium]` - Typing 'cafe' fails to match a clip containing 'cafe' with an accent, Turkish dotted/dotless i mismatches, and CJK/diacritic input does not match, so non-English users cannot find clips they know they copied. → **vbuff:** Normalize both query and content with Unicode NFC/NFD plus locale-aware case folding and optional diacritic-insensitive matching (e.g. unicode-normalization + a collation/folding step). Use locale-aware comparison, not byte or naive ASCII lowercasing. _(seen in: Paste, Alfred)_
596. **IME composition not supported in the search field** `[high]` - Chinese/Japanese/Korean and other IME users cannot type multi-keystroke characters into the history search: composition is interrupted, candidate windows do not appear or are mispositioned, or Enter during composition wrongly triggers paste instead of committing the character. → **vbuff:** Use native text-input widgets that handle IME composition (preedit/commit events) and never treat keystrokes during an active composition as navigation/paste commands. Position the candidate window relative to the caret. Test with a CJK IME on every platform.
597. **Truncation by byte count splits multibyte characters / graphemes** `[medium]` - Cutting a preview at a fixed byte length splits a UTF-8 sequence or an emoji/combining sequence, producing mojibake or replacement squares, and can corrupt what is shown for CJK/emoji/flag clips - confusing for everyone and unreadable for screen readers. → **vbuff:** Truncate on extended grapheme cluster boundaries (unicode-segmentation crate), measure display width with East-Asian-width awareness, and append a proper ellipsis. Store full content untouched; only the preview is shortened.
598. **Locale-insensitive dates, times, and numbers** `[medium]` - Timestamps shown as hardcoded MM/DD/YYYY or 12-hour AM/PM confuse the ~80% of the world that expects DD/MM, 24-hour, localized month names, or a different calendar; item counts use the wrong decimal/grouping separator, making the metadata feel foreign and error-prone. → **vbuff:** Format dates, times, relative times, and numbers through ICU/locale-aware formatters (e.g. icu4x) using the user's locale and OS regional settings, including 12/24h preference, week start, and digit shaping where appropriate. Never format with hardcoded patterns.
599. **Selection/paste actions not exposed to the accessibility API** `[high]` - Even when items are labeled, the screen reader offers no way to invoke 'paste', 'pin', or 'delete' because the controls expose no actions/roles, so assistive-tech users can hear the list but cannot act on it. → **vbuff:** Expose actionable accessibility roles with named actions: each item is an activatable button/listitem with a default action 'Paste', plus context actions ('Pin', 'Delete') discoverable via the AT actions API (NSAccessibility actions, UIA Invoke/SelectionItem patterns, AT-SPI Action interface).
600. **No live-region / announcement feedback for state changes** `[medium]` - When a clip is captured, pasted, pinned, deleted, or history is cleared, nothing is announced, so a blind user gets no confirmation an action happened and cannot tell success from silent failure. → **vbuff:** Post accessibility announcements / live-region updates for meaningful events ('Pasted', 'Pinned', 'History cleared', 'Copied item added') using NSAccessibility post-notification, UIA LiveRegion/RaiseNotification, or AT-SPI signals. Keep them concise and not chatty on every keystroke.
601. **Tray/menubar icon and menu inaccessible** `[high]` - The only way to reach settings or the history is a system-tray/menubar icon that has no accessible name, no keyboard activation path, and a menu the screen reader cannot read, so AT users cannot configure or open the app at all. → **vbuff:** Give the tray/status item an accessible label and ensure its menu is a real, navigable accessible menu reachable via the platform's status-area keyboard path (e.g. macOS Control-F8 / VoiceOver menubar, Windows Win+B to status area). Provide a global hotkey that does not depend on clicking the icon.
602. **Hover-only previews and tooltips with no keyboard/AT equivalent** `[medium]` - Full-content previews or metadata that appear only on mouse hover are invisible to keyboard and screen-reader users, and dismiss-on-move tooltips are unreachable, hiding information sighted mouse users get. → **vbuff:** Trigger previews on keyboard focus as well as hover, make tooltip content part of the item's accessible description, and ensure WCAG 1.4.13 (content on hover/focus is dismissable, hoverable, persistent). Never gate essential info behind hover alone.
603. **Hardcoded UI width clips translated/long strings** `[medium]` - Fixed-width buttons and labels sized for English overflow or truncate in German, Finnish, Russian, or with large fonts ('Preferences' -> 'Einstellungen'), cutting off settings labels so users cannot tell what a control does. → **vbuff:** Let controls size to content and wrap, leave ~30-40% expansion room, avoid fixed pixel widths on text containers, and test pseudo-localization (lengthened, accented strings) plus real translations at 200% font scale before shipping.
604. **Emoji, ligatures, and complex scripts rendered with a non-shaping font** `[medium]` - Previews using a bitmap/monospace fallback show tofu boxes for emoji, broken Indic/Arabic shaping (unjoined letters), and missing CJK glyphs, so a large share of real clipboard content is unreadable in the picker. → **vbuff:** Render text with a proper shaping/fallback stack (system text engine or HarfBuzz-backed rendering) that does font fallback for emoji, CJK, Indic, and Arabic, supports color emoji, and applies the correct script shaping. Do not pin a single fixed font with no fallback.
605. **Window title, image clips, and icons have no accessible alt text** `[high]` - Image/file clips are announced as nothing or just 'image', and the picker window/buttons carry icon-only controls with no labels, so screen-reader users cannot distinguish image clips or operate the trash/pin/settings buttons. → **vbuff:** Give every icon-only control a text accessible name, set an accessible window title, and label image clips with type plus available metadata (dimensions, source app, OCR text if present) and file clips with filename - never leave them empty or as bare 'image'.
606. **Modal dialogs without focus trap or Escape handling** `[medium]` - Settings/confirmation dialogs do not trap keyboard focus (Tab escapes behind the dialog), do not move focus to the dialog on open, and do not close on Escape, so keyboard and AT users get stranded or operate hidden controls. → **vbuff:** On open, move focus into the dialog and trap Tab/Shift-Tab within it; expose role=dialog with an accessible name; restore focus to the triggering control on close; honor Escape to cancel and Enter for the default button (respecting platform button order).
607. **Sort/comparison and search use byte order instead of locale collation** `[low]` - Alphabetical sorting or grouping of clips/pins puts accented and non-Latin entries in nonsensical order (e.g. Swedish a-ring/o-umlaut, German umlauts, Turkish i) because it compares raw bytes/codepoints, so users cannot scan an ordered list predictably. → **vbuff:** Use Unicode locale-aware collation (ICU/icu4x collator with the user's locale) for any alphabetical sorting or grouping, with appropriate case and accent sensitivity, instead of naive codepoint/byte comparison.
608. **Auto-detected locale cannot be overridden and falls back wrong** `[low]` - Users whose OS language differs from their preferred app language (e.g. an English macOS but a French-preferring user, or a region with no translation) are stuck, and missing-translation fallback shows raw keys or a hard crash instead of graceful English. → **vbuff:** Default to the OS locale but provide an explicit in-app language picker; resolve via the OS preferred-languages list; fall back gracefully along the locale chain (fr-CA -> fr -> en) and never display raw message keys or crash on a missing string.

## 18. Business, trust & maintenance mistakes

609. **One-time purchase switched to subscription** `[high]` - Users who bought a small utility expecting to own it suddenly must pay rent forever to keep using it. This reads as a bait-and-switch, generates 'not worth a subscription' reviews, and burns the goodwill of exactly the early adopters who promoted the app. → **vbuff:** vbuff stays free and open source. If any paid tier ever exists, never retroactively convert an existing perpetual license to a subscription. Honor what users already bought, and gate only genuinely new, server-cost-bearing features (not local clipboard history) behind any future paid tier. _(seen in: Paste, CopyClip)_
610. **Paywalling clipboard history itself** `[high]` - History is the single feature 95% of users install a clipboard manager for. Locking it behind a premium tier treats the core promise as an upsell, so the free product is deliberately useless and users feel manipulated into paying for the obvious. → **vbuff:** Local clipboard history, search, pinning, and paste-back are permanently free and unlimited in vbuff. Never meter clip count, history depth, or search behind a paywall. _(seen in: Alfred, Raycast)_
611. **Bundle-only distribution that stops working if you stop paying** `[high]` - When the app is only available inside a recurring bundle with no standalone perpetual purchase, ending the subscription disables an app that is still installed on disk. Users lose access to a tool and the local data behind it through nothing they did wrong. → **vbuff:** vbuff is distributed as a standalone, perpetually-usable binary. Stopping any optional payment never disables the installed app or locks the user out of their own local history database. _(seen in: Paste)_
612. **Creeping subscription price with no perpetual escape hatch** `[high]` - Prices ratchet up year over year while the locked-in user has no one-time-purchase alternative to fall back to. Trust erodes because the only choices are pay-more or lose-everything. → **vbuff:** vbuff has no recurring price to creep. If a paid sync service is ever added, always offer a self-hosted/local-only path that costs nothing, so users are never trapped by price increases. _(seen in: Paste)_
613. **Forced re-purchase of a paid major rewrite** `[medium]` - Shipping '2.0' as a separate paid app forces previous buyers to re-buy what they thought they owned. Existing users feel punished for being loyal, and reviews fill with 'paid for v1, now they want me to pay again'. → **vbuff:** vbuff treats major versions as free upgrades to existing users. Never ship a major version as a separate paid SKU that abandons prior purchasers. _(seen in: CopyClip 2 (FIPLAB))_
614. **Charging later for what was the app's signature free feature** `[medium]` - Users adopt the app specifically for a feature that is free, build workflows on it, then the vendor moves that same feature behind a paywall. The switching cost is now high, so it functions as coercion rather than a fair offer. → **vbuff:** Features that ship free in vbuff stay free. Document a clear policy: no feature is ever moved from the free tier into a paid tier after release. New paid features must be genuinely additive. _(seen in: ClipboardFusion (Binary Fortress))_
615. **Silent Pro trial that paywalls workflows after it lapses** `[medium]` - The same installer quietly enables a Pro trial, so users unknowingly build daily workflows on Pro-only features. When the trial expires those features vanish and the app feels broken, pressuring a purchase the user never consciously evaluated. → **vbuff:** vbuff ships with no hidden trial. The free tier is the actual baseline experience. Any optional paid features are clearly labeled before use and never silently active, so nothing the user relies on can disappear later. _(seen in: ClipboardFusion (Binary Fortress))_
616. **In-app ads and cross-promotion for the developer's other apps** `[medium]` - A tool sitting over the most sensitive data on the machine (passwords, keys, card numbers) showing ads or 'check out our other app' nags feels cheap and untrustworthy. It signals the vendor sees the user as an ad surface, not a customer to protect. → **vbuff:** vbuff has zero ads, zero cross-promotion, zero upsell nags. No banners, no 'rate us' interruptions, no promotional notifications. The UI surface is reserved for the user's clips only. _(seen in: CopyClip, CopyClip 2 (FIPLAB))_
617. **App phones home from a sensitive-data tool** `[high]` - A clipboard manager that makes network calls is the worst possible candidate for unexplained outbound traffic, because the data it touches is uniquely sensitive. Even benign 'check for updates' pings raise legitimate fear that copied secrets could leave the machine. → **vbuff:** vbuff makes no network calls by default. Update checks, if present, are off by default and explicitly opt-in, clearly scoped to version metadata only, and never transmit clip content. Document every endpoint the app can ever contact. _(seen in: CopyClip)_
618. **Telemetry/analytics with no opt-out (opt-out instead of opt-in)** `[high]` - Bundling 'anonymous' analytics into a clipboard app, on by default, violates the trust baseline this category demands. Users object to any phone-home from software over their copy/paste path, and opt-out consent means data is collected before they ever agree. → **vbuff:** vbuff collects no telemetry. If any diagnostics are ever added they are strictly opt-in (off until the user turns them on), local-first, content-free, and inspectable. Never make analytics a precondition for using the app. _(seen in: ClipboardFusion, vendor closed-source managers)_
619. **Closed source with unverifiable privacy claims** `[high]` - A proprietary manager that claims 'we never see your data' or 'private iCloud only' puts the entire trust burden on marketing copy that nobody can independently audit. For a tool over secrets, unverifiable claims are not reassurance. → **vbuff:** vbuff is fully open source under a clear license, with reproducible builds where feasible, so every privacy and storage claim can be inspected and verified by anyone. No trust-me-bro promises. _(seen in: Paste, PastePal, PasteNow, commercial sync managers)_
620. **Opaque or absent privacy policy** `[medium]` - Without a clear, plain-language statement of what is stored, where, for how long, and what (if anything) leaves the device, users cannot make an informed decision about trusting the app with their clipboard. Silence is read as something to hide. → **vbuff:** vbuff ships a short, concrete, plain-language privacy statement in-repo and in-app: what is captured, where the DB lives, retention, that nothing is transmitted by default, and exactly what any optional sync would send. Update it with every behavior change.
621. **App abandonment leaving a sensitive tool unmaintained** `[critical]` - The developer stops shipping updates and stops responding, leaving a tool that records your clipboard with no security or OS-compatibility fixes. Users keep trusting it with secrets long after it has quietly rotted, then it breaks on an OS update with no recourse. → **vbuff:** vbuff is open source with a documented governance/maintenance plan, a CONTRIBUTING guide, and more than one maintainer where possible. Publish a clear statement that if the project is ever sunset, a final release and the source remain usable and forkable. _(seen in: Clipboard++, Clipboard (open source, last release 2020), ClipMenu, 1Clipboard, Comfort Clipboard)_
622. **Slow/uncertain update cadence with no public roadmap** `[medium]` - Long unexplained gaps between releases leave users unsure whether a tool holding sensitive data is still maintained. The uncertainty itself is harmful: people cannot tell if a bug will ever be fixed or if they should migrate off. → **vbuff:** vbuff publishes a changelog, tags releases, and keeps a visible public issue tracker and roadmap. Even a 'no changes this quarter, still maintained' note beats silence. Respond to issues with at least triage labels. _(seen in: Pastebot (Tapbots), ClipMenu)_
623. **Beta-forever / shipping unfinished as a finished product** `[medium]` - Marketing the app as a complete universal product while it carries unresolved 'unknown issues' on software that handles private data misleads users into trusting an immature tool. They discover the gaps only after relying on it. → **vbuff:** vbuff labels release maturity honestly (alpha/beta/stable) and does not market beta-quality builds as finished. Known limitations are listed in the README. Stable releases meet a documented quality bar before that label is applied. _(seen in: 1Clipboard)_
624. **Breaking changes that silently wipe history on update** `[critical]` - An update empties the saved history or breaks capture, so after upgrading nothing new is recorded and old clips vanish with no warning and no recovery path. Users lose data they expected to be durable and lose trust in every future update. → **vbuff:** vbuff versions its on-disk schema, runs forward migrations automatically, and never destroys existing data on upgrade. Before any migration it makes a backup of the DB. Release notes call out any data-affecting change explicitly. _(seen in: Ditto, Maccy)_
625. **No data export / lock-in of the user's own history** `[medium]` - If history lives only in a proprietary opaque store with no export, users cannot migrate, back up, or leave. They are hostages of the format, which is especially galling for data they themselves generated by copying. → **vbuff:** vbuff stores history in a documented, open format (e.g. plain SQLite with a published schema) and provides one-click export (JSON/CSV) and import. The user can always take their data elsewhere or back it up.
626. **Broken or unverified auto-update channel** `[high]` - An auto-updater that fails silently leaves users stranded on broken versions; one that pulls unsigned or unverified payloads is a remote-code-execution vector into a process that reads the clipboard. Either way the update mechanism becomes a liability, not a safety feature. → **vbuff:** vbuff signs every release, and any auto-update verifies signatures before applying. Updates are opt-in and can be fully disabled (important for managed/enterprise installs). Distribute via reproducible signed artifacts and standard package managers so users can verify provenance.
627. **Damaged / unsigned-binary launch blocked by the OS** `[medium]` - Signing or notarization gaps make the app report itself as 'damaged' or 'cannot be opened', blocking first use entirely. New users hit a wall before they ever try the product and assume it is malware or broken. → **vbuff:** vbuff ships properly signed and notarized builds for macOS, signed installers for Windows, and standard packaged builds for Linux, so the first-run experience is clean on every platform without right-click-open workarounds. _(seen in: CopyQ)_
628. **Single-platform lock-in marketed as complete** `[medium]` - A Mac-only (or Windows-only) tool implicitly sold as a full solution frustrates the large set of users who live across desktops and phones. They adopt it, build habits, then discover it will never cover their other devices. → **vbuff:** vbuff is genuinely cross-platform (macOS, Windows, Linux) from the start, with honest documentation of which platforms are supported to what maturity. Do not imply coverage that does not exist. _(seen in: Maccy, Paste Pal, CopyClip, Raycast (clipboard))_
629. **Poor or absent user support channels** `[medium]` - When there is no issue tracker, no responsive email, and no community, users with data-loss or capture bugs have nowhere to go. Problems with a sensitive tool fester unaddressed and users churn to competitors out of frustration, not feature gaps. → **vbuff:** vbuff provides a public GitHub issue tracker with templates, a discussions/forum space, and documented response expectations. Triage incoming reports promptly and keep a visible 'known issues' list so users aren't left guessing.
630. **Undocumented behavior limitations discovered after adoption** `[medium]` - Limits users only learn by hitting them (e.g. only plain text persists, images silently dropped, a hidden item cap) erode trust because the tool quietly failed to do what was reasonably expected, with no upfront disclosure. → **vbuff:** vbuff documents every limit and supported content type upfront in the README and in-app help, and surfaces in-app warnings when a limit is reached (e.g. 'item too large to store') rather than silently dropping data. _(seen in: Windows 11 Clipboard History)_
631. **Vendor cloud sync as the only sync, forcing data off-device** `[high]` - Making the vendor's cloud the sole sync path forces users to trust an opaque third party with their clipboard and ties a basic feature to a business relationship. If the vendor folds, raises prices, or is breached, the user's most sensitive data is exposed and the feature dies. → **vbuff:** vbuff makes local-only the default and offers self-hosted / LAN / bring-your-own-storage sync as first-class options. Any optional vendor-hosted sync is end-to-end encrypted, clearly opt-in, and never required for core functionality. _(seen in: Windows 11 Clipboard History (Win+V), 1Clipboard, Paste)_
632. **Account/login required to use a local utility** `[high]` - Forcing account creation, email, or login to use a local clipboard manager is gratuitous data collection that creates a vendor dependency and a breach surface for a tool that needs neither. It also blocks offline and air-gapped use. → **vbuff:** vbuff requires no account, no email, and no login for full local functionality. An account is only ever needed for genuinely optional server-backed sync, and even then a no-account self-hosted path exists.
633. **Hostile or data-destroying uninstall** `[medium]` - An uninstaller that leaves clip databases, WAL files, and config behind means secret-laden history survives what the user believed was a clean wipe; one that has no clean removal at all makes users feel trapped. Both betray trust at the exit, when goodwill matters most. → **vbuff:** vbuff ships a documented uninstall path that, with user consent, securely removes the history database and all residual files. Document exactly which paths the app writes to so users can verify removal themselves. _(seen in: Ditto, SQLite-based managers)_
634. **Stagnant OS support / no native build for current architecture** `[medium]` - Low minimum-OS targets and no native build for current CPU architectures mean the app runs only in compatibility layers and will stop working on future OS releases. Users on a tool they trust with secrets get blindsided when a routine OS update kills it. → **vbuff:** vbuff ships native builds for current architectures (Apple Silicon, ARM64, x86_64), tracks and tests against current OS releases, and states its supported-OS matrix publicly. Rust's cross-compilation makes maintaining native targets cheap. _(seen in: Clipy, Flycut, ClipMenu, Pastebot)_
635. **License that misleads users about cost or ownership** `[low]` - An ambiguous or restrictive license (or a 'free' app that is actually time-limited shareware) leaves users unsure what they are allowed to do, whether they truly own their install, or whether a charge is coming. Uncertainty about terms is itself a trust failure for a daily-driver tool. → **vbuff:** vbuff uses a clear, well-known open-source license stated prominently in the repo and About box, with no hidden time limits or nag-to-pay mechanics. What the user can do with the software is unambiguous.
636. **Donation/upsell nags interrupting the core workflow** `[low]` - Even without ads, repeated 'support us' / 'upgrade to Pro' / 'rate the app' interruptions in a tool meant to be instantaneous and invisible add friction to the most frequent action on the machine and make the app feel like adware-lite. → **vbuff:** vbuff keeps any donation or support links confined to the About screen and project page only. No modal nags, no startup interstitials, no notifications interrupting copy/paste. The hot path is sacred.
637. **No reproducible builds / unverifiable distributed binaries** `[medium]` - Even open-source software is only trustworthy if the shipped binary matches the published source. Without reproducible or verifiable builds, a compromised build pipeline could ship a clipboard-stealing binary while the source looks clean, and nobody could tell. → **vbuff:** vbuff aims for reproducible builds and publishes checksums plus signed provenance (e.g. SLSA-style attestations) for every release artifact, so the community can confirm the binary was built from the audited source.
638. **Ownership transfer / acquisition that quietly changes the deal** `[medium]` - A trusted indie tool gets acquired and the new owner adds telemetry, a subscription, or weaker privacy terms under the same name and trust users already extended. Users keep running it unaware the underlying commitments changed. → **vbuff:** vbuff's open-source license and public history mean the community can fork if stewardship changes for the worse. Commit to announcing any ownership/governance change loudly and keeping the last fully-open release available, so trust can never be silently transferred.
